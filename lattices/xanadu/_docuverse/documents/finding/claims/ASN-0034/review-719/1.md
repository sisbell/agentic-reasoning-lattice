### "Contrapositively" in NAT-sub strict-positivity annotation misdescribes the applied proof step
**Class**: OBSERVE
**Foundation**: NAT-order (NatStrictTotalOrder) — exactly-one trichotomy Consequence, `¬(m < n ∧ m = n)` conjunct
**ASN**: NAT-sub (NatPartialSubtraction) — Formal Contract *Consequence:* strict-positivity annotation: `"contrapositively n < m ⟹ n ≠ m"`
**Issue**: The prose derivation immediately above the Formal Contract reaches contradiction by establishing both `n < m` (from the hypothesis) and `n = m` (from collapsing `0 + n = m` via left-identity), then citing `¬(n < m ∧ n = m)` directly — the contradiction is a positive conjunction against a negative conjunct. The Formal Contract annotation adds the gloss "contrapositively `n < m ⟹ n ≠ m`," which describes an equivalent but distinct logical form: apply the implication to obtain `n ≠ m`, then contradict `n = m`. A reader tracing the cited conjunct through the annotation will apply a different derivation shape than the prose actually performs. The conjunct itself is correctly identified; the "contrapositively" characterization is the mismatch.
