### T10a.5 postcondition: `domain(·)` vs `dom(·)` notation inconsistency
**Class**: OBSERVE
**Effective class**: REVISE (classifier override)
**Classifier rationale**: The artifact uses two distinct notations (`dom(·)` and `domain(·)`) for the same defined concept within the same formal contract, creating genuine ambiguity about whether they denote the same or different things, which is a correctness defect in a formal context.
**Foundation**: T10a (AllocatorDiscipline) — Definitions section defines `dom(A) = {tₙ : n ≥ 0}`
**ASN**: T10a Formal Contract Postconditions, T10a.5 entry: "for all `x ∈ domain(X)`, `y ∈ domain(Y)`"; versus T10a.6 entry: "`dom(X) ∩ dom(Y) = ∅`"
**Issue**: The Definitions section and T10a.6's postcondition use `dom(·)`; T10a.5's postcondition uses `domain(·)` for the same concept. A reader consulting the Formal Contract sees two names for one thing.
