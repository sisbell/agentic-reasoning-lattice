### `0 < 1` misclassified as *Axiom:* — body demonstrates it is a consequence
**Class**: REVISE
**Foundation**: NAT-closure (NatArithmeticClosureAndIdentity) — *Axiom:* section
**ASN**: NAT-closure formal contract: `0 < 1` (distinctness of named constants)
**Issue**: NAT-closure's formal contract lists `0 < 1` under *Axiom:*, but the body text of the same claim demonstrates the derivation: `(A n ∈ ℕ :: 0 < n + 1)` instantiated at `n := 0` gives `0 < 0 + 1`; left identity `(A n ∈ ℕ :: 0 + n = n)` at `n := 1` gives `0 + 1 = 1`; substitutivity of `=` then yields `0 < 1`. The body explicitly says the distinctness clause "is recovered as the case `n := 0` together with the left-identity rewrite `0 + 1 = 1`." The word "recovered" is the body's own admission that this is not an independent axiom. The ASN format uses separate *Axiom:* and *Consequence:* slots precisely to distinguish primitive posits from derived claims. Placing a derivable claim in *Axiom:* is a structural inconsistency between the formal contract's classification and the body's own derivation. A downstream consumer reading the formal contract to identify the axiomatic basis of NAT-closure will incorrectly treat `0 < 1` as primitive, when the claim itself establishes it follows from `0 < n + 1` and left identity.
**What needs resolving**: `0 < 1` must be reclassified: either moved to *Consequence:* with a proof citing `(A n ∈ ℕ :: 0 < n + 1)`, left identity, and substitutivity of `=`; or removed from the formal contract entirely if the universal clause is taken to make it superfluous. The *Axiom:* section should contain only claims that are genuinely posited rather than derived within NAT-closure.
