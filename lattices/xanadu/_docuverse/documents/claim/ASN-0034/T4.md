## Hierarchical structure

Tumblers encode a containment hierarchy. Nelson uses zero-valued components as structural delimiters:

> "There are four major fields, each expandable indefinitely, with three major dividers between them. These fields are Server, User, Document and Contents."

> "A tumbler address may have at most three zero digits... These are interpreted as the major dividers, and have lexical significance as punctuation."

We formalize this, writing *node* for Nelson's *server* and *element* for his *contents* and retaining *user* and *document* verbatim; the renaming is notational. An address tumbler has up to four fields separated by zero-valued components; which fields are present in a given tumbler is determined by its zero count, and T4's Definition below gives the per-`k` canonical written form explicitly. The maximal written form Рђћ the case with all four fields present, `zeros(t) = 3` Рђћ is:

`t = NРѓЂ. ... .NРѓљ . 0 . UРѓЂ. ... .Uрхд . 0 . DРѓЂ. ... .DрхД . 0 . EРѓЂ. ... .E╬┤`

where `0 < Nрхб, 0 < UР▒╝, 0 < DРѓќ, 0 < EРѓЌ` at every position. The four fields Рђћ each of which may be absent at lower zero counts Рђћ are:

- **Node field** `NРѓЂ. ... .NРѓљ`: identifies the server. "The server address always begins with the digit 1, since all other servers are descended from it."
- **User field** `UРѓЂ. ... .Uрхд`: identifies the account. "Typically, the user will have no control over the node address he, she or it is assigned; but once assigned a User account, the user will have full control over its subdivision forevermore."
- **Document field** `DРѓЂ. ... .DрхД`: identifies the document and version. Nelson notes the boundary between base document and version is not syntactically marked Рђћ "the version, or subdocument number is only an accidental extension of the document number, and strictly implies no specific relationship of derivation."
- **Element field** `EРѓЂ. ... .E╬┤`: identifies the content element. T4 constrains only `0 < EРѓЌ` at every element-field position; typed interpretations of `EРѓЂ` are the concern of downstream ASNs.

The count of zero-valued components determines the specificity level; T4c is the single definitional site for the resulting address-kind labels.

**T4 (HierarchicalParsing).** Define `zeros(t) = |{i : 1 РЅц i РЅц #t РѕД tрхб = 0}|`, with `|┬и|` NAT-card's set-cardinality operator (distinct from T0's tumbler-length `#┬и`, which acts on sequences); the argument `{i : 1 РЅц i РЅц #t РѕД tрхб = 0}` is a subset of the index domain `{1, Рђд, #t} Ріє РёЋ` (T0), so NAT-card applies at `n = #t` and `zeros(t) Рѕѕ РёЋ` with `zeros(t) РЅц #t`. We say a position `i` of `t` is a *field separator* iff `tрхб = 0`, and a *field component* otherwise; the two roles are decidable by inspecting `tрхб`. Call `t Рѕѕ T` *T4-valid* iff the four conditions `zeros(t) РЅц 3`, `(A i : 1 РЅц i < #t : ┬г(tрхб = 0 РѕД tрхбРѓіРѓЂ = 0))`, `tРѓЂ РЅа 0`, `t_{#t} РЅа 0` all hold; we name the last three conjuncts collectively the *field-segment constraint*. T4 is purely definitional: it characterises T4-valid as a predicate on `T` (the four-conjunct conjunction above) without asserting which `t Рѕѕ T` satisfy it. T4-validity is a precondition at the use-site, not derived here. The *field segments* of `t` are the maximal contiguous sub-sequences of field-component positions Рђћ `k+1` of them when `zeros(t) = k`, separated by the field separators.

*Exhaustion.* We derive as a Consequence Рђћ universally quantified by the bound alone Рђћ that `(A t Рѕѕ T : zeros(t) РЅц 3 : zeros(t) Рѕѕ {0, 1, 2, 3})`, i.e. for every `t Рѕѕ T` with `zeros(t) РЅц 3`, `zeros(t) Рѕѕ {0, 1, 2, 3}`. The derivation uses only `zeros(t) РЅц 3`, NAT-zero, NAT-order's exactly-one trichotomy (Consequence), NAT-discrete, and NAT-closure's left additive identity (cited at the `m = 0` step below). NAT-card's codomain secures `zeros(t) Рѕѕ РёЋ` so that trichotomy applies at `zeros(t)`. NAT-zero supplies `0 < zeros(t) Рѕе 0 = zeros(t)` Рђћ equivalently `0 РЅц zeros(t)` by NAT-order's definition of `РЅц`. Trichotomy at `(zeros(t), m)` asserts that exactly one of `zeros(t) < m`, `zeros(t) = m`, `m < zeros(t)` holds. The mechanism for excluding the branch `zeros(t) < m` given a lower bound `m РЅц zeros(t)` is uniform: unfolding `РЅц` to `m < zeros(t) Рѕе m = zeros(t)` shows that one of trichotomy's other two alternatives holds, and trichotomy's exactly-one clause Рђћ exported by NAT-order as a Consequence Рђћ then forbids the third. At `m = 0`: the case `zeros(t) < 0` is excluded by `0 РЅц zeros(t)` via the exactly-one route just described, so either `zeros(t) = 0`, whereupon `zeros(t) Рѕѕ {0, 1, 2, 3}`, or `0 < zeros(t)`, which NAT-discrete at `(m, n) := (0, zeros(t))` promotes to `0 + 1 РЅц zeros(t)`, which NAT-closure's left additive identity `(A n Рѕѕ РёЋ :: 0 + n = n)` at `n := 1` reduces to `1 РЅц zeros(t)`. At `m = 1` (continuing the `0 < zeros(t)` branch): `zeros(t) < 1` is excluded by `1 РЅц zeros(t)` by the same route, so either `zeros(t) = 1`, whereupon `zeros(t) Рѕѕ {0, 1, 2, 3}`, or `1 < zeros(t)`, which NAT-discrete promotes to `2 РЅц zeros(t)`. At `m = 2` (continuing the `1 < zeros(t)` branch): `zeros(t) < 2` is excluded by `2 РЅц zeros(t)` by the same route, so either `zeros(t) = 2`, whereupon `zeros(t) Рѕѕ {0, 1, 2, 3}`, or `2 < zeros(t)`, which NAT-discrete promotes to `3 РЅц zeros(t)`. At the final step, trichotomy at `(zeros(t), 3)` is confronted with two bounds: `3 РЅц zeros(t)` excludes `zeros(t) < 3` (unfolding `3 РЅц zeros(t)` to `3 < zeros(t) Рѕе 3 = zeros(t)` makes one of the other two trichotomy alternatives hold, forcing `zeros(t) < 3` out by exactly-one), and symmetrically `zeros(t) РЅц 3` excludes `3 < zeros(t)` (unfolding to `zeros(t) < 3 Рѕе zeros(t) = 3` makes one of the other two alternatives hold, forcing `3 < zeros(t)` out by exactly-one), leaving `zeros(t) = 3`. Every branch terminates with `zeros(t) Рѕѕ {0, 1, 2, 3}`.
