# Cone Review — ASN-0034/TA6 (cycle 6)

*2026-04-17 15:27*

### T4c postcondition uses level labels ("node address", "user address", etc.) that are not defined outside `fields(t)`, yet T4c claims no precondition on T4b

**Foundation**: T4 (HierarchicalParsing), Formal Contract *Postconditions* (T4c) and *Preconditions* (T4c entry).

**ASN**: T4c's postcondition: `(A t ∈ T : t is T4-valid :: zeros(t) = 0 ↔ node address (node field only) ∧ zeros(t) = 1 ↔ user address (node and user fields) ∧ zeros(t) = 2 ↔ document address (node, user, and document fields) ∧ zeros(t) = 3 ↔ element address (all four fields))`. The parentheticals characterize each level in terms of which fields are *present* ("node field only", "node and user fields", ...). The Preconditions entry states: "T4c (LevelDetermination) requires no inter-consequence precondition: T4's axiom directly supplies both the bound `zeros(t) ≤ 3` and the role-assignment biconditional `tᵢ = 0 ⇔ i is a field separator`, so the equality between `zeros(t)` and the separator count ... follows from T4 alone, without depending on T4b's `fields(t)` decomposition." T4a's framing paragraph further says the semantic reading of segment-non-emptiness "(once T4c supplies labels)" depends on T4c for labels.

**Issue**: The predicates "node address", "user address", "document address", "element address" are not defined in T4's Axiom, nor anywhere else in T4's contract except through T4c itself. T4's body gloss ("A tumbler with zero zeros addresses a node. One zero: a user account. ...") identifies levels *by zero count*, which makes T4c's four biconditionals tautological (`zeros(t) = k ↔ zeros(t) = k`). The only non-tautological reading of these labels is via the *field decomposition* — "node address" = "only the node field of `fields(t)` is non-empty", etc. — which is precisely what T4b's Definition clause introduces. So either (i) T4c's labels are defined through T4b's `fields(t)`, in which case T4c does require T4b and the "no inter-consequence precondition" claim is wrong, or (ii) the labels are defined by zero count alone, in which case T4c's content collapses to triviality and the parentheticals "(node field only)", "(node and user fields)", etc. are unmotivated. T4a's framing sentence compounds the confusion by asserting that T4c "supplies labels" without saying how. Downstream ASNs citing T4c for a statement about "element address" cannot tell whether the citation grounds out on T4 alone or carries an implicit T4b dependency.

**What needs resolving**: Make the definition of "node address"/"user address"/"document address"/"element address" explicit, and align T4c's Precondition list with that definition. Either (a) give these labels a formal definition in T4's Axiom (e.g., in terms of the canonical written form or `fields(t)`'s components) and, if the definition flows through `fields(t)`, restore T4c → T4b in the Preconditions with the field-decomposition dependency as the load-bearing edge; or (b) define the labels purely by zero count — replacing the parentheticals "(node field only)" etc. with the count-based form — and acknowledge that T4c's biconditionals are then the assignment of names to zero-count values rather than a bridge between two independently characterised notions.
