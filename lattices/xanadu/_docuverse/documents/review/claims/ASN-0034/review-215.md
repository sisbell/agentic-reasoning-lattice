# Cone Review — ASN-0034/TA-strict (cycle 3)

*2026-04-18 00:19*

### TumblerAdd's dominance-proof "divergence" sub-case is under-cited relative to its sibling action-point sub-case
**Foundation**: TumblerAdd (this ASN) — *dominance* paragraph
**ASN**: TumblerAdd's dominance proof, divergence branch: "If some `aⱼ > 0` for `j < k`, the least such `j` is a divergence point with `rⱼ > wⱼ`, so T1 case (i) gives `r > w`."
**Issue**: The sibling sub-case `aₖ > 0` at the action point gets a four-step chain (NAT-addcompat right order-compatibility → NAT-closure additive identity → NAT-cancel summand absorption → NAT-order defining clause) with explicit per-step sourcing, including a footnote arguing why the symmetric route via `aₖ ≥ 1` is unavailable. The divergence sub-case discharges the analogous T1 case (i) invocation in a single sentence, leaving three sub-claims unsourced: (i) the agreement condition `rᵢ = wᵢ` for `1 ≤ i < j` — which needs the minimality of `j` to give `aᵢ = 0` for `i < j` (NAT-wellorder) plus ActionPoint's zeros-below-action-point fact for `wᵢ = 0` (since `i < j < k`); (ii) the bound `j ≤ #w` — chaining `j < k` (minimality) with `k ≤ n = #w` (ActionPoint) via NAT-order's transitivity; (iii) the bound `j ≤ #r` — using the result-length identity `#r = #w`. NAT-wellorder *is* listed in TumblerAdd's Depends, but the citation paragraph attributes it only to "the least such `j`" naming step, not to the agreement-condition argument that the same minimality silently underwrites.
**What needs resolving**: TumblerAdd's dominance proof must either thread per-step citations through the divergence sub-case at the same granularity as the action-point sub-case, or explicitly delegate the bookkeeping (with a sentence locating the load-bearing axioms — minimality for the agreement zeros, ActionPoint for `wᵢ = 0`, result-length identity for the `j ≤ #r` bound) so a reviser tightening any of these axioms has Depends-backed visibility into this site.

### Asymmetric labelled-handle coverage of TumblerAdd's four postconditions
**Foundation**: TumblerAdd (this ASN) — Postconditions: "a ⊕ w ∈ T, #(a ⊕ w) = #w, a ⊕ w > a (T1), **a ⊕ w ≥ w (T1, T3)**"
**ASN**: TA0 covers `a ⊕ w ∈ T` and `#(a ⊕ w) = #w`; TA-strict covers `a ⊕ w > a`; the fourth postcondition `a ⊕ w ≥ w` has no labelled-handle property.
**Issue**: The ASN establishes a labelled-handle pattern — TA0 and TA-strict each exist explicitly so that "downstream users can reference one corollary rather than cherry-pick from TumblerAdd's four-postcondition list" (TA0's text) / "rather than quote TumblerAdd's four-postcondition list" (TA-strict's text). Both motivational paragraphs justify their existence by pointing at the structural cost of cherry-picking. Yet the dominance result `a ⊕ w ≥ w` — the most arithmetically expensive of the four postconditions, with the longest sub-proof — has no labelled handle. Any downstream consumer needing dominance must do exactly the cherry-picking that TA0 and TA-strict were instituted to prevent, reaching past three other postconditions to extract the fourth. The pattern is applied at three of four positions and the omission is not addressed.
**What needs resolving**: Either institute a labelled handle for `a ⊕ w ≥ w` (parallel to TA0/TA-strict, citing TumblerAdd's *dominance guarantee* paragraph), or the ASN must articulate why dominance is exempt from the labelled-handle convention while the other three postconditions are not — e.g., by identifying actual downstream consumers and showing none of them need dominance, or by relegating the labelling pattern to a documented subset of postconditions.

### TA-Pos and ActionPoint independently reconstruct the same least-element argument
**Foundation**: TA-Pos (this ASN) and ActionPoint (this ASN)
**ASN**: TA-Pos's proof: "the set `{i : 1 ≤ i ≤ #t ∧ tᵢ ≠ 0}` is a nonempty subset of ℕ ... so NAT-wellorder's well-ordering of ℕ furnishes a smallest element `k` with `1 ≤ k ≤ #t` and `tₖ ≠ 0`; for all `i < k` we have `tᵢ = 0`, by minimality of `k`." ActionPoint's text: "the set `{i : 1 ≤ i ≤ #w ∧ wᵢ ≠ 0}` is a nonempty subset of ℕ ... NAT-wellorder's least-element principle supplies the least element that min names. The bound `1 ≤ actionPoint(w) ≤ #w` follows because actionPoint(w) is a member of that set, and `wᵢ = 0` for all `1 ≤ i < actionPoint(w)` follows from the minimality of actionPoint(w)..."
**Issue**: TA-Pos and ActionPoint discharge the identical construction — least element of the same set `{i : 1 ≤ i ≤ #t ∧ tᵢ ≠ 0}` via NAT-wellorder, with the identical minimality-yields-zeros-below corollary — at two separate sites. TA-Pos's `k` is precisely `actionPoint(t)`, and TA-Pos's "for all `i < k` we have `tᵢ = 0`" is precisely ActionPoint's second postcondition `wᵢ = 0 for 1 ≤ i < actionPoint(w)`. Yet TA-Pos can't cite ActionPoint (ActionPoint's Depends list cites TA-Pos for the `Pos(w)` predicate, so the dependency runs the other way), and ActionPoint reconstructs the argument rather than reading TA-Pos's witness off TA-Pos's proof. This is the structural twin of the TA-strict/TumblerAdd duplication that Previous Findings flagged: a sister property reproduces a foundational construction with the same axiom citations rather than delegating. A reviser tightening NAT-wellorder's invocation pattern at one site has no Depends-backed signal that the other site needs the same change.
**What needs resolving**: Either (a) extract the action-point construction into a single property cited by both TA-Pos and ActionPoint (e.g., ActionPoint as the construction, with TA-Pos's positive-vs-zero-tumbler postcondition consuming `actionPoint(t)` as a witness) — restructuring the dependency so the construction lives in one place; or (b) keep both proofs but make the relationship explicit, with a paragraph in each acknowledging the parallel and the convention by which both sites independently invoke NAT-wellorder.
