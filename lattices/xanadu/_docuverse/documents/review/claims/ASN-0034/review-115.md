### T10a-N misframes T10a.2's role relative to T10's precondition
**Foundation**: (internal consistency)
**ASN**: T10a-N (AllocatorDisciplineNecessity) states *"T10a.2 (NonNestingSiblingPrefixes) — T10a's own within-allocator consequence that distinct sibling outputs of one allocator are prefix-incomparable ... T10a.2 is the vehicle by which T10a supplies T10's non-nesting precondition on co-sibling prefixes"*. The same framing recurs in the T10a axiom proof at Consequence 5 and in T10a.6's Case 2. T10 (PartitionIndependence), however, takes preconditions `p₁, p₂ ∈ T` with `p₁ ⋠ p₂ ∧ p₂ ⋠ p₁` where `p₁`, `p₂` are the *domain prefixes of two distinct allocators* (e.g., Case 3 of GlobalUniqueness: *"The two allocators A₁ and A₂ are both non-root, with domain prefixes p₁ and p₂ satisfying p₁ ⋠ p₂ ∧ p₂ ⋠ p₁"*). The property that actually discharges T10's precondition for distinct non-ancestor/descendant allocators is T10a.5 (CrossAllocatorIncomparability), not T10a.2. T10a.2 concerns co-siblings of one allocator — pairs to which T10 is not applied.
**Issue**: T10a.2 is described as supplying T10's non-nesting precondition, but T10 is never invoked on co-sibling pairs of a single allocator; its precondition is always sought at cross-allocator domain prefixes, which is T10a.5's job. The "co-sibling prefixes" phrasing is a category error — T10 doesn't distinguish co-sibling vs. cross-allocator prefixes, and co-siblings of one allocator are not prefixes of distinct allocators at all.
**What needs resolving**: The ASN must reconcile the roles of T10a.2 and T10a.5 with respect to T10's preconditions — identifying which theorem actually supplies T10's inputs at each downstream use site, and restating T10a-N's necessity conclusion without claiming T10a.2 is the bridge to T10.

### "Allocator" is used as a first-class entity without identity criteria
**Foundation**: (internal consistency)
**ASN**: T10a defines `dom(A) = {tₙ : n ≥ 0}` and `same_allocator(a, b) ≡ ∃A : a ∈ dom(A) ∧ b ∈ dom(A)`, but never states what an allocator *is* as an object or what makes two allocators equal vs. distinct. T10a.6 (DomainDisjointness) quantifies over "distinct allocators `X` and `Y`", T10a.5 over "two allocators X and Y not in an ancestor-descendant relationship", GlobalUniqueness over "distinct allocation events" assigned to "producing allocators", and PartitionMonotonicity over "parent allocator", "depth-1 child", "depth-2 child". The allocator-tree structure (depth, ancestor/descendant, parent, child) is used throughout, but no axiom formalizes the tree or the identity relation on its nodes.
**Issue**: `same_allocator`'s existential quantifier ranges over an unspecified domain. T10a.6's "distinct allocators" is undefined without an identity criterion. Two allocators with the same base address could — depending on unstated convention — be the same allocator (identified by base) or distinct (identified by creation event). GlobalUniqueness's producing-allocator taxonomy hinges on the second reading; T10a.7's enumeration injectivity implicitly assumes the first. The spec leans on intuition without committing.
**What needs resolving**: The ASN must either axiomatize allocators (as tree nodes, creation events, or via some equivalent structure with an explicit identity relation) or show that the downstream claims go through without requiring allocator identity. As written, `same_allocator`, `dom(·)`, and "distinct allocators" lack a formal denotation.

### `fields(t)` has no stated signature or behavior on absent fields
**Foundation**: (internal consistency)
**ASN**: Vocabulary: *"fields(t) — function that extracts the node, user, document, and element fields from tumbler t"*. T4b (UniqueParse) states *"the function `fields(t)` that decomposes a tumbler into its node, user, document, and element sub-sequences is well-defined and uniquely determined by `t` alone"*, but gives no codomain. T6 writes *"Write `N(t)`, `U(t)`, `D(t)`, `E(t)` for the node, user, document, and element fields of `t` respectively. Each is a finite (possibly absent) sequence"* and treats missing fields via separate presence checks using `zeros(t)`. T7 assumes `a.E₁`, `b.E₁` denote positions — but only meaningful when the element field exists.
**Issue**: The signature of `fields(t)` is never stated (does it return a 4-tuple with absent slots, a partial function, a dictionary keyed by present fields?), and the behavior on tumblers with fewer than four fields is left to the reader. T6 works around this with `zeros(t)`-gated presence checks, but T7's `a.E₁ ≠ b.E₁` expression presupposes both have element fields without stating the precondition explicitly; the formal contract's precondition `zeros(a) = zeros(b) = 3` covers this only after the fact. Downstream consumers cannot uniformly rely on `fields` without a signature.
**What needs resolving**: The ASN must state the codomain of `fields(t)` and specify behavior on each hierarchical level (how absent fields are represented), so that `N(t)`, `U(t)`, `D(t)`, `E(t)` and their component-access notation `a.E₁` have determinate meaning across the cases T4c distinguishes.

### TS2's empty-sequence case is unreachable under T0
**Foundation**: (internal consistency)
**ASN**: TS2 (ShiftInjectivity) proof: *"If m = 0, both v₁ and v₂ are the unique empty sequence (), so v₁ = v₂ and we are done. Assume m ≥ 1 for the remainder."* The TS2 preconditions state `v₁, v₂ ∈ T` and `#v₁ = #v₂ = m`. T0 (CarrierSetDefinition) requires `#a ≥ 1` for every `a ∈ T`, and the Formal Contract writes `#a ≥ 1 for all a ∈ T`.
**Issue**: The `m = 0` branch of TS2's proof contemplates length-0 tumblers, but T0 forbids them — no tumbler has length 0, so the case is vacuous. The branch isn't incorrect, but it is dead code that signals either a misremembered carrier convention or confusion about whether T admits the empty sequence. Other properties in the same ASN (e.g., OrdinalShift, TS1, TS3, TS4, TS5) consistently assume `m ≥ 1` without such a branch.
**What needs resolving**: Remove the `m = 0` case or note that T0 excludes it, so the proof's case analysis matches the carrier set as defined.
