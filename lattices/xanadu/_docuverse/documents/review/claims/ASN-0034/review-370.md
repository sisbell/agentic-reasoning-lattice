# Regional Review — ASN-0034/T4c (cycle 5)

### T4b's Preconditions are per-`t` but its Definition and Postconditions describe global function behaviour
**Foundation**: structural — a claim's *Preconditions* slot should match the claim's logical form (universal on `t`, or scheme declaring global objects)
**ASN**: T4b *Preconditions*: "`t` satisfies the T4 constraints." T4b *Definition*: "The four partial functions `N, U, D, E : T ⇀ T` are characterised as follows. `dom(N)` is the T4-valid subset of `T`; `dom(U) = {t ∈ dom(N) : zeros(t) ≥ 1}`; …" T4b *Postconditions*: "`N, U, D, E : T ⇀ T` are partial functions. `dom(N)` is the T4-valid subset of `T`; … Outside the T4-valid subdomain, none of the projections is assigned a value; consumers must carry T4-validity as a precondition."
**Issue**: T4b's Precondition is phrased as a constraint on a particular `t` — as if T4b were a per-`t` claim. But T4b's Definition and Postconditions describe four global partial functions on `T`, including explicit behaviour outside the T4-valid subdomain ("none is assigned a value"). If T4b is about an individual `t`, the global function description over-specifies; if T4b is a global function definition, the per-`t` Precondition is ill-typed (the functions are defined independent of any particular `t`). A reader cannot tell which kind of claim T4b is, and the "consumers must carry T4-validity" note in Postconditions is already redundant with a per-`t` Precondition if T4b has one.
**What needs resolving**: Either drop the per-`t` Precondition (so T4b is a global definition of four partial functions; consumers discharge T4-validity per-use), or restrict the Definition/Postconditions to per-`t` values (so T4b is a statement about what each projection returns on a T4-valid input).

### NAT-zero's Axiom uses `≤` before NAT-order defines it
**Foundation**: NAT-zero (`0 ∈ ℕ` and `(A n ∈ ℕ :: 0 ≤ n)`); NAT-order, which *defines* `≤` by `m ≤ n ⟺ m < n ∨ m = n`
**ASN**: NAT-zero *Axiom*: "`0 ∈ ℕ` (zero is a natural number); `(A n ∈ ℕ :: 0 ≤ n)` (zero is a lower bound of ℕ)." NAT-order *Axiom*: "The non-strict relation `≤` on ℕ is defined by `m ≤ n ⟺ m < n ∨ m = n`."
**Issue**: NAT-zero's second clause uses `≤`, but `≤` is introduced — as a defined relation, not a primitive — only in NAT-order. Read as a self-contained axiom block, NAT-zero invokes an undefined symbol. Every other axiom in this ASN that uses a symbol from another claim does so via an explicit dependency citation; NAT-zero has no Depends slot. A precise reader auditing NAT-zero in isolation cannot interpret `0 ≤ n` without silently importing NAT-order's definition.
**What needs resolving**: Either restate NAT-zero in terms of `<` and `=` directly (e.g., `(A n ∈ ℕ :: 0 < n ∨ 0 = n)`), or add an explicit dependency from NAT-zero on NAT-order's definition of `≤`, or note in NAT-order that `≤` is in scope for subsequent axiom bodies.

### "Field separator" is defined twice — once in the Hierarchical structure prose, once in T4
**Foundation**: structural — a term should be defined once in the ASN and reused; T4 is the formal locus for the parsing constraints
**ASN**: Hierarchical structure section (main-body prose, before T4): "Define a *field separator* as a position `i` of `t` with `tᵢ = 0`." T4 main body: "T4 stipulates that a position `i` is a *field separator* iff `tᵢ = 0`, and that every other position carries a *field component* — strictly positive by the preceding argument". T4 *Axiom*: "T4 stipulates that a position `i` of `t` is a *field separator* iff `tᵢ = 0`."
**Issue**: The term is defined three times, in three places, with two different framings: the pre-T4 prose treats it as a bare definition (for any `t`), while T4's body and Axiom re-state it as a T4 stipulation. If the pre-T4 definition is in force, T4's "stipulates" language is redundant; if T4's stipulation is the binding one, the earlier definition is premature. A reader tracing where *field separator* is fixed encounters it three times and must decide which occurrence is load-bearing.
**What needs resolving**: Fix a single definition site and reference it from the others (e.g., define once in T4 and drop the earlier prose definition, or define once in the Hierarchical structure section and have T4 merely cite it).

### T4's Postconditions list T4a as a "direct consequence", but T4a's own Depends cites foundations beyond T4
**Foundation**: structural — a Postcondition of T4 should be a fact that T4, under its own dependencies, establishes; standalone theorems with their own foundation lists are not Postconditions of a single prerequisite
**ASN**: T4 *Postconditions*: "T4a (SyntacticEquivalence), a direct consequence: the field-segment constraint is equivalent to every field segment of `t` being non-empty." T4a *Depends*: T0, NAT-zero, NAT-discrete, NAT-order, NAT-closure, NAT-sub, NAT-card, and T4 (eight entries).
**Issue**: T4's Postcondition clause labels T4a a "direct consequence" of T4, suggesting T4 alone implies T4a. But T4a's own Formal Contract enumerates a Depends list of eight foundation claims — including NAT-sub and NAT-card beyond what T4's Depends lists, and with a local unpacking of T0's "nonempty" into `#t ≥ 1`. T4a's proof performs non-trivial arithmetic (`s₁ ≥ 2`, `s_k ≤ #t − 1`, `s_{j+1} ≥ s_j + 2`) grounded in those extra dependencies. A direct consequence of T4 alone would need no foundation beyond T4's; T4a needs several. T4a is a standalone theorem that combines T4 with additional foundations — not a Postcondition of T4.
**What needs resolving**: Either remove T4a from T4's Postconditions (leaving it as the first-class numbered theorem it is elsewhere described as, downstream of T4 plus additional foundations), or pare T4a's Depends down to what T4 alone supplies and justify the arithmetic steps without the extra foundations.
