# Cross-cutting Review — ASN-0034 (cycle 2)

*2026-04-16 12:52*

Reviewing the ASN for cross-cutting issues beyond those in Previous Findings.

### `Σ` used in `AllocatedSet` before being introduced by `NoDeallocation`; not declared in `AllocatedSet`'s Depends
**Foundation**: (internal) — other forward references in this ASN are explicitly marked (e.g., `PositiveTumbler`'s Depends labels `TA0` "forward reference — TA0 is stated in § Tumbler arithmetic, after this section"; `T1`'s Depends labels `T3` similarly).
**ASN**: `AllocatedSet` uses `Σ` in its prose and Formal Contract: "*State transition:* s → s' is the application of one operation from Σ. … Each allocation-affecting transition does exactly one of two things …". `AllocatedSet`'s Depends list names only `T10a` and `T9`. `Σ` is first *introduced* in `NoDeallocation`, which comes later: "Let Σ denote the system's transition vocabulary — the complete set of operations the system may perform."
**Issue**: `AllocatedSet` appeals to a symbol whose definition and closure assumption are stipulated only by a later property, yet `AllocatedSet` neither imports it nor marks it as a forward reference in the style used elsewhere. Downstream consumers (`T8`, `T9`, `GlobalUniqueness`) cite `AllocatedSet` as a dependency and thereby inherit a transition-system semantics grounded in an undeclared symbol. A TLA+-style extraction that walks contracts to build the dependency DAG will not discover that `AllocatedSet` — and consequently everything that depends on it — relies on `NoDeallocation`'s frame declaration of `Σ`.
**What needs resolving**: Anchor `Σ` to a single definition site and make the dependency explicit in every contract that invokes it. Either introduce `Σ` within `AllocatedSet` (and have `NoDeallocation` import it), or mark `Σ` as a forward reference to `NoDeallocation` in `AllocatedSet`'s Depends with the same convention used for other forward references in this ASN.

### `op(s) defined` partiality predicate is used without being characterized
**Foundation**: (internal) — `AllocatedSet` and `NoDeallocation` both treat transitions as partial, but neither states this as a framing assumption about `Σ`.
**ASN**: `NoDeallocation`'s axiom: "`(A op ∈ Σ, s ∈ 𝒮 :: op(s) defined ⟹ allocated(s) ⊆ allocated(op(s)))`". `AllocatedSet` uses the same partiality implicitly: "every operation from Σ. Each allocation-affecting transition does exactly one of two things …" with non-allocating transitions listed separately, and the prose "where `op(s)` is defined".
**Issue**: The predicate `op(s) defined` is load-bearing — `NoDeallocation`'s axiom is vacuously satisfied when `op` is totally undefined at `s`, and `T8`'s monotonicity conclusion implicitly restricts its state-pair quantifier to those pairs `(s, s')` arising from a defined application. Nothing in the ASN records that operations in `Σ` are partial functions on `𝒮`, nor under what conditions `op(s)` is defined. A formalizer must choose a partiality convention (e.g., `op : 𝒮 ⇀ 𝒮`, or `op ⊆ 𝒮 × 𝒮`) that this ASN does not supply.
**What needs resolving**: Declare the signature of elements of `Σ` once (either as partial functions or as binary relations on `𝒮`), either in `AllocatedSet` or as part of `NoDeallocation`'s frame, and let the axiom's `op(s) defined` clause be interpreted against that signature.

### `PositiveTumbler`'s postcondition uses component projection `zᵢ` on an arbitrary `z ∈ T` without the bound `i ≤ #z`
**Foundation**: `T0 (CarrierSetDefinition)` — component projection `·ᵢ` is defined for `1 ≤ i ≤ #a`.
**ASN**: `PositiveTumbler`'s Formal Contract Postcondition: "`(A t ∈ T, z ∈ T : Pos(t) ∧ (A i : 1 ≤ i ≤ #z : zᵢ = 0) :: z < t)`". The guard `(A i : 1 ≤ i ≤ #z : zᵢ = 0)` is well-scoped, but compare `TA6 (ZeroTumblers)` conjunct 2: "`(A s, t ∈ T : (A i : 1 ≤ i ≤ #s : sᵢ = 0) ∧ (E j : 1 ≤ j ≤ #t : tⱼ > 0) ⟹ s < t)`". The two postconditions assert very similar facts — PositiveTumbler's is the contrapositive skeleton of TA6's forward direction.
**Issue**: `PositiveTumbler` and `TA6` establish essentially the same relation "every zero tumbler precedes every positive tumbler" and each carries its own proof. `PositiveTumbler`'s postcondition talks about a positive `t` and a zero-tumbler `z`; `TA6`'s second conjunct talks about a zero-tumbler `s` and a tumbler `t` with some positive component (i.e., `Pos(t)`). These are the same statement with operand names swapped. Two independently proved statements of the same fact invite drift: a downstream cite to one may later fail if the other's statement is tightened, and a reviewer cannot tell which is canonical.
**What needs resolving**: Designate one of `PositiveTumbler`'s postcondition or `TA6`'s second conjunct as the canonical statement, and have the other cite it (or remove it) rather than reprove it. If both are kept, make the relationship explicit so the pair can be checked for mutual consistency.
