# Sub-Questions — Allocation Order Invariants

**Inquiry:** What must the system guarantee about the relationship between the temporal order in which content is allocated and its spatial position in V-space? Must allocation order correspond to document order?

1. [nelson] Must the position assigned to content within a document reflect the chronological order in which that content was created by its author?
2. [nelson] When a user inserts new content between two existing passages, must the new content receive a position that falls between the positions of those passages, or may it receive a position that reflects only when it was added?
3. [nelson] Does the system guarantee that content added later in time always appears after content added earlier, or can a user place new content at any position within a document regardless of when it was composed?
4. [nelson] Must the system's permanent identity for a piece of content encode any information about when that content was allocated relative to other content?
5. [nelson] If two authors create content simultaneously in separate documents, must the system impose a global ordering on their content, or may their positions be entirely independent?
6. [nelson] When content is rearranged within a document through editing, must the original allocation order remain recoverable as a distinct property from the current document order?
7. [nelson] Must the system distinguish between the order in which content was brought into existence and the order in which a reader encounters it within a finished document?
8. [nelson] If a document is constructed by assembling previously existing content into a new arrangement, must the document order bear any relationship to the creation dates of those pieces?
9. [nelson] Does the system guarantee that the permanent address space grows monotonically with time, such that later allocations always occupy higher positions than earlier ones?
10. [nelson] Must the system preserve the ability to reconstruct the exact temporal sequence of allocations, even when the resulting document presents content in a completely different reading order?
11. [gregory] After INSERT "A" at V:1.1 then INSERT "B" at V:1.1 (same position, LIFO), the later-allocated "B" (higher I-address) appears before the earlier-allocated "A" in V-space — is this inversion of allocation-order vs. document-order the only mechanism that produces such inversions, or can REARRANGE and COPY also create them?
12. [gregory] When COPY transcludes content from doc1 to doc2, the target POOM maps new V-positions to old I-addresses — does the POOM's `incontextlistnd` insertion-sort handle the case where transcluded I-addresses are interleaved with locally-allocated I-addresses that have higher values but earlier V-positions?
13. [gregory] Does REARRANGE (pivot or swap) ever update or recompute I-address-to-V-address mappings in the POOM's second dimension, or does it only modify V-displacements while leaving I-displacements untouched — meaning the 2D enfilade's I-dimension permanently reflects allocation order regardless of current V-order?
14. [gregory] When `retrievevspansetpm` walks the POOM to return document content, does it enumerate entries in V-address order or in I-address order, and what guarantees that the returned spans reflect current document order rather than allocation order?
15. [gregory] In `findcbcinarea2d`, when the 2D tree traversal encounters POOM entries where V-order and I-order diverge (e.g., after a REARRANGE), does the diagonal-sum ordering used in `recombinend` for rebalancing risk placing entries in a physical tree order that conflicts with the V-address insertion-sort in `incontextlistnd`?
16. [gregory] After a sequence INSERT "ABC" then DELETE "B" then INSERT "XY" at the gap, the new "XY" has higher I-addresses than "C" but sits between "A" and "C" in V-space — does `compare_versions` (SHOWRELATIONOF2VERSIONS) correctly identify the shared "A" and "C" spans despite the I-address gap created by the interleaved allocation?
17. [gregory] When `vspanset2sporglset` converts a contiguous V-span to sporgls, and the V-span covers content from multiple temporal insertions (so I-addresses are non-contiguous), does it split at every I-address discontinuity, and is the resulting sporgl set ordered by V-position or by I-address?
18. [gregory] For the spanfilade's DOCISPAN entries, each INSERT creates one entry per contiguous I-span — when a document's V-order completely disagrees with I-order (after many rearrangements), does `findlinksfromtothreesp` still correctly discover links by searching the I-dimension, independent of the document's current V-ordering?
19. [gregory] Does the `isanextensionnd` coalescing optimization in `insertpm` check I-address contiguity only, or does it also verify that the new entry's V-position is adjacent to the existing entry's V-span — and if a COPY places I-address-contiguous content at a non-adjacent V-position, does it incorrectly coalesce?
20. [gregory] When two documents share I-addresses via COPY but have different V-orderings of that content (e.g., doc1 has "AB" at V:1.1 and doc2 has the same content reversed via REARRANGE), does `find_links` with a V-span covering only "A" in doc2 correctly resolve to the same I-address as "A" in doc1, even though "A" occupies different V-positions in each document?
