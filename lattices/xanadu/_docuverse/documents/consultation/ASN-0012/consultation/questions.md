# Sub-Questions — Enfilade Properties

**Inquiry:** What abstract properties must the indexing data structure satisfy? What guarantees about range queries, composable widths, and structural balance does the system require — independent of any particular tree implementation?

1. [nelson] When a reader requests a contiguous range of a document — say, characters 5 through 50 — what must the system guarantee about the completeness and ordering of what is returned?
2. [nelson] If a document is built from several contributed pieces, must the total length of the document always equal the sum of the lengths of its constituent pieces, and must this hold at every level of aggregation?
3. [nelson] Must the system guarantee that retrieving any range of a document takes time proportional to the size of that range, regardless of how large the document has grown or how many edits it has undergone?
4. [nelson] The system must answer two kinds of questions about the same content — "what is at this position?" and "where did this content originate?" Must both queries be answerable from a single unified structure, or could they be served independently?
5. [nelson] When new content is inserted into the middle of a document, must the positions of all subsequent content shift by exactly the inserted amount, with no gaps and no overlaps?
6. [nelson] When the same original content appears in multiple documents through transclusion, must the structure allow discovering all the places that content appears without scanning every document?
7. [nelson] When a new version of a document is created, must the structure for the new version share the unchanged portions with the original, or is copying the entire document acceptable?
8. [nelson] If a link points to a range within a document, and content is later inserted before that range, must the structure ensure the link's target can still be located at its shifted position?
9. [nelson] Must the structure preserve its performance guarantees after any sequence of insertions, deletions, and rearrangements, or are there editing patterns that could degrade retrieval?
10. [nelson] If the system crashes partway through updating the structure for an insertion, must the structure be recoverable to a consistent state — either with the insertion fully applied or fully absent?
11. [gregory] When a range query specifies a V-address interval [a, b), does the enfilade guarantee that every bottom crum whose logical span overlaps [a, b) is visited exactly once, or can partial-overlap crums be visited multiple times during traversal?
12. [gregory] The displacement at each tree level accumulates additively — child grasp equals parent displacement plus child's own displacement. Is additive composition the only algebraic requirement, or must displacements also satisfy cancellation (i.e., if d₁ + d₂ = d₁ + d₃ then d₂ = d₃) for correctness?
13. [gregory] In the 2D enfilade (POOM, spanfilade), the root displacement tracks the minimum child address via `setwispnd`. Is the invariant that root displacement equals the componentwise minimum of all descendant displacements, or is it the minimum of direct children only?
14. [gregory] Does the system require that a range query on an interval entirely outside the enfilade's populated region return empty in O(1), or is it acceptable for the traversal to walk down to a leaf before determining emptiness?
15. [gregory] When `recombinend` sorts children by the diagonal sum of their two displacement coordinates, does correctness require that this diagonal ordering be a total order on children, or would any ordering that groups spatially adjacent children together suffice for rebalancing?
16. [gregory] The granfilade uses `MAXBCINLOAF=1` (one entry per bottom crum), while the POOM uses `MAX2DBCINLOAF=4`. Is the essential property that bottom crum occupancy is bounded above, or does the POOM also require a minimum occupancy guarantee (e.g., at least 2 entries after a split)?
17. [gregory] When `incontextlistnd` performs insertion-sort by V-address during retrieval, is this sorting necessary because the tree traversal order can differ from logical V-order, or is it a defensive measure against a specific class of tree shapes produced by certain operation sequences?
18. [gregory] Does the width stored in an upper crum need to equal the exact sum of its children's widths, or is it sufficient for the width to be an upper bound that contains all children — i.e., could a "bounding box" semantics work for 2D enfilades instead of exact aggregation?
19. [gregory] After a split promotes a new sibling via `splitcrumupwards`, must the parent's displacement and width be recomputed from scratch over all children, or does the split algorithm maintain them incrementally — and which approach does correctness actually require?
20. [gregory] If two bottom crums in the same POOM have V-spans that are logically adjacent (the end of one equals the start of the next) but map to non-contiguous I-addresses, does the enfilade guarantee they remain as separate crums, or could a rebalance operation merge them into a single crum with a discontinuous I-mapping?
