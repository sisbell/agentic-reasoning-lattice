### TA-Pos's step `0 < 0 + 1` ‚Üí `0 < 1` elides an unsourced identity, and NAT-closure is missing from TA-Pos's Depends
**Foundation**: NAT-closure (NatArithmeticClosureAndIdentity) ‚Äî additive identity `0 + n = n`; TA-Pos's Depends does not list NAT-closure.
**ASN**: TA-Pos proof, Case `#z ‚â• k`: "NAT-addcompat's strict successor inequality `n < n + 1` instantiated at `n = 0` supplies `0 < 0 + 1`, **equivalently `0 < 1`**"; Depends paragraph on NAT-addcompat: "threads through `0 < 1` as the stepping stone, which NAT-addcompat's strict successor inequality ... supplies directly as `0 < 0 + 1`. No other listed axiom produces `0 < 1` in TA-Pos's scope".
**Issue**: NAT-addcompat's `n < n + 1` at `n = 0` yields `0 < 0 + 1`, *not* `0 < 1`. The "equivalently" step rewrites `0 + 1` to `1`, which requires the additive identity `0 + n = n` ‚Äî an axiom this ASN explicitly sources from NAT-closure at analogous sites (cf. TumblerAdd's dominance sub-case `a‚Çñ > 0`, where the identical rewrite of `0 + w‚Çñ` to `w‚Çñ` is cited as "NAT-closure's additive identity `0 + w‚Çñ = w‚Çñ` rewrites the right-hand side"). TA-Pos treats the rewrite as if it were syntactic, whereas TumblerAdd ‚Äî exemplifying the per-step citation convention this ASN otherwise enforces ‚Äî treats it as a distinct axiom step. The Depends paragraph on NAT-addcompat even claims "No other listed axiom produces `0 < 1`", conflating the conclusion NAT-addcompat actually supplies (`0 < 0 + 1`) with the conclusion consumed downstream (`0 < 1`), obscuring rather than discharging the gap.
**What needs resolving**: TA-Pos must either source the `0 + 1 = 1` rewrite by adding NAT-closure to its Depends (and citing NAT-closure's additive identity at this step, parallel to TumblerAdd's treatment of the mirrored rewrite), or document a convention under which `0 + 1` and `1` denote the same numeral without axiomatic mediation ‚Äî and apply that convention uniformly across the ASN, including retrofitting the TumblerAdd citations that currently treat the identity as load-bearing.

### T1 Trichotomy uses the natural-language connective "exactly one of" inside a formal quantifier
**Foundation**: T1 (LexicographicOrder, this ASN) ‚Äî the formal contract's Postconditions line.
**ASN**: T1 formal contract, Postconditions (b): "Trichotomy ‚Äî `(A a,b ‚àà T :: exactly one of a < b, a = b, b < a)`".
**Issue**: "exactly one of" is English, not a logical connective. Inside a three-part universal quantifier `(A x : range : body)`, the body must be a predicate. Compare postcondition (a), which uses `¬¨(a < a)` ‚Äî a proper formula ‚Äî and (c), which uses `a < c` ‚Äî likewise a proper formula. The proof body of part (b) discharges both the existence-of-one-outcome claim (three exhaustive cases each producing one) and the uniqueness-of-outcome claim (no witness establishes a second), so the substance is there, but the formal statement does not distinguish these or package them as a single logical expression (e.g., `(a < b ‚àß ¬¨(a = b) ‚àß ¬¨(b < a)) ‚à® (¬¨(a < b) ‚àß a = b ‚àß ¬¨(b < a)) ‚à® (¬¨(a < b) ‚àß ¬¨(a = b) ‚àß b < a)`, or an equivalent ExactlyOne predicate). Downstream consumers invoking "T1's trichotomy" ‚Äî e.g., the fourteen trichotomy sites enumerated in T1's own Depends list, and the "exactly one" clause reuses that the Depends paragraph repeatedly cites as "exactly-one clause" ‚Äî rely on a formal biconditional that the postcondition itself does not supply in machine-checkable form.
**What needs resolving**: Either expand "exactly one of" into a formal disjunction-of-conjunctions (or an explicitly named `ExactlyOne`/`AtMostOne ‚àß AtLeastOne` pair), or split (b) into two postconditions ‚Äî existence of one of the three relations and pairwise mutual exclusion ‚Äî each stated formally ‚Äî so that the "exactly-one clause" the Depends paragraph cites fourteen times has a formal referent rather than a natural-language shorthand.
