# Regional Review — ASN-0034/T4c (cycle 4)

*2026-04-22 04:36*

### T4c's Exhaustion cites "NAT-order transitivity" for a step that also requires equality substitution
**Foundation**: NAT-order (NatStrictTotalOrder), whose stated axioms are irreflexivity, transitivity (of `<`), and trichotomy. The non-strict relation is defined `m ≤ n ⟺ m < n ∨ m = n`. Substitution of equals under `<` (the congruence `a = b ∧ b < c ⟹ a < c`) is not among NAT-order's listed properties.
**ASN**: T4c Exhaustion (at `m = 0`): "the case `zeros(t) < 0` is excluded by NAT-zero's `0 ≤ zeros(t)` — unfolding ≤ to `0 < zeros(t) ∨ 0 = zeros(t)`, either disjunct together with `zeros(t) < 0` yields `0 < 0` **via NAT-order transitivity**, contradicting NAT-order irreflexivity". The same justification is reused at `m = 1` ("via the same transitivity-and-irreflexivity route"), at `m = 2` ("similarly"), and at `m = 3` ("by the same route"/"symmetrically").
**Issue**: When `≤` is unfolded to `< ∨ =`, two sub-cases arise. In the strict disjunct (`0 < zeros(t) ∧ zeros(t) < 0`), transitivity does yield `0 < 0`. In the equality disjunct (`0 = zeros(t) ∧ zeros(t) < 0`), the step to `0 < 0` is substitution of equals under `<`, not transitivity — a congruence inference that is *not* among the properties NAT-order's axiom actually lists (irreflexivity, transitivity of `<`, trichotomy). The exhaustion step reuses this pattern four times, always naming transitivity as the sole justification. A precise reader auditing which NAT-order clauses license each step cannot match the equality-disjunct inference to any of NAT-order's stated properties.
**What needs resolving**: Either extend NAT-order's axiom to include substitution of equals under `<` (or name it as a meta-inference available from equality), split the case analysis in T4c's Exhaustion so the equality disjunct is handled separately (e.g., by eliminating `0 = zeros(t)` directly rather than combining it with `zeros(t) < 0`), or eliminate the `= ` disjunct first (say, by observing that `0 = zeros(t) ∧ zeros(t) < 0` is refuted by irreflexivity alone after substitution — which is the same gap under a different name).

### Terminology drift between T4's prose ("user account") and T4c's formal labels ("user address")
**Foundation**: the ASN should fix one vocabulary for each concept and use it throughout
**ASN**: T4 prose: "A tumbler with zero zeros addresses a node. One zero: **a user account**. Two zeros: **a document**. Three zeros: **an element**. The count of zero-valued components determines the specificity level." T4c Postconditions: "`(A t ∈ T : t is T4-valid :: (zeros(t) = 0 ↔ t is a node address) ∧ (zeros(t) = 1 ↔ t is **a user address**) ∧ (zeros(t) = 2 ↔ t is **a document address**) ∧ (zeros(t) = 3 ↔ t is **an element address**))`". T4c prose: "`t` is a *node address* iff `zeros(t) = 0`, a *user address* iff `zeros(t) = 1`, a *document address* iff `zeros(t) = 2`, and an *element address* iff `zeros(t) = 3`."
**Issue**: T4's informal level names ("a user account", "a document", "an element") and T4c's formal level labels ("user address", "document address", "element address") refer to the same concept — the class of T4-valid tumblers with a given zero count — but the vocabulary shifts between the two sections. In particular, "user account" (T4) versus "user address" (T4c) is a different noun, not just a missing modifier. A precise reader looking up what T4c means by "user address" finds only "addresses a user account" in T4's prose, with no explicit identification of the two. The renaming bridge "We write *node* for Nelson's *server* and *element* for his *contents*" covers the Nelson-to-formal translation but does not cover the T4-prose-to-T4c-formal translation.
**What needs resolving**: Either align T4's prose to T4c's formal labels (e.g., "zero zeros: a node address; one zero: a user address; ..."), align T4c's labels to T4's prose nouns (e.g., "t is a user account iff zeros(t) = 1"), or add an explicit identification sentence that ties the two vocabularies together.

### T0's Axiom implies `#t ≥ 1` informally but does not state it, while downstream claims cite T0 for this inequality
**Foundation**: T0 (CarrierSetDefinition). T0's stated *Axiom* is: "T is the set of all nonempty finite sequences over ℕ, equipped with length `#· : T → ℕ` and component projection `·ᵢ` whose index domain for each `a ∈ T` is `{1, …, #a} ⊆ ℕ`, with `aᵢ ∈ ℕ` at each `i ∈ {1, …, #a}`."
**ASN**: T4a Depends: "T0 (CarrierSetDefinition) — fixes the carrier as ℕ; supplies `#t ≥ 1` as the precondition for `#t − 1 ∈ ℕ`." T4b Depends: "T0 (CarrierSetDefinition) — supplies ℕ, and `#t ≥ 1` in case `k = 0`". T4a proof: "NAT-sub's conditional-closure clause applied at the hypothesis `#t ≥ 1` (T0) and `1 ∈ ℕ` gives `#t − 1 ∈ ℕ`". T4b proof: "`#t ≥ 1` (T0)".
**Issue**: T4a and T4b both cite T0 explicitly for the inequality `#t ≥ 1` — used to discharge NAT-sub's conditional-closure precondition for `#t − 1 ∈ ℕ` and to establish non-emptiness of the single field segment in case `k = 0`. T0's Axiom says `T` is "the set of nonempty finite sequences over ℕ"; "nonempty" is a natural-language adjective on the set description that implies `#t ≥ 1`, but the Axiom does not state an inequality over `#t` that downstream claims can cite directly. Every other arithmetic inequality in this ASN (`zeros(t) ≤ 3`, `s_k ≤ #t − 1`, `s_{j+1} ≥ s_j + 2`, `0 ≤ tᵢ`) is licensed by an explicitly quantified axiom; `#t ≥ 1` alone is licensed only by interpretation of "nonempty".
**What needs resolving**: Either extend T0's Axiom to state `(A t ∈ T :: #t ≥ 1)` explicitly alongside the set description, or recast T4a and T4b's NAT-sub / case-`k=0` steps so they cite T0's "nonempty" clause and translate to `#t ≥ 1` locally (making the step the citing claim performs, not a T0 postcondition).
