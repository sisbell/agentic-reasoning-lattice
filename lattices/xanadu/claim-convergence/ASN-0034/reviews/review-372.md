### NAT-discrete's axiom uses `≤` and `+1` without declared dependencies
**Foundation**: NAT-discrete (NatDiscreteness). Its Formal Contract has no Preconditions or Depends slot. Its axiom body: `(A m, n ∈ ℕ :: m < n ⟹ m + 1 ≤ n)`; equivalently `(A m, n ∈ ℕ :: m ≤ n < m + 1 ⟹ n = m)`.
**ASN**: The NAT-zero finding in previous reviews was resolved by rephrasing NAT-zero to use only `<` and `=` (self-contained). Under that same discipline, NAT-discrete's axiom invokes two further constructs: `≤`, which is *defined* only in NAT-order (`m ≤ n ⟺ m < n ∨ m = n`), and `m + 1`, which requires `+` on ℕ and the numeral `1 ∈ ℕ` — the unstated NAT-closure.
**Issue**: NAT-discrete is invoked by T4, T4a, T4b, and T4c to promote `m < n` to `m + 1 ≤ n`. But NAT-discrete's *own* axiom body presupposes symbols from NAT-order and (the unstated) NAT-closure, without a Depends slot declaring either. A precise reader auditing NAT-discrete as a standalone foundation claim cannot interpret `m + 1 ≤ n` without silently importing two other claims. The same pattern is then repeated in NAT-addcompat, whose left/right order-compatibility clauses use `≤` and `+` without declared dependencies, and whose strict-successor clause uses `+ 1` likewise.
**What needs resolving**: Either restate NAT-discrete in primitives it controls (e.g., drop the `≤` form and phrase purely in `<` and `=`, with the `+ 1` grounded by a stated NAT-closure), or add explicit Depends slots to NAT-discrete and NAT-addcompat citing the foundations they invoke.

### Pre-T4 prose asserts a single four-field canonical form, while T4's Axiom schema admits four different forms by zero count
**Foundation**: T4's Axiom schema — "for each `k ∈ ℕ` with `0 ≤ k ≤ 3` at which `zeros(t) = k`, the form is — `k = 0`: `t = N₁. ... .Nₐ`; `k = 1`: `t = N₁. ... .Nₐ . 0 . U₁. ... .Uᵦ`; `k = 2`: …; `k = 3`: …"
**ASN**: Pre-T4 prose: "An address tumbler has the form: `t = N₁. ... .Nₐ . 0 . U₁. ... .Uᵦ . 0 . D₁. ... .Dᵧ . 0 . E₁. ... .Eδ` where `0 < Nᵢ, 0 < Uⱼ, 0 < Dₖ, 0 < Eₗ` at every position. The four fields are: — Node field … — User field … — Document field … — Element field …"
**Issue**: The prose commits to a single form with all four fields present (the `zeros(t) = 3` case), introduced with definite wording "An address tumbler has the form …". T4's Axiom then gives four distinct canonical forms indexed by zero count, three of which have fewer than four fields. T4b's partiality (e.g., `U(t)` undefined when `zeros(t) = 0`) and T4c's level labels for zero counts 0/1/2 only make sense on the weaker reading. A reader working through the prose first will arrive at T4's Axiom expecting the 3-zero form and must silently broaden their understanding. The prose also lists the four fields as structural components without qualifying that a given tumbler may lack some.
**What needs resolving**: Either qualify the pre-T4 prose to indicate the four-field form is the maximal case (with partial forms at lower zero counts), replace the single form with the same per-`k` schema T4's Axiom uses, or annotate the field list as describing the positions available at zero count 3.

### T4 main body asserts "zero counts 0, 1, 2, 3" as a definite enumeration before T4c derives exhaustion
**Foundation**: T4c's Exhaustion paragraph, which derives `zeros(t) ∈ {0, 1, 2, 3}` on the T4-valid subdomain from the bound `zeros(t) ≤ 3` via iterated trichotomy and NAT-discrete; T4's own Axiom explicitly disclaims this pre-commitment ("Exhaustion of `zeros(t) ∈ {0, 1, 2, 3}` is not asserted here; T4c's Exhaustion paragraph derives it").
**ASN**: T4 main body: "The count of zero-valued components determines the specificity level — the zero counts 0, 1, 2, 3 correspond to four levels, for which T4c is the single definitional site, assigning the labels node address, user address, document address, and element address respectively."
**Issue**: T4's Axiom is carefully phrased to avoid exhaustion ("quantified per-`k` and not pre-committing to exhaustion"), with the `∈ {0, 1, 2, 3}` step deferred to T4c. But T4's main prose, one paragraph earlier, uses the definite phrase "the zero counts 0, 1, 2, 3" — a closed-form enumeration of zero-count values — paired with "correspond to four levels". This reads as asserting the exhaustion that T4's Axiom deliberately defers. A precise reader who takes T4's Axiom as authoritative finds it contradicted by the main body's earlier enumeration; a reader who takes the main body as authoritative finds T4c's Exhaustion redundant.
**What needs resolving**: Either rephrase the main body's "zero counts 0, 1, 2, 3" as a forward reference ("for each `k ∈ {0, 1, 2, 3}` at which a T4-valid tumbler has `zeros(t) = k`, T4c assigns …"), drop the enumeration in favour of "zero counts bounded by 3", or remove the parallel disclaimer from T4's Axiom (accepting the exhaustion as part of T4's stipulation).

### Foundation axioms are stated in the document after three claims that cite them
**Foundation**: structural — the four foundation axiom blocks NAT-zero, NAT-discrete, NAT-order, NAT-addcompat are each cited in the Depends slots of T4, T4a, and T4b
**ASN**: Document ordering, top to bottom: T0; hierarchical prose; T4; T4a; T4b; NAT-zero; NAT-discrete; NAT-order; NAT-addcompat; T4c. T4's Formal Contract Depends cites NAT-zero, NAT-discrete, NAT-order, NAT-addcompat; T4a cites the first three; T4b cites the first three.
**Issue**: T4, T4a, and T4b each invoke NAT-zero, NAT-discrete, NAT-order (and T4 additionally NAT-addcompat) in their proofs and Depends lists, but the axiom blocks stating these foundations appear *after* T4b in the document. A precise reader tracing "what does NAT-discrete assert?" from T4's proof must scan forward past T4, T4a, T4b to find the axiom block. Only T4c, at the end of the document, sits after the axioms it depends on. Within a single ASN, axiom statements should be introduced before the theorems that consume them — the ordering here inverts that discipline for three of the four theorems.
**What needs resolving**: Reorder the document so the foundation axiom blocks (NAT-zero, NAT-discrete, NAT-order, NAT-addcompat, and — once stated — NAT-closure, NAT-sub, NAT-card) appear before T4, or group them into a dedicated "Foundation Statements" section at the front.
