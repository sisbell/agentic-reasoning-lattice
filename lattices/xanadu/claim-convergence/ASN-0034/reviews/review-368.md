# Regional Review — ASN-0034/T4c (cycle 3)

### Seq(X) and ε introduced in T4b with no foundation axiom
**Foundation**: structural — the ASN goes to considerable lengths to ground each arithmetic operator and numeral in a named foundation claim (NAT-closure for `1 ∈ ℕ` and `+`, NAT-sub for partial `−`, NAT-addcompat for successor, NAT-card for `|·|`). Set- and type-level constructors should be treated with the same care.
**ASN**: T4b: "Write `Seq(X)` for the set of finite, possibly empty sequences over `X`, and `ε` for its unique zero-length element." The signature `fields : T ⇀ Seq(ℕ⁺) × Seq(ℕ⁺) × Seq(ℕ⁺) × Seq(ℕ⁺)` appears in T4b's prose and Definition; each projection has codomain `Seq(ℕ⁺)`; absence is represented by `ε ∈ Seq(ℕ⁺)`. T4b's Postconditions: "Each projection `N, U, D, E : T ⇀ Seq(ℕ⁺)` shares the T4-valid subdomain and returns a finite sequence over `ℕ⁺`".
**Issue**: `Seq(X)` is a type constructor — finite, possibly empty sequences over an arbitrary underlying set — and `ε` is a distinguished element of it. T0 axiomatizes only `T` (nonempty finite sequences over ℕ specifically) and supplies no general sequence-type constructor. None of the stated foundation claims (T0, NAT-zero, NAT-discrete, NAT-order, NAT-addcompat) gives `Seq(·)` or `ε` any axiomatic grounding. The prose "`Seq(·)` differs from T's construction only by adjoining `ε`" is a narrative comparison, not a derivation — T0 doesn't describe "T's construction" to be contrasted with. A reader cannot consult an axiom to verify that `Seq(ℕ⁺)` is a well-defined set, that `ε` is its unique zero-length element, or that the embedding of nonempty `Seq(ℕ⁺)` elements into `T` (used to ground `t.X₁ := (X(t))₁`) is licensed.
**What needs resolving**: Either introduce a foundation claim axiomatizing `Seq(X)` (its domain, its elements, the unique `ε`, and the embedding of nonempty `Seq(X) ⊆ X`-element sequences into `T`-like sets when `X ⊆ ℕ`) and cite it in T4b's Depends, or restate T4b without the `Seq(ℕ⁺)` codomain machinery using constructs already in scope.

### NAT-card cited by T4/T4a/T4b/T4c but never stated as an axiom block
**Foundation**: structural — each cited foundation claim needs a body somewhere in scope (the same concern previously raised for NAT-closure and NAT-sub)
**ASN**: T4 Depends: "NAT-card (NatFiniteSetCardinality) — axiomatizes `|·|` on subsets of every initial segment `{1, …, n} ⊆ ℕ` with codomain ℕ…". T4a Depends, T4b Depends, T4c Depends each also cite NAT-card as the source of `|·|` and its enumeration characterisation. The stated axiom blocks in this ASN are NAT-zero, NAT-discrete, NAT-order, and NAT-addcompat.
**Issue**: The prior cycle's finding flagged NAT-closure and NAT-sub as cited-but-not-stated. NAT-card is a third, distinct foundation claim in exactly the same position: its axiom (the "enumeration characterisation" identifying `|S|` with the length of the unique strictly increasing enumeration of `S`) is invoked by T4b to justify `k = zeros(t)`, by T4 to type `zeros(t) ∈ ℕ`, and by T4c in exhaustion — but no NAT-card axiom block appears in this ASN and the declared-depends list is empty. The prior finding's named list (`NAT-closure`, `NAT-sub`) does not include NAT-card, so the gap is not closed by resolving that finding alone.
**What needs resolving**: Either add a stated NAT-card (NatFiniteSetCardinality) axiom block in this ASN with explicit content (codomain, the enumeration characterisation, separation from `#·`), declare it as an external dependency with its statement quoted, or recast the definition of `zeros(t)` to avoid invoking the cardinality operator.

### T4c's Exhaustion uses trichotomy from NAT-order, but T4c's Depends credits NAT-order only with injectivity's chain
**Foundation**: NAT-order (NatStrictTotalOrder), whose axiom includes trichotomy: "for any `m, n ∈ ℕ`, exactly one of `m < n`, `m = n`, `n < m` holds"
**ASN**: T4c Exhaustion: "NAT-discrete supplies the missing step: applied at `m ∈ {0, 1, 2}`, each instance promotes `m < zeros(t)` to `m + 1 ≤ zeros(t)`, so from `0 < zeros(t)` obtain `1 ≤ zeros(t)`, from `1 < zeros(t)` obtain `2 ≤ zeros(t)`, and from `2 < zeros(t)` obtain `3 ≤ zeros(t)`; the upper bound `zeros(t) ≤ 3` blocks any further promotion. Hence `zeros(t) ∈ {0, 1, 2, 3}`." T4c Depends for NAT-order: "supplies transitivity to chain `0 < 1 < 2 < 3` and irreflexivity to exclude equality within the chain, so that `0, 1, 2, 3` are pairwise distinct for injectivity."
**Issue**: NAT-discrete alone produces only the conditional `m < zeros(t) ⟹ m + 1 ≤ zeros(t)` (or its equivalent form). To conclude `zeros(t) ∈ {0, 1, 2, 3}` from `0 ≤ zeros(t) ≤ 3`, the argument must case-split at each `m ∈ {0, 1, 2}` on whether `m < zeros(t)` or `m = zeros(t)` or `zeros(t) < m` — which is trichotomy from NAT-order's axiom. T4c's body does this implicitly ("from `0 < zeros(t)` obtain …") without naming the case-split, and T4c's Depends entry for NAT-order names only transitivity and irreflexivity in service of Injectivity. The Exhaustion use of NAT-order is therefore both implicit in the body and absent from the Depends role.
**What needs resolving**: Either extend T4c's Depends role for NAT-order to name trichotomy's use in Exhaustion (and make the case-split explicit in the proof), or recast Exhaustion to proceed without trichotomy — e.g., by citing an exhaustion axiom for ℕ directly or by using NAT-discrete's "equivalent" form together with stated case analysis.
