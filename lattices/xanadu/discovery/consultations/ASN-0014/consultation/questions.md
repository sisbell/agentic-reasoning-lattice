# Sub-Questions — Distributed Replication Consistency

**Inquiry:** What must a multi-server Xanadu deployment guarantee about the convergence of replicated global state? How do the system's structural properties constrain the replication model?

1. [nelson] Since content, once stored, can never be deleted or altered, does this mean two servers that each accept new material can never contradict each other — only be at different stages of completeness?
2. [nelson] Every piece of content receives a permanently unique address at the moment of creation. What must the system guarantee so that two servers, acting independently, never assign the same address to different content?
3. [nelson] If a user creates a document on one server and another user creates a link to that document's content from a different server, must both servers eventually agree that the link exists and points to the same content?
4. [nelson] When a new version of a document is created on one server, must every other server in the network eventually recognize that version as part of the document's history, or can servers legitimately disagree about which versions exist?
5. [nelson] Must the order in which independently created documents and links become visible be the same on every server, or does the design only require that the final accumulated state converge?
6. [nelson] If two users on different servers each create a new version of the same document simultaneously, does the design require that these versions be ordered, or can they coexist as parallel branches in the version history?
7. [nelson] A document built entirely from shared content — transclusions of material stored elsewhere — depends on that material being findable. What must the network guarantee about the availability of content that other documents reference?
8. [nelson] Does the principle that editing one document must leave all other documents unaffected simplify or constrain how servers propagate changes to each other?
9. [nelson] Links are permanent and their endpoints never change. Does this immutability mean a server can accept and store a link without coordinating with the server that holds the linked content?
10. [nelson] Must the system guarantee that a reader consulting any server in the network will eventually see every piece of content and every link that has been created anywhere, or is partial knowledge of the global state acceptable?
11. [gregory] When two separate backend processes each call `findisatoinsertmolecule` for the same document simultaneously, does the tumbler node-account hierarchy (the `1.NODE.0.ACCOUNT.0.DOC` prefix) provide sufficient partitioning to prevent I-address collisions, or was single-process execution the only mechanism preventing duplicate I-address allocation?
12. [gregory] If two replicas of the granfilade each append content under the same document's I-address range and later merge, does the append-only property guarantee that both sets of I-addresses coexist without conflict, or can overlapping allocations produce two different content bytes at the same I-address?
13. [gregory] Was the spanfilade's write-only design (no `deletespanf` exists) an intentional choice to make DOCISPAN entries safe for uncoordinated replication — meaning two servers adding entries for the same I-span in different documents would simply union without conflict — or was it purely a simplification for single-server operation?
14. [gregory] Since POOMs are the sole locus of destructive mutation (INSERT shifts, DELETE removes, REARRANGE reorders), and two concurrent POOM modifications to the same document produce order-dependent V-address layouts, did you or Ted ever discuss a mechanism for reconciling divergent POOM states — such as replaying the journal of operations rather than merging tree structures directly?
15. [gregory] Does the tumbler allocation scheme — where each node gets a distinct first-level address (`1.1`, `1.2`, etc.) and each account gets a distinct second-level address — represent an intentional design for federated operation where separate Xanadu servers would own disjoint tumbler prefixes, making granfilade and spanfilade entries from different servers automatically non-conflicting by construction?
