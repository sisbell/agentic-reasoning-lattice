### T0's NAT-addcompat description says compatibility with `<` but the body states `≥`-compatibility
**Foundation**: T0 (CarrierSetDefinition) — enumeration bullet: "NAT-addcompat (NatAdditionOrderAndSuccessor): left and right order compatibility of addition with `<`, and the strict successor inequality `n < n + 1`."
**ASN**: NAT-addcompat body: "Left order compatibility: `n ≥ p ⟹ m + n ≥ m + p` for every `m, n, p ∈ ℕ`. Right order compatibility: `n ≥ p ⟹ n + m ≥ p + m` for every `m, n, p ∈ ℕ`. Strict successor inequality: `n < n + 1` for every `n ∈ ℕ`." The formal contract repeats the `≥ ⟹ ≥` form for both compatibilities.
**Issue**: T0's enumeration announces NAT-addcompat as compatibility of addition "with `<`", i.e., strict-order compatibility `n < p ⟹ m + n < m + p`. What NAT-addcompat actually states are non-strict compatibility clauses `n ≥ p ⟹ m + n ≥ m + p` (and the mirror). These are distinct statements — neither subsumes the other without an auxiliary cancellation/injectivity argument — and the axiom body delivers the non-strict form. Every downstream consumer in the ASN (TumblerAdd's strict-advancement `aₖ + wₖ ≥ aₖ + 1` from `wₖ ≥ 1`, and its dominance sub-case `aₖ + wₖ ≥ 0 + wₖ` from `0 ≤ aₖ`) uses the non-strict form. The scope clause T0 publishes therefore over-describes the axiom in the direction callers don't need and under-describes it in the direction they do.
**What needs resolving**: Reconcile T0's enumeration with the axiom body — either correct the T0 bullet to describe NAT-addcompat as compatibility with `≥` (or "the order relation" in a form that matches the body's quantifier shape), or extend NAT-addcompat's body to include the strict-order compatibility clause the enumeration currently advertises.

### T0 attributes `0 ∈ ℕ` to NAT-zero but the NAT-zero claim body does not state it as axiom content
**Foundation**: T0 (CarrierSetDefinition) — enumeration bullet: "NAT-zero (NatZeroMinimum): `(A n ∈ ℕ :: 0 ≤ n)`; supplies `0 ∈ ℕ`."
**ASN**: NAT-zero body: "`0` is the minimum of ℕ: `(A n ∈ ℕ :: 0 ≤ n)`. In words: every natural number is at least `0`. Equivalently, `0` is a lower bound of ℕ — the least element under `≤`." Formal contract: `(A n ∈ ℕ :: 0 ≤ n)` (zero is a lower bound of ℕ). No clause of the contract asserts `0 ∈ ℕ`.
**Issue**: T0 explicitly lists `0 ∈ ℕ` as something NAT-zero *supplies*, treating it as a separate deliverable alongside the lower-bound statement. The axiom body delivers only the quantified `0 ≤ n` statement. Membership `0 ∈ ℕ` is implicitly presupposed by the well-formedness of `0 ≤ n` (since `≤` is defined on ℕ) and by NAT-closure's `(A n ∈ ℕ :: 0 + n = n)` (which forms `0 + n` in ℕ), but no claim in the ASN states it as axiom content that a downstream consumer could cite. A downstream site that needs "`0 ∈ ℕ`" directly — e.g., to instantiate an existentially-quantified ℕ-statement — cannot extract it from `(A n ∈ ℕ :: 0 ≤ n)` as written.
**What needs resolving**: Either extend NAT-zero's formal contract to state `0 ∈ ℕ` as an explicit axiom clause (matching T0's attribution), or rewrite T0's enumeration bullet so the "supplies `0 ∈ ℕ`" phrasing is dropped and membership is instead attributed to whichever axiom (or structural definition) actually carries it.

### NAT-wellorder's body contains a defensive independence-via-counterexample essay in a formal-axiom slot
**Foundation**: NAT-wellorder (NatWellOrdering) — stated as a foundation axiom supplying the least-element principle.
**ASN**: NAT-wellorder body, final paragraph: "Well-ordering is independent: strict total order and discreteness alone do not entail it — ℤ with the usual `<` is strictly totally ordered and discrete but not well-ordered."
**Issue**: No claim in this ASN uses this independence statement — not in T1, T3, TA-Pos, ActionPoint, or TumblerAdd. It is not part of NAT-wellorder's formal contract. It is defensive meta-prose answering an unasked "why can't this be derived from the other NAT-* axioms?" — a justification for the axiomatisation design, delivered via a ℤ counterexample that lives outside the ASN's carrier domain. A reader following the formal contract must skip past this paragraph to reach the next claim, and nothing in the proof text downstream cites it. This is distinct from the PF-flagged axiom-body prose that cited specific (absent) downstream proofs: this paragraph cites only ℤ, which has no role in the ASN at all, so the resolution direction is removal rather than disclosure of load-bearing consumers.
**What needs resolving**: Either relocate the independence observation out of NAT-wellorder's claim body (e.g., into a commentary section adjacent to the axiom, or a meta-note explaining the axiomatisation choice), or remove it — NAT-wellorder's formal-contract slot should carry only the axiom content, not essay prose about why alternative axiomatisations would not suffice.
