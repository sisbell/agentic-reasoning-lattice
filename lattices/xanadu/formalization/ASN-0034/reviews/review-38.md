### T8 — referenced substantively but absent from the ASN
**Foundation**: (internal consistency — dependency chain completeness)
**ASN**: T9 section: *"T8 tells us that addresses, once allocated, are permanent."* Also: *"Positions on the tumbler line that have been allocated but have no stored content are what Nelson calls 'ghost elements' (T8 above)."* Also: *"A consequence of T8 and T9 together: the set of allocated addresses is a growing set in the lattice-theoretic sense."*
**Issue**: T8 is referenced three times in the T9 section — as a named property ("T8 tells us"), as a positional reference ("T8 above"), and as a co-dependency of an informal theorem ("A consequence of T8 and T9 together"). Yet T8 has no property block anywhere in the reviewed content. It is not listed in the declared-depends metadata (which is empty). T8 is distinct from the properties covered by previous findings: it is neither a foundational definition (finding #2 covers T0, Prefix), nor a dangling side-reference (finding #3 covers T6, T7), nor a sub-property of T4 or TA5 (finding #4). T8 is not used in any formal proof — T9's proof depends only on TA5(a) and T1(c), and GlobalUniqueness does not reference T8. However, the growing-set consequence is stated as a factual claim depending on T8, and the ghost-element characterization links T8 to the system's semantic model of allocated-but-empty positions.
**What needs resolving**: T8 needs a formal property block within this ASN (the reference "T8 above" implies it belongs here, and the numbering places it between T4 and T9), or the declared-depends must cite the ASN that defines it. The growing-set consequence — that the set of allocated addresses only grows, with new elements at each allocator's frontier — should either be a formal corollary with T8 as a stated precondition, or be marked as an informal remark.

### GlobalUniqueness formal invariant excludes the root's initial base address
**Foundation**: (internal consistency — formal statement vs. proof scope)
**ASN**: GlobalUniqueness proof: *"An allocation event is any invocation of `inc(t, k)` within a conforming system."* Formal contract invariant: *"For every pair of addresses `a, b` produced by distinct allocation events in any reachable system state: `a ≠ b`."* T10a axiom: *"The root allocator begins from a base address satisfying T4."* Domain definition: *"`dom(A) = {tₙ : n ≥ 0}` where `tₙ₊₁ = inc(tₙ, 0)`."*
**Issue**: The root allocator's base address `t₀` is produced by initialization, not by any invocation of `inc`. Under the stated definition of "allocation event," `t₀` is not an allocation event. The formal invariant — scoped to "addresses produced by distinct allocation events" — therefore excludes every pair `(t₀, x)`. The proof handles `t₀` correctly: `t₀` is the `n = 0` element of `dom(root)` by the domain definition; Case 1 covers `t₀` vs. later root siblings via T9 (which applies to all domain members including the base); Cases 3/4 cover `t₀` vs. non-root outputs via length separation (root outputs have length `γ`, non-root outputs have length `> γ` by T10a.3). The proof establishes distinctness for all domain members, but the formal invariant claims it only for `inc` outputs. Additionally, the exhaustiveness claim — *"The domain taxonomy assigns every address to exactly one allocator"* — quantifies over "every address," which is broader than "every `inc` output," but the taxonomy rule (*"every `inc` output belongs to exactly one domain — sibling outputs to the producing allocator's domain, child-spawning outputs to the child's domain"*) covers only `inc` outputs; `t₀`'s membership in `dom(root)` comes from the domain definition, not from the taxonomy rule, leaving a gap in the chain that justifies "every address."
**What needs resolving**: The formal invariant should be broadened to cover all addresses in any allocator domain (matching the scope of the proof), or the definition of "allocation event" should be extended to include the root initialization. The exhaustiveness claim should explicitly address the root's base — either by noting that domain membership (not the taxonomy rule) assigns it, or by incorporating the root initialization into the taxonomy.
