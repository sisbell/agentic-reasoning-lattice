### Cancellation of addition on ℕ is used in proofs but not axiomatized

**Foundation**: (foundation ASN — internal consistency; the NAT-* axiom suite is intended to carry ℕ-level facts per T0's stated convention)
**ASN**: TA-LC (LeftCancellation) invokes cancellation three times. Case `k₁ < k₂`: *"From `a ⊕ x = a ⊕ y` we obtain `a_{k₁} + x_{k₁} = a_{k₁}`, hence `x_{k₁} = 0`"* — derives `x = 0` from `a + x = a` (cancellation via additive identity). At position `i = k`: *"From `a ⊕ x = a ⊕ y` we get `a_k + x_k = a_k + y_k`, hence `x_k = y_k` by cancellation in ℕ."* TA-MTO (ManyToOne) does the same at position `i = k`: *"`aₖ + wₖ = bₖ + wₖ`, hence `aₖ = bₖ` by cancellation in ℕ."* None of NAT-closure, NAT-order, NAT-discrete, NAT-addcompat, NAT-wellorder, or NAT-addassoc states left or right cancellation of addition on ℕ.
**Issue**: Two properties explicitly invoke "cancellation in ℕ" as an axiomatic step, yet no NAT-* axiom supplies it. The convention set by T0 ("each proof cites only the ℕ facts it actually uses") is followed scrupulously for discreteness, well-ordering, order-compatibility, strict successor, and associativity, but the cancellation fact has no citable source. A reader cannot reconstruct TA-LC or TA-MTO from the axiom set as stated.
**What needs resolving**: Either add a NAT-* axiom stating cancellation of addition on ℕ (and have TA-LC and TA-MTO cite it), or derive cancellation from the existing NAT-* suite inside the proofs that need it (NAT-order trichotomy combined with NAT-addcompat can reject equality in `a + c = b + c` given `a ≠ b`, if commutativity is available — see next finding).

### Commutativity of addition on ℕ is used but not axiomatized

**Foundation**: (foundation ASN — internal consistency)
**ASN**: NAT-addcompat is stated one-sidedly: *"`(A m, n, p ∈ ℕ : n ≥ p : m + n ≥ m + p)`"* — `m` is always added on the left. Several proofs require the symmetric right-addition form. TA1 sub-case `j = k`: *"Since `aₖ < bₖ`, NAT-addcompat's order-compatibility of addition with NAT-order's trichotomy promotes the strict ℕ-inequality through the shared increment `wₖ`, yielding `aₖ + wₖ < bₖ + wₖ`."* Here `wₖ` is added on the right of variable summands `aₖ`, `bₖ`; NAT-addcompat as written gives only `wₖ + aₖ ≤ wₖ + bₖ`. TA-LC's final step *"`#x = #(a ⊕ x) = #(a ⊕ y) = #y`"* implicitly chains equalities across right-added quantities, and TA-MTO's `aₖ + wₖ = bₖ + wₖ` cancellation step presupposes the two forms of addition are identified.
**Issue**: NAT-addcompat's literal statement is left-addition only. Every proof that promotes a strict inequality through a right-added fixed term — or that interchanges `a + c` with `c + a` — tacitly assumes commutativity. No NAT-* axiom states it.
**What needs resolving**: Either state NAT-addcompat in a form symmetric over left/right addition (covering both `m + n ≥ m + p` and `n + m ≥ p + m`), or add a separate commutativity axiom and have proofs that need the right-addition form cite it explicitly.

### Broad pattern of missing Depends clauses beyond T12 and TA-assoc

**Foundation**: (internal consistency — Previous Finding 2 named only T12 and TA-assoc; both now carry Depends)
**ASN**: The following formal contracts carry proofs that invoke other properties but omit a Depends line: D2, OrdinalDisplacement, OrdinalShift, ReverseInverse, T4a, T4b, T4c, T6, T7, TA-LC, TA-MTO, TA-RC, TA-strict, TA1-strict, TA3-strict, TA4, TA5-SigValid, TA5a, TA7a, TS1, TS2, TS3, TS4, TS5. For example, TS3's proof explicitly cites OrdinalShift, TumblerAdd, TA0, T3, and NAT-addassoc; its Formal Contract lists only Preconditions, Postconditions, and Frame. TA4's proof invokes TumblerAdd, TA0's result-length identity, TumblerSub, T1, T3, and implicitly the action-point definition; none is named in its Formal Contract.
**Issue**: The citation convention that Depends should enumerate every property a proof invokes is applied to some properties (ActionPoint, TA0, TumblerAdd, TA5, TA-Pos, TumblerSub, T1, T3, T4, T5, T8, T9, T10, T10a, T10a-N, T10a.2, T10a.4, T10a.5, T10a.6, T10a.7, T12, TA-assoc, TA1, TA2, TA3, TA6, Prefix, PrefixOrderingExtension, ZPD, Divergence, AllocatedSet, GlobalUniqueness, PartitionMonotonicity, D0, D1) but not to others (the list above). A reviser tightening preconditions on, say, TumblerAdd has Depends-backed visibility into which properties consume the tightened clause — except for the ~24 properties whose consumption is invisible to that audit. The coverage gap is substantively larger than Previous Finding 2 indicates.
**What needs resolving**: Either extend the citation convention uniformly so every non-axiom formal contract carries a Depends clause matching the level of granularity used in properties like TA0, TA5, and T1 — or declare that Depends is optional and document what role it plays when present. Mixed adherence across ~60 properties undermines the role Depends plays in validating precondition chains across the ASN.
