### TA-Pos's divergence example does not involve `0` as an operand, so it doesn't witness the stated `Pos(t)` vs `t > 0` divergence
**Foundation**: TA-Pos (PositiveTumbler) — body motivates the choice of `Pos` notation by arguing that `Pos(t)` and `t > 0` diverge.
**ASN**: TA-Pos body: "The predicate `Pos(t)` is not written `t > 0`, because `>` already denotes T1's lexicographic ordering, and the two notions diverge. The zero tumbler `0.0.0` exhibits the divergence: under T1, `0.0 < 0.0.0` by the prefix rule (the length-2 tumbler `0.0` is a proper prefix of the length-3 tumbler `0.0.0`), yet `Zero(0.0.0)` holds."
**Issue**: The stated divergence is between `Pos(t)` and `t > 0` — a comparison of `t` against the specific tumbler `0` on the right. The witness, however, is `0.0 < 0.0.0`: neither operand is `0`. What the example establishes is "`0.0.0` is T1-greater than `0.0` while `Zero(0.0.0)` holds" — a divergence between `Zero` and "T1-greater-than some other zero tumbler", not between `Pos(t)` and `t > 0`. A tight witness would be `0 < 0.0.0` under T1 with `Zero(0.0.0)` (showing `0.0.0 > 0` yet ¬Pos(0.0.0)), or equivalently `0 < 0.0` with `Zero(0.0)`. The current example, presumably chosen to dodge the prior `0`-overload issue, gains clean operand typing at the cost of no longer aligning with the claim it is cited to support.
**What needs resolving**: Either rephrase the motivation so "the two notions diverge" refers to the broader Zero-vs-T1-order divergence the example actually exhibits, or replace the example with one whose right-hand side of `<` is the tumbler `0` so the witness directly matches `t > 0`.

### ActionPoint's derivation of `1 ∈ ℕ` rests on an unlicensed identification `1 denotes 0 + 1`
**Foundation**: NAT-closure (NatArithmeticClosureAndIdentity), NAT-zero (NatZeroMinimum) — the axioms whose consumption yields `1 ∈ ℕ` in the derivation.
**ASN**: ActionPoint derivation: "NAT-zero supplies 0 ∈ ℕ, and NAT-closure's successor closure `(A n ∈ ℕ :: n + 1 ∈ ℕ)` at n = 0 yields `0 + 1 ∈ ℕ`, which is `1 ∈ ℕ` since 1 denotes 0 + 1. The additive identity at n = 1 then gives `0 + 1 = 1`, rewriting `0 + 1 ≤ w_{actionPoint(w)}` to `1 ≤ w_{actionPoint(w)}`."
**Issue**: No claim in the ASN defines the numeral `1` — neither as a primitive element of ℕ nor as notation for `0 + 1`. The numeral appears in NAT-discrete's `m + 1 ≤ n`, NAT-closure's `n + 1`, and the TA-Pos / ActionPoint post-conditions `≥ 1`, but nowhere is it tied to stated content. The derivation here asserts "1 denotes 0 + 1" as a notational identification that is unlicensed. Further, that identification makes the subsequent rewrite internally awkward: if `1` is definitionally `0 + 1`, then `0 + 1 = 1` is textually the trivial equality `0 + 1 = 0 + 1`, and "rewriting `0 + 1 ≤ w_{actionPoint(w)}` to `1 ≤ w_{actionPoint(w)}`" is a substitution of name for expression rather than an application of the additive identity. If instead `1` is a primitive numeral, the step `0 + 1 ∈ ℕ` ⟹ `1 ∈ ℕ` requires a stated rule mapping the numeral to the expression. Either reading leaves the load-bearing conclusion `1 ∈ ℕ` standing on content the ASN does not commit to.
**What needs resolving**: Establish the status of the numeral `1` in stated content — e.g., extend some axiom to assert `1 ∈ ℕ` directly, or introduce a definitional clause (in T0 or a NAT-* claim) identifying `1` with `0 + 1` (or with `S(0)`) and reconcile the subsequent "additive identity" step accordingly so the derivation's two uses of `1` cohere.
