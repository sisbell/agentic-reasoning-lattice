# Regional Review — ASN-0034/ActionPoint (cycle 5)

Reading the current ASN for new cross-claim issues not in Previous Findings.

### `min` is used in ActionPoint's definition but no claim formally introduces the `min` operator
**Foundation**: NAT-wellorder (NatWellOrdering) — formal contract: `(A S : S ⊆ ℕ ∧ S ≠ ∅ : (E m ∈ S :: (A n ∈ S :: m ≤ n)))`. The body prose mentions "This least-element principle is what makes `min(S)` well-defined whenever `S` is nonempty."
**ASN**: ActionPoint formal contract — Definition: "actionPoint(w) = min({i : 1 ≤ i ≤ #w ∧ wᵢ ≠ 0})." The derivation says "By NAT-wellorder, this set has a least element, which min names."
**Issue**: `min` appears as a named operator in ActionPoint's Definition clause — a load-bearing piece of the formal contract. But no axiom's formal contract introduces `min`: NAT-wellorder's contract states only the existential least-element principle; it does not define `min : P(ℕ) → ℕ` as the operator that picks that element. NAT-wellorder's body prose mentions `min(S)` being "well-defined" but that sentence is commentary, not contract content. ActionPoint's derivation glosses over the gap by saying "the least element, which min names" — a notational identification introduced at the point of use rather than a stated rule. A precise reader has no claim to resolve `min` against; the definition of `actionPoint` therefore stands on an operator the ASN's formal contracts do not supply.
**What needs resolving**: Either extend NAT-wellorder's formal contract (or another axiom) to introduce `min` explicitly — e.g., as the function yielding the least element whose existence the principle asserts — or rewrite ActionPoint's Definition to avoid `min`, e.g., "actionPoint(w) is the element m ∈ S such that (A n ∈ S :: m ≤ n) where S = {i : 1 ≤ i ≤ #w ∧ wᵢ ≠ 0}" so the definition expands against stated axiom content rather than an unreified operator.

### ActionPoint's postcondition uses `≥` but no axiom introduces the relation
**Foundation**: NAT-order (NatStrictTotalOrder) — formal contract introduces `<` (with irreflexivity, transitivity, trichotomy) and defines `≤` via `m ≤ n ⟺ m < n ∨ m = n`. No clause introduces `≥`.
**ASN**: ActionPoint formal contract Postconditions: "1 ≤ actionPoint(w) ≤ #w; wᵢ = 0 for all 1 ≤ i < actionPoint(w); w_{actionPoint(w)} ≥ 1." The derivation concludes with `1 ≤ w_{actionPoint(w)}` and the postcondition restates this as `w_{actionPoint(w)} ≥ 1`.
**Issue**: NAT-order's formal contract establishes `<` and `≤` on ℕ. The symbol `≥` is not defined anywhere in the ASN — no axiom names it as notation for `n ≥ m ⟺ m ≤ n`, and NAT-order's contract does not mention it. ActionPoint's third postcondition `w_{actionPoint(w)} ≥ 1` is a commitment against a relation the ASN has not introduced; the derivation produces `1 ≤ w_{actionPoint(w)}` and the postcondition silently flips the operands via an undefined symbol. A downstream consumer citing this postcondition has no stated rule to translate `≥` back into the `≤`-form the derivation actually proved.
**What needs resolving**: Either extend NAT-order's formal contract to introduce `≥` (and, if used, `>`) as defined notation for the reversed relation, or rewrite ActionPoint's postcondition to use the `≤`-form the derivation supplies (`1 ≤ w_{actionPoint(w)}`) so the postcondition is stated in symbols the ASN has defined.

### TA-Pos's Definition uses the literal `0` but the Depends list does not cite the axiom that introduces it
**Foundation**: NAT-zero (NatZeroMinimum) — formal contract: "`0 ∈ ℕ` (zero is a natural number); `(A n ∈ ℕ :: 0 ≤ n)`." This is the claim in the ASN that establishes the natural-number literal `0`.
**ASN**: TA-Pos formal contract — Definition: "`Pos(t)` iff `(E i : 1 ≤ i ≤ #t : tᵢ ≠ 0)`; `Zero(t)` iff `(A i : 1 ≤ i ≤ #t : tᵢ = 0)`; **Z** = {t ∈ T : Zero(t)}." Depends: "T0 (CarrierSetDefinition) — carrier `T`, length `#t`, component projection `tᵢ`."
**Issue**: TA-Pos's Definition clause references the natural-number literal `0` three times — in `tᵢ ≠ 0`, `tᵢ = 0`, and implicitly in the definition of `Z` via `Zero(t)`. The canonical source of `0` as an element of ℕ is NAT-zero's `0 ∈ ℕ` clause. T0's axiom body and formal contract introduce only `T`, `#·`, and `·ᵢ`; T0 does not mention `0`. TA-Pos therefore cites T0 for notation it does supply (`T`, `#t`, `tᵢ`) but uses the literal `0` without citing the axiom that licenses it. ActionPoint's Depends list, by contrast, correctly includes NAT-zero because its derivation invokes `0 ≤ w_{actionPoint(w)}` — but ActionPoint also imports `Pos(w)` from TA-Pos, and the `Pos` predicate itself rests on `0` without a dependency trace back through TA-Pos to NAT-zero. The Depends list is under-declared for the symbols the Definition actually uses.
**What needs resolving**: Either add NAT-zero to TA-Pos's Depends list (with a pointer phrase like "supplies `0 ∈ ℕ` for the literal appearing in `tᵢ = 0`/`tᵢ ≠ 0`"), or — if `0` is meant to be a primitive numeral not requiring an axiom citation — say so explicitly at the point it is first used, so TA-Pos's Depends list is either complete with respect to the symbols the Definition uses or the status of `0` as unaxiomatized primitive is declared content.
