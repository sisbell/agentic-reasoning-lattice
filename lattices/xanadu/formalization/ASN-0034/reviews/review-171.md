# Cone Review — ASN-0034/TA6 (cycle 5)

*2026-04-17 15:20*

### T4c postcondition omits explicit domain restriction to T4-valid tumblers

**Foundation**: T4 (HierarchicalParsing), Formal Contract *Postconditions* (T4c).

**ASN**: T4's Axiom establishes the bound `zeros(t) ≤ 3` as a condition on *valid address tumblers*, not on all of T — "Valid address tumblers satisfy the following positional conditions on the component sequence... `zeros(t) ≤ 3`". T4's Postcondition for T4c then states: "T4c (LevelDetermination): `zeros(t)` bijects with hierarchical level — `zeros(t) = 0` ↔ node address, `zeros(t) = 1` ↔ user address, `zeros(t) = 2` ↔ document address, `zeros(t) = 3` ↔ element address." The body prose gloss carries the same form: "`zeros(t) = 0` ↔ node address (node field only); `= 1` ↔ user address...". Contrast this with T4b's Postcondition, which fixes its object explicitly: "whose domain is exactly the T4-valid subset of `T`". The justification paragraph for "T4c requires no inter-consequence precondition" also hangs on T4's axiomatic bound — "both the zero↔separator biconditional and the bound `zeros(t) ≤ 3` that drive its bijection are clauses of T4's own axiom" — which itself is a clause constraining valid addresses, not all tumblers.

**Issue**: T4c is stated without a quantifier or domain restriction. For a generic `t ∈ T`, `zeros(t)` can be any value in ℕ — tumblers like `[1, 0, 0, 2]` have `zeros(t) = 2` but violate T4 (adjacent zeros), and tumblers with four or more non-adjacent zeros have `zeros(t) ≥ 4` and fall outside the enumeration. The four biconditionals T4c asserts are statements about T4-valid tumblers only — the axiomatic bound `zeros(t) ≤ 3` that T4c's justification leans on is axiomatically scoped to valid addresses, not universally to T. Without an explicit domain clause, T4c either (i) quantifies over all of T, in which case the enumeration is non-exhaustive (no case for `zeros(t) ≥ 4`) and the labels "node address"/etc. must be given meaning for invalid tumblers, or (ii) quantifies implicitly over T4-valid tumblers, in which case that restriction is load-bearing for both the exhaustivity of the four cases and the existence of the bound T4c's justification cites — and should be recorded in the postcondition alongside T4b's parallel "T4-valid subset of T" specification. Downstream ASNs citing T4c cannot tell from the contract which reading applies, nor can they tell whether a citation at an arbitrary `t ∈ T` is licensed or requires a prior T4-validity check.

**What needs resolving**: Make T4c's domain explicit in the Formal Contract. Either (a) rewrite T4c's postcondition with a universal quantifier that carries a T4-validity guard, e.g., `(A t ∈ T : t is T4-valid :: zeros(t) = 0 ↔ node address ∧ ...)`, paralleling T4b's "domain is exactly the T4-valid subset of T" treatment; or (b) if the intended quantification is over all of T, extend T4c's enumeration to cover `zeros(t) ≥ 4` (likely as "addresses no level" / not an address) and state the labels' meaning on tumblers that fail T4's positional conditions. Either way, the implicit restriction that T4c's justification paragraph currently relies on needs to surface in T4c's own exported statement.
