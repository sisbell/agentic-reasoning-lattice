### T4b's Depends (and T4's Preconditions clause) mischaracterize where T4a is consumed, routing it through a case analysis that never cites it
**Foundation**: T4a (SyntacticEquivalence) — body proves the equivalence between T4's positional field-segment constraint and segment non-emptiness, but T4b's case analysis on `k = zeros(t) ∈ {0, 1, 2, 3}` derives non-emptiness directly from T4's clauses. **ASN**: T4's Formal Contract, Preconditions — "T4b (UniqueParse) also requires T4a (SyntacticEquivalence) — T4a converts T4's positional field-segment constraint (no two adjacent zeros, `t₁ ≠ 0`, `t_{#t} ≠ 0`) into the equivalent segment-non-emptiness reading that T4b's case analysis on `zeros(t) ∈ {0, 1, 2, 3}` consumes when it certifies each prescribed-non-empty projection as having at least one source position." **Issue**: The case analysis never invokes T4a. Each of the four cases derives non-emptiness by *direct* appeal to T4's positional clauses (`t₁ ≠ 0`, `t_{#t} ≠ 0`, no adjacent zeros, and `#t ≥ 1` from T0 for k=0), without routing through T4a's equivalence. The closing remark that pairs T4 with T4a reads as bibliographic restatement, not a proof step: the "Consequently…" sentence summarises what the cases already established. Genuine load-bearing use of T4a in T4b is confined to the absence-marker clause, where the reverse direction of "`X(t) = ε` iff X is absent" leans on "every present field segment is non-empty." So T4's parent description of T4b's T4a-dependency (and T4b's own self-description of it) attributes T4a's role to the wrong portion of the proof, and the claim that T4b's case analysis "consumes" T4a into "prescribed-non-empty projection… certification" is not borne out by the text. Under per-property review each proof looks fine (case analysis correctly uses T4; absence clause correctly uses T4a), but the cross-cutting mismatch is that the DAG rationale pinned on the case analysis is exactly where the citation does not occur.
