# Cone Review — ASN-0034/TA-strict (cycle 7)

*2026-04-18 01:27*

### TA-PosDom is referenced as load-bearing but not defined in the ASN content
**Foundation**: TA-Pos (this ASN) — "The converse direction does hold — `Pos(t)` implies `t` is T1-greater than every zero tumbler — and is established downstream as TA-PosDom (PositiveDominatesZero), which consumes `actionPoint(t)` as its T1 divergence witness and therefore depends on ActionPoint."
**ASN**: The content shown contains exactly six property blocks — TumblerAdd, TA0, TA-Pos, ActionPoint, T1, TA-strict. No `TA-PosDom (PositiveDominatesZero)` block is present. The promise is also load-bearing for TA-Pos's own cycle-resolution argument: "Splitting TA-PosDom off from TA-Pos keeps TA-Pos's own Depends (T0 only) acyclic with ActionPoint's … the prior mutual citation TA-Pos ↔ ActionPoint … is now carried by two distinct Depends edges on distinct properties (TA-Pos → ActionPoint via ActionPoint's `Pos(w)` precondition, and ActionPoint → TA-PosDom via TA-PosDom's proof consuming `actionPoint(t)`)."
**Issue**: The old mutual citation TA-Pos ↔ ActionPoint (Previous Finding #6 / #11) was supposed to be broken by *splitting off* TA-PosDom so that the `Pos(t) ⟹ z < t` theorem lives on its own property with its own Depends edge to ActionPoint. TA-Pos's text now reads as if that split has occurred — the definition block has had its theorem half excised, the proof section is gone, and the formal contract has no Postconditions field — but the target of the excision does not appear in the ASN. A reader verifying "the converse does hold" has no property to consult; a downstream consumer needing `z < t` has no labelled handle to cite; the DAG edge `ActionPoint → TA-PosDom` (or the other direction, see next finding) is claimed but has no source node at one end. The cycle-resolution argument rests on a property that does not exist in the ASN.
**What needs resolving**: Either add the TA-PosDom property to the ASN (with its statement, proof consuming `actionPoint(t)` as T1 divergence witness, and formal contract citing T0, T1, TA-Pos, ActionPoint, plus whichever NAT-* axioms the proof uses), or remove the references to TA-PosDom from TA-Pos and supply an alternative account of how the TA-Pos ↔ ActionPoint cycle is broken. In the current state, the cycle is not broken — it is relocated to a phantom node.

### Arrow directions in TA-Pos's cycle-resolution prose contradict the cited Depends edges
**Foundation**: TA-Pos (this ASN) Depends lists only T0; ActionPoint (this ASN) Depends lists TA-Pos.
**ASN**: TA-Pos prose: "the prior mutual citation TA-Pos ↔ ActionPoint … is now carried by two distinct Depends edges on distinct properties (**TA-Pos → ActionPoint** via ActionPoint's `Pos(w)` precondition, and **ActionPoint → TA-PosDom** via TA-PosDom's proof consuming `actionPoint(t)`)".
**Issue**: In the current Depends metadata, ActionPoint cites TA-Pos (because ActionPoint's precondition needs `Pos(w)`), so the depends-on edge runs ActionPoint → TA-Pos. The prose writes it as "TA-Pos → ActionPoint via ActionPoint's `Pos(w)` precondition" — but the reason given (*ActionPoint's precondition*) is a reason for ActionPoint to depend on TA-Pos, not the reverse. The mirror claim "ActionPoint → TA-PosDom via TA-PosDom's proof consuming `actionPoint(t)`" exhibits the same reversal: TA-PosDom's proof consuming `actionPoint(t)` is a reason for TA-PosDom to depend on ActionPoint, so the edge is TA-PosDom → ActionPoint, not the written direction. Either both arrows are written with the convention "X → Y means Y depends on X" (the reverse of the usual topological-sort reading), in which case the ASN must declare that convention somewhere the reader can find it, or the directions are simply inverted. Compounding the confusion, the very purpose of the prose is to argue that the old cycle is now two *distinct* directed edges — a claim whose verifiability requires the arrow direction to be unambiguous.
**What needs resolving**: Reconcile the prose arrow directions with the actual Depends edges. If "X → Y" is intended to mean "X depends on Y" (the convention under which a depends-on DAG's topological order has dependencies earlier), the two edges should be written "ActionPoint → TA-Pos via ActionPoint's `Pos(w)` precondition" and "TA-PosDom → ActionPoint via TA-PosDom's proof consuming `actionPoint(t)`". If the opposite convention is intended, declare it. Without one of these fixes, a reader checking that the DAG is acyclic cannot determine from the prose which way the edges run.

### Zero tumbler has no predicate symbol while `Pos` does, forcing every downstream reference to spell out the universal
**Foundation**: TA-Pos (this ASN) Definition: "A tumbler `t ∈ T` is *positive*, written `Pos(t)`, iff at least one of its components is nonzero: `(E i : 1 ≤ i ≤ #t : tᵢ ≠ 0)`. A tumbler is a *zero tumbler* iff every component is zero: `(A i : 1 ≤ i ≤ #t : tᵢ = 0)`."
**ASN**: TA-Pos's own prose references "every zero tumbler" (in the TA-PosDom motivation), "the all-zero tumbler `[0, 0]`", "`¬Pos([0, 0])`" (as the substitute for "is a zero tumbler"). Downstream consumers (TumblerAdd's *dominance* paragraph, TA-strict's motivational paragraph about "no-op model") reference the concept obliquely without a notation.
**Issue**: The positive predicate is introduced with a symbol (`Pos(t)`) that downstream properties bind in preconditions (TumblerAdd's `Pos(w)`, ActionPoint's `Pos(w)`, TA0's `Pos(w)`, TA-strict's `Pos(w)`, etc.). The dual predicate — zero tumbler — is introduced as a *named concept* without a predicate symbol. A property wishing to state "for every zero tumbler `z`, `z < t`" must spell out the universal `(A i : 1 ≤ i ≤ #z : zᵢ = 0)` inline each time it is used, because there is no symbol to bind. This asymmetry is what forces TA-Pos's own prose to substitute `¬Pos([0, 0])` (i.e., the negation of the sibling predicate) as the stand-in when it needs to speak about `[0, 0]` being a zero tumbler — a workaround that conflates "is a zero tumbler" with "is not positive" even though the latter is equivalent only because `t ∈ T` forces `#t ≥ 1`. Downstream TA-PosDom (once defined) will inherit the notational gap: its statement will need either to spell out the universal, to appeal to `¬Pos`, or to introduce the predicate symbol after the fact.
**What needs resolving**: Either introduce a symbol for "zero tumbler" (e.g., `Zero(t)` with the universal as its defining clause) alongside `Pos(t)` at this site so that TA-PosDom and downstream properties can bind it uniformly, or establish a convention that `¬Pos(t)` is the canonical expression for "zero tumbler" and articulate why the non-triviality of `t ∈ T` (forcing `#t ≥ 1`) makes the equivalence rather than a strictly-weaker negation the intended reading. At present the dual concept has a name but no symbol, and the inline universals that will be needed downstream are not anchored to a definition the consumer can cite.
