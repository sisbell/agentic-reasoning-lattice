### Variable-name slip: T10a's `k' ∈ {0, 1}` conflicts with its own `k' ∈ {1, 2}` restriction
**Foundation**: T10a's own axiom statement — "child-spawning uses exactly one `inc(·, k')` with `k' ∈ {1, 2}`" — binds the variable `k'` specifically to the child-spawning parameter. TA5a's parameter is named `k`, not `k'`. The document distinguishes these consistently elsewhere: T10a.4's standalone proof writes "TA5a establishes that `inc(t, k)` preserves T4 if and only if `k = 1` … or `k = 2`" (using `k` for TA5a) alongside "`k' = 1` imposes no further runtime bound" (using `k'` for T10a).
**ASN**: T10a body, Consequence 4: "TA5a guarantees that `inc(·, 0)` and `inc(·, 1)` unconditionally preserve T4 on any T4-valid input (cases `k = 0` and `k = 1`), and the axiom restricts child-spawning at `k' = 2` to inputs satisfying TA5a's `zeros(t) ≤ 2` bound; the inductive hypothesis supplies T4-validity of every `t ∈ dom` of the parent allocator, closing the premise of the `k' ∈ {0, 1}` branches…"
**Issue**: The phrase "the `k' ∈ {0, 1}` branches" conflates two variables: T10a's `k'` (restricted to {1, 2} and used only for child-spawning) and TA5a's `k` (ranging over {0, 1, 2, …}). Under T10a's own binding, "`k' = 0`" is outside the axiom's permitted range, and "`k' ∈ {0, 1}`" would admit a child-spawning event at `k' = 0` — directly contradicting the axiom's clause that restricts child-spawning to `k' ∈ {1, 2}`. The intended reading is "TA5a's `k ∈ {0, 1}` branches" (sibling via `k = 0`, child-spawning via `k' = 1` which instantiates TA5a at `k = 1`), but the written form breaks the variable binding that T10a's own axiom and Consequence 4's preceding sentence just established.
**What needs resolving**: Consequence 4 must disambiguate the variable: either rewrite as "closing the premise of TA5a's `k ∈ {0, 1}` branches" (making explicit that the range refers to TA5a's parameter, not T10a's `k'`), or split the sentence to reference the sibling chain's `k = 0` case of TA5a and T10a's `k' = 1` spawning case separately. Without resolution, the proof text asserts the discipline permits a `k' = 0` spawning branch that its own axiom forbids.

### Notational mismatch between "proper prefix" text and `≼` symbol at ancestor–descendant nesting step
**Foundation**: Prefix (definition) — `p ≼ q` is the non-strict prefix relation; `p ≺ q` is proper prefix, requiring `p ≠ q` (equivalently `#p < #q`). The document states these as distinct symbols and uses `≺` for proper prefix in the Prefix block itself.
**ASN**: T10a body, Consequence 5 introductory paragraph (also restated in the separate T10a.5 theorem's intro and Formal Contract): "…TA5(b) gives agreement on all positions `1..#tᵢ` and TA5(d) gives `#inc(tᵢ, k') > #tᵢ`, so `tᵢ ≼ inc(tᵢ, k')` — the spawning output is a proper prefix of the child's base address…"
**Issue**: The prose says "proper prefix" but the symbol used is `≼`, which the Prefix definition reserves for non-strict prefix. The proof has established both conjuncts of *proper* prefix (agreement via TA5(b), strict length inequality via TA5(d)+NAT-addcompat), so the conclusion should be `tᵢ ≺ inc(tᵢ, k')` under the document's own notation. Downstream reasoning that quotes this step ("the spawning output is a proper prefix of the child's base") relies on proper-ness (equality would be incompatible with the ancestor-descendant distinctness the subsequent sentence claims from T10a.3), but what is written is the weaker `≼` relation.
**What needs resolving**: Either change `≼` to `≺` at every site that claims "proper prefix" (Consequence 5 intro, T10a.5 theorem intro and Formal Contract's "Ancestor-descendant pairs inherently nest" sentence), or weaken the prose from "proper prefix" to "prefix" where the weaker relation actually suffices. If the strict version is in fact what downstream consumers (e.g., T10a.6 Case 1) rely on, the symbol must be upgraded consistently.
