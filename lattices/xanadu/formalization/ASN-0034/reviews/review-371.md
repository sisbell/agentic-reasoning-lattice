### T4's Axiom slot contains derived canonical forms, not axioms
**Foundation**: structural — the Axiom slot should state what the claim posits; derived consequences belong in Postconditions or are established by proof
**ASN**: T4 *Formal Contract — Axiom*: "Valid address tumblers satisfy: `zeros(t) ≤ 3`; `(A i : 1 ≤ i < #t : ¬(tᵢ = 0 ∧ tᵢ₊₁ = 0))`; `t₁ ≠ 0`; `t_{#t} ≠ 0`. T4 stipulates that a position `i` of `t` is a *field separator* iff `tᵢ = 0`. **The canonical written form of a T4-valid address tumbler is keyed to `zeros(t) ∈ {0, 1, 2, 3}`, with one form per zero count: `zeros(t) = 0` gives `t = N₁. ... .Nₐ`; `zeros(t) = 1` gives `t = N₁. ... .Nₐ . 0 . U₁. ... .Uᵦ`; ...**"
**Issue**: The final sentence of T4's Axiom is not axiomatic. It states canonical forms keyed to `zeros(t) ∈ {0, 1, 2, 3}` — but "`zeros(t) ∈ {0, 1, 2, 3}`" is precisely the exhaustion that T4c's Exhaustion paragraph derives from the bound `zeros(t) ≤ 3` by iterated trichotomy and NAT-discrete. Stating the set-membership `∈ {0, 1, 2, 3}` in T4's Axiom therefore either silently includes T4c's exhaustion result inside T4 (making T4c's Exhaustion circular or redundant) or is shorthand for "for each value in {0, 1, 2, 3} at which zeros(t) happens to sit, the form is …" — but the Axiom slot gives a precise reader no way to tell which. Either reading puts derivational content in a slot reserved for what the claim posits.
**What needs resolving**: Either move the canonical-forms paragraph to a Postconditions slot (where it is understood to follow from T4's Axiom plus dependencies), demote the set-keying to a schema ("for each `k ∈ ℕ` with `0 ≤ k ≤ 3` at which `zeros(t) = k`, the form is …") that does not pre-commit to exhaustion, or fold the canonical forms into T4c's section where exhaustion is actually established.

### T4b's Definition slot contains proof-shaped justifications rather than the definition alone
**Foundation**: structural — the Definition slot should characterize the claim's new construct; justification belongs to the proof body or the Postconditions
**ASN**: T4b *Formal Contract — Definition*: "The four partial functions `N, U, D, E : T ⇀ T` are characterised as follows. `dom(N)` is the T4-valid subset of `T`; … for `k = 3`, `N(t) = (t₁, ..., t_{s₁ - 1})`, `U(t) = (t_{s₁ + 1}, ..., t_{s₂ - 1})`, `D(t) = …`, `E(t) = …`. **Each projection's image lies in the subset of `T` whose every component is in `ℕ⁺`; this subset is cut out by T0's carrier `ℕ` together with NAT-zero and NAT-discrete. Each index `s_i + 1` lies in ℕ by NAT-closure's successor closure, and each `s_i − 1` lies in ℕ by NAT-sub's conditional-closure clause applied at `s_i ≥ 1` (T0's index domain) and `1 ∈ ℕ` (NAT-closure). The segment-length inequalities `s₁ ≥ 2`, `s_k ≤ #t - 1` (when `k ≥ 1`), and `s_{j+1} ≥ s_j + 2` — obtained above by unpacking T4a's conclusion that every field segment is non-empty, with numeral `2 := 1 + 1 ∈ ℕ` by NAT-closure and `#t − 1 ∈ ℕ` by NAT-sub under the locally unpacked `#t ≥ 1` (from T0's nonempty qualification of `t ∈ T`) — keep every index `s_i ± 1` appearing above within `{1, …, #t}`.** Outside the stated domains no projection is assigned a value."
**Issue**: Beyond the domain-and-value specification, the Definition slot contains: (a) a claim about each projection's image with NAT-zero/NAT-discrete citation, (b) a well-formedness argument for `s_i + 1`, `s_i − 1` ∈ ℕ with NAT-closure/NAT-sub citations, (c) a re-derivation of the segment-length inequalities from T4a's Postcondition with numeral-grounding citations. These are all justifications of the construction, and nearly identical content also appears in the proof body above and in the Postconditions below. A Definition that carries its own proof triples the document's load and makes it ambiguous which statement is the definitional commitment and which is a discharged obligation.
**What needs resolving**: Restrict T4b's Definition slot to stating `dom(N), dom(U), dom(D), dom(E)` and the values of each projection case-by-case. Move the image-containment, index-in-ℕ, and inequality-re-expression justifications into the proof body (or cite them once as Postconditions), so the Definition is a characterization rather than a characterization-plus-proof.

### T4c's "no constructor producing tumblers" framing names no specific axiom gap
**Foundation**: T0 (CarrierSetDefinition) — "T is the set of all nonempty finite sequences over ℕ"
**ASN**: T4c main body: "Realisation — existence of a T4-valid tumbler at each zero count — is not claimed: NAT-closure's `1 ∈ ℕ` with NAT-addcompat's `0 < 1` (the strict successor inequality instantiated at `n = 0`) already places `1` in `ℕ⁺`, so strictly positive components are available, **but the foundations in scope supply no constructor producing tumblers of the lengths these four cases require (from `1` up to `7`)**, so T4c stands as a pure definition on whatever T4-valid tumblers exist." T4's prose on T4c echoes: "the foundations in scope supply no constructor producing tumblers of the required lengths."
**Issue**: T0's Axiom declares T to be "the set of **all** nonempty finite sequences over ℕ". Under that axiom, any nonempty finite tuple of ℕ-elements — e.g. `1.0.1.0.1.0.1` once `0, 1 ∈ ℕ` — is in T by membership, with no "constructor" required. The caveat's phrase "supply no constructor" does not match any stated axiom gap: it is not "T0 fails to assert membership" (T0 asserts set-containment of all such sequences), and it is not "no positive numeral exists" (NAT-closure + NAT-addcompat address that, as the same sentence notes). What T4c actually needs for realisation — an existence inference from "there is an 8-tuple with the right pattern of zeros and ones" to "a specific such element is in T" — is a meta-logical point about tuple-formation, not a missing "constructor axiom". A precise reader trying to audit which foundation is missing cannot pin the caveat to any clause in T0 or the NAT- claims.
**What needs resolving**: Either pinpoint the actual gap (e.g., "T0 asserts T's content set-theoretically but this ASN lacks an inference rule for exhibiting specific sequences as T-members in a proof") and phrase the caveat in those terms, drop the caveat and reconcile T0's "set of all" wording with whatever formal framework downstream ASNs use, or add a foundation claim licensing the explicit exhibition of finite tuples as T-members.

### T4b's Preconditions abbreviate "T4 constraints" while T4c's spells them out
**Foundation**: structural — Preconditions slots across sibling claims should state their preconditions with comparable precision
**ASN**: T4b *Preconditions*: "`t` satisfies the T4 constraints." T4c *Preconditions*: "`t` satisfies the T4 constraints (`zeros(t) ≤ 3`, no two zeros adjacent, `t₁ ≠ 0`, `t_{#t} ≠ 0`)." T4 *Axiom* is the source of the four constraints.
**Issue**: T4b and T4c have the same precondition — T4-validity — but express it with different granularity. T4c spells out the four conditions inline; T4b refers to them by label only. A downstream consumer citing T4b must cross-reference T4's Axiom to unpack what "T4 constraints" expands to, while a consumer of T4c does not. The difference is not motivated by content (T4b in fact uses every one of the four conditions internally), so the abbreviation degrades auditability without saving anything a reader could not read off T4c.
**What needs resolving**: Either spell out the four T4 constraints in T4b's Preconditions as T4c does, or abbreviate T4c's Preconditions to "`t` satisfies the T4 constraints" as T4b does — and use the same convention for both.

### Pre-T4 prose declares the four level labels before T4c formally defines them
**Foundation**: structural — a term should have a single definitional site, cited by other uses, rather than being posited informally and re-posited formally
**ASN**: Hierarchical-structure prose (before T4's Formal Contract): "A tumbler with zero zeros is a *node address*. One zero: a *user address*. Two zeros: a *document address*. Three zeros: an *element address*. The count of zero-valued components determines the specificity level, **and these are the level labels T4c fixes formally.**" T4c Postconditions: "`(A t ∈ T : t is T4-valid :: (zeros(t) = 0 ↔ t is a node address) ∧ (zeros(t) = 1 ↔ t is a user address) ∧ (zeros(t) = 2 ↔ t is a document address) ∧ (zeros(t) = 3 ↔ t is an element address))`".
**Issue**: The pre-T4 prose introduces the four labels with definitional force ("is a *node address*"), italicizing them like first-use terms, and then admits the formal fixation lives in T4c. A precise reader cannot tell whether the pre-T4 paragraph *is* the definition (in which case T4c merely exports it, and "fixes formally" is misleading), or whether it is informal foreshadowing (in which case the italicized "is a" wording over-commits). T4's own Axiom uses the labels `Node / User / Document / Element` as *field* names, T4c reuses them as *level* labels — both grounded only by the pre-T4 prose and T4c's biconditionals. The combination of the pre-T4 forward reference and T4c's "`t` is a node address iff zeros(t) = 0" produces two candidate definition sites for the same term.
**What needs resolving**: Pick a single definitional site — either let the pre-T4 prose define the labels (and have T4c's Postconditions state a theorem, not a definition, relating them to `zeros(t)`), or let T4c define the labels (and recast the pre-T4 prose as a forward-reference summary rather than a definition with italicized first uses).
