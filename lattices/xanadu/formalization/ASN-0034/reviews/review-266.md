### TS3 quantifier range predicate omits `v ∈ T`
**Foundation**: OrdinalShift precondition `v ∈ T`; T0 (CarrierSetDefinition) — `#·` is defined over T; T3 (CanonicalRepresentation) — equality of tumblers presupposes both operands in T.
**ASN**: TS3 formal statement: "`(A v, n₁, n₂ : n₁ ∈ ℕ ∧ n₂ ∈ ℕ ∧ n₁ ≥ 1 ∧ n₂ ≥ 1 ∧ #v = m : shift(shift(v, n₁), n₂) = shift(v, n₁ + n₂))`". Preconditions list: "v ∈ T, n₁ ∈ ℕ, n₂ ∈ ℕ, n₁ ≥ 1, n₂ ≥ 1, #v = m".
**Issue**: The universally-quantified range predicate binds `v` but never constrains `v ∈ T`. It also uses `m` as a free variable linked only by `#v = m`. The Preconditions list does specify `v ∈ T`, but the two representations of TS3's hypothesis list disagree. Without `v ∈ T` in the range predicate, the statement quantifies over an unbounded `v`, and `#v` (used both in the range and the proof body) has no domain guarantee. Under the per-step discipline the ASN enforces (OrdinalDisplacement distinguishes order preconditions from carrier-typing preconditions), the formal statement and the Preconditions list must agree on the hypothesis set.
**What needs resolving**: TS3's range predicate and Preconditions list must express the same hypotheses. Either add `v ∈ T` (and make the treatment of `m` explicit — a local abbreviation for `#v`, or an existential witness) to the quantifier's range predicate, or reconcile the Preconditions list with a range predicate that intentionally omits it.

### TS3 does not discharge `u ∈ T` before the second OrdinalShift invocation
**Foundation**: OrdinalShift preconditions `v ∈ T, n ∈ ℕ, n ≥ 1`; TA0 (WellDefinedAddition) exports `a ⊕ w ∈ T`.
**ASN**: TS3 left-side, second shift: "Now we apply the second shift. Before invoking OrdinalShift at `(u, n₂)` we discharge its `n ∈ ℕ` precondition from TS3's own precondition `n₂ ∈ ℕ` under the substitution `n = n₂`, again routing the carrier-typing through the separately named `n₂ ∈ ℕ` precondition rather than through the order precondition `n₂ ≥ 1`."
**Issue**: OrdinalShift's precondition list is `v ∈ T, n ∈ ℕ, n ≥ 1`. At the invocation `shift(u, n₂)`, the first-argument slot (OrdinalShift's `v`) is filled with `u = v ⊕ δ(n₁, m)`, so the check requires `u ∈ T`. TS3 explicitly discusses only the `n ∈ ℕ` precondition at this invocation; `u ∈ T` is never stated. The fact is available from TA0's exported postcondition `a ⊕ w ∈ T` applied at `a = v, w = δ(n₁, m)`, but TS3's prose skips that step. Under the same per-step citation discipline TS3 enforces for `n₁ + n₂ ∈ ℕ` (routed through NAT-closure) and for the three `n ∈ ℕ` discharges at each shift site, the `u ∈ T` discharge must also be named. The `#u = m` bound TS3 uses a sentence later (for TA0's action-point precondition `k = m ≤ m = #u`) itself presupposes `u ∈ T` for `#u` to type, so the gap is not merely cosmetic.
**What needs resolving**: TS3 must explicitly source `u ∈ T` (naturally via TA0's first postcondition on `v ⊕ δ(n₁, m)`) before invoking OrdinalShift at `(u, n₂)`, in parallel with the existing `n ∈ ℕ` discharge, so every OrdinalShift precondition at that invocation has a named source.
