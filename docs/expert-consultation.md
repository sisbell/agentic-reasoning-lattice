# Expert Consultation — Nelson and Gregory Channels

The pipeline uses two independent expert channels to gather evidence for specification. Nelson provides design intent from Ted Nelson's writings. Gregory provides implementation evidence from Roger Gregory's udanax-green codebase. The channels are structurally separated — each has its own vocabulary, source material, and agent architecture.

## Why Two Channels

The specification must capture what the system *should* guarantee (design intent) and what it *actually does* (implementation evidence). If both come from the same source, errors in understanding propagate to both and become invisible.

This is the same independence that model checking requires (Baier & Katoen): the property specification and the system model must come from separate sources, or verification is circular.

In practice: early versions of the pipeline mixed Nelson and Gregory evidence in a single consultation. The resulting ASN properties coupled to udanax-green's data structures and didn't generalize. Separating the channels and enforcing a vocabulary firewall produced better properties — ones that capture design guarantees independent of implementation.

## The Vocabulary Firewall

Each channel is restricted to its own vocabulary at the prompt level:

### Nelson vocabulary (design terms)
Content, identity, permanence, links, documents, sharing, versions, transcopy, inclusion lists, virtual documents, addresses, spans, ownership.

### Gregory vocabulary (implementation terms)
I-addresses, V-space, V-positions, spanfilade, crum, wid, subspaces, enfilades, tumblers, loafs, poomfilade, sensfilade, orgl, itype.

**The firewall is enforced in the decomposition prompts:**
- `scripts/prompts/discovery/nelson-questions.md` bans implementation terms
- `scripts/prompts/discovery/gregory-questions.md` uses KB technical vocabulary

If Nelson can say "spanfilade," the answers couple to udanax-green's data structures and the properties won't hold for any other conforming implementation. The firewall is structural, not stylistic.

## Nelson Channel

Nelson answers questions about Xanadu's design intent using Ted Nelson's writings as source material.

### Source material
- `nelson/literary-machines/` — Literary Machines (PDF pages)
- `nelson/xanadu-concepts/` — Core concept files
- `nelson/nelson-intent/` — Design reasoning documents

### Architecture
- **5 parallel threads** per consultation round
- Each thread answers one Nelson sub-question
- Uses opus for reasoning depth
- Prompt: `scripts/prompts/discovery/nelson-agent.md`

### What Nelson provides
- Design guarantees ("content must be permanent")
- Conceptual definitions ("what is a transclusion?")
- Intent clarification ("should links survive deletion?")
- Vocabulary grounding ("what does Nelson mean by 'virtual document'?")

### Why Nelson first
Nelson shapes definitions. If Gregory defines terms first, the definitions anchor to implementation details. Nelson establishes what the concepts *mean* in the design, then Gregory provides evidence about how they're *realized*.

## Gregory Channel

Gregory answers questions about udanax-green's implementation using two parallel agents per question.

### Source material
- `udanax-test-harness/knowledge-base/` — KB synthesis (pre-analyzed findings)
- `udanax-test-harness/` — udanax-green C source code
- `udanax-test-harness/findings/` — Analysis findings
- `udanax-test-harness/bugs/` — Known bugs with root causes

### Architecture
Two parallel agents per question:
- **KB synthesis agent** — answers from the knowledge base, pre-analyzed findings, and known bugs. Fast, broad coverage. Prompt: `scripts/prompts/discovery/gregory-synthesis-agent.md`.
- **Code exploration agent** — answers by reading udanax-green C source directly. Tool-enabled (Read, Bash, Grep, Glob). Slower, precise. Prompt: `scripts/prompts/discovery/gregory-code-agent.md`.

Both agents' answers are merged into the consultation output.

### What Gregory provides
- Observable behavior ("INSERT at position p shifts subsequent content")
- Data structure details ("the poomfilade stores position-to-address mappings")
- Edge cases ("DELETE of width 0 is a no-op in the current implementation")
- Bug evidence ("there's a known off-by-one in span boundary handling")
- Code-level confirmation or contradiction of Nelson's design intent

## Pipeline Consultation vs. Ad-hoc Consultation

### Pipeline consultation
Runs as part of discovery or review. Questions are generated by decomposition agents and answers feed into the next pipeline step. Session logs are written to `vault/experts/ASN-NNNN/`.

### Ad-hoc consultation
Direct questions to either channel, outside the pipeline. Useful for exploring a topic before writing an inquiry, or investigating a specific question during manual review.

```bash
python scripts/consult.py nelson "What is Nelson's intent for withdrawal?"
python scripts/consult.py gregory "What does INSERT do at a span boundary?"
```

Ad-hoc consultation results are displayed to the terminal and logged to `vault/experts/` for traceability.

## Session Logging

Every consultation is logged for traceability:

```
vault/experts/ASN-NNNN/
  consultation/           # Initial discovery consultations
    questions.md          # Decomposed sub-questions
    answers.md            # Consolidated answers
  consultation-N/         # Per-review-cycle consultations
  sessions/               # Individual agent transcripts
    nelson-1/
    nelson-2/
    gregory-1/
    gregory-2/
```

Session logs capture the full agent transcript — prompt, tool calls, and response. This allows auditing the evidence chain from inquiry through consultation to ASN property.

## Asking Good Questions

Guidelines for writing consultation questions (applies to both pipeline decomposition and ad-hoc queries):

**One focused question per call.** "What must DELETE guarantee about content permanence?" is better than "Explain the DELETE operation." Focused questions produce atomic guarantees; broad questions produce narratives.

**Ask about guarantees, not mechanisms.** "What must be true after INSERT?" (guarantee) vs. "How does INSERT work?" (mechanism). Guarantees map to formal properties. Mechanisms map to implementation details.

**Nelson: target design intent.** "What should happen when content is deleted from a shared document?" — asks about the design's answer, not the implementation's behavior.

**Gregory: target observable behavior.** "What does the code actually do when content is deleted from a document with active transclusions?" — asks about concrete behavior that confirms or challenges the design intent.

## CLI Reference

```bash
# Nelson consultation
python scripts/consult.py nelson "What is Nelson's intent for withdrawal?"

# Nelson with page image access (enables reading scanned PDF pages)
python scripts/consult.py nelson --with-png "question"

# Gregory consultation (KB synthesis + code exploration)
python scripts/consult.py gregory "What does INSERT do at a span boundary?"

# Gregory KB synthesis only (no code exploration)
python scripts/consult.py gregory --kb-only "question"
```

### Flags

| Flag | Description |
|------|-------------|
| `--with-png` | Enable page image access for Nelson (scanned PDF pages) |
| `--kb-only` | Gregory: KB synthesis only, skip code exploration agent |

## Design Decisions

**Why independent channels rather than one combined expert?** Independence prevents cross-contamination. If a single agent has both Nelson's writings and the C source code, it will naturally mix design intent with implementation detail. The vocabulary firewall is only possible because the channels are separate processes with separate prompts.

**Why 5 parallel Nelson threads?** Nelson questions are independent — each asks about a different aspect of the design, from the same source material. Parallel execution cuts wall-clock time. Gregory is sequential because the code exploration agent may benefit from context established by earlier questions.

**Why two Gregory agents per question?** The KB synthesis agent is fast and covers pre-analyzed material. The code agent is slow but can discover things not in the KB. Running both in parallel provides comprehensive coverage without serial bottleneck.

**Why log every session?** Traceability from ASN property back to evidence source. When a review questions a property, the session logs show exactly what evidence it was derived from. This is the pipeline's equivalent of Abrial's traceability chain.
