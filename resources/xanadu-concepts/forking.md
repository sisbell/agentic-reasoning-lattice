# Forking

Source: Literary Machines, pages 4/18-4/21 (raw 185-188)

## Semantic Intent

### What It Means

Forking is the mechanism by which the docuverse grows without bound while preserving all existing addresses. Any numbered item can "spin off" descendants - servers spawn servers, accounts spawn accounts, documents spawn documents, and so on infinitely. This creates a tree of ownership and delegation that mirrors how creative work actually proliferates: authors derive from authors, ideas branch from ideas.

The insight is that the entire address space is one unified growing tree. Every entity in the docuverse - every server, every account, every document, every version, every character position - exists on branches of this single forking structure. Nelson describes this as "all really one big forking document."

Forking works through ownership: whoever owns a number can create any sub-numbers beneath it. This is called "baptism" - the owner designates new addresses by adding digits to the right. Server 1 can create accounts 1.1, 1.2, 1.3; account 1.2 can create documents 1.2.1, 1.2.2; and so on indefinitely.

### User Guarantee

**You can always create more.** No matter how full the address space seems, you can always fork new addresses under any address you own. Item 2 can become 2.1, 2.2, 2.3... and item 2.4 can become 2.4.1, 2.4.2, 2.4.3... with no limit. Creation never exhausts the space; insertion never invalidates existing references.

### Principle Served

Forking solves the fundamental coordination problem of a decentralized system: how can millions of independent actors create content without a central authority allocating addresses? The answer is delegation through ownership. Each owner controls their own subtree and can subdivide it without asking permission from anyone.

This parallels how intellectual lineage actually works - ideas fork from ideas, responses fork from texts, versions fork from documents. The address structure captures this natural pattern of proliferation.

### How Users Experience It

- Creating a new document doesn't require "finding" an unused address - you simply fork a new one under your account
- Your addresses belong to you; no one else can create content at addresses under your ownership
- You can see provenance in the address: 1.3.27.4 tells you the item descended through server 1, account 3, document 27
- The network can grow without limit - new servers fork from existing servers, new users join under accounts
- Every piece of content, no matter how small, has its place in the universal tree
- No reorganization ever happens - once an address is forked, its relationship to its parent is permanent

### Nelson's Words

> "A server node, or station, has ancestors and may have possible descendant nodes. An account, too, and a document, all have possible descendants." (4/19)

> "The entire tumbler works like that: nodes can spin off nodes; accounts can spin off accounts; nodes can spin off accounts; and so on. Thus all numeration in the docuverse is compressed into a single mechanism." (4/19)

> "So the docuverse is all really one big forking document." (4/18)

> "One digit can become several by a forking or branching process. This consists of creating successive new digits to the right; we call these 'under' the previous digit." (4/20)

> "The owner of a given item controls the allocation of the numbers under it." (4/20)

> "For instance, if there is an item 2, it can branch into several more items, each of which is a daughter item. Thus 2.1, 2.2, 2.3, 2.4... are successive items being placed under 2." (4/20)
