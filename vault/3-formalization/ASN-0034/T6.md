## Decidable containment

The total order T1 determines *sequence* (which address comes first). But the system also needs *containment* — does address `a` belong to account `b`? Is document `d₁` under the same server as document `d₂`? These are not ordering questions; they are prefix questions.

**T6 (DecidableContainment).** For any two tumblers `a, b ∈ T`, the following are decidable from the addresses alone:

  (a) Whether `a` and `b` share the same node field.

  (b) Whether `a` and `b` share the same node and user fields.

  (c) Whether `a` and `b` share the same node, user, and document fields (*exact equality of the full document projection*).

  (d) Whether the document field of `a` is a prefix of the document field of `b` (*structural subordination within a document family* — the prefix relation on the document projection, strictly weaker than the exact equality of (c)).

*Proof.* We show that each of (a)–(d) admits a terminating decision procedure whose only inputs are the tumbler representations of `a` and `b`. The argument rests on three ingredients, which we establish first.

*Ingredient 1 (field extraction).* By T4, valid address tumblers have at most three zero-valued components, no two zeros are adjacent, and the tumbler neither begins nor ends with zero; by T0, every component lies in ℕ, so every non-zero component is strictly positive. The partial function `fields : T ⇀ Seq(ℕ⁺) × Seq(ℕ⁺) × Seq(ℕ⁺) × Seq(ℕ⁺)` introduced in T4b decomposes a tumbler into its node, user, document, and element fields by locating all zero-valued components — which, by T4's definitional role-assignment, are exactly the field separators, leaving the strictly positive components (via T0) as field contents. T4b fixes the domain of `fields` as exactly the T4-valid subset of `T`; outside that subdomain — tumblers with `zeros(t) ≥ 4` or violating the field-segment constraint — `fields(t)` is not assigned a value. The T6 precondition requires `a, b` to satisfy T4, so both `a` and `b` lie in the domain of `fields` and each invocation below is well-defined; every consumer of a projection in cases (a)–(d) inherits this discharge. By T4(b), `fields(t) = (N(t), U(t), D(t), E(t))` is well-defined and uniquely determined by `t` alone on this subdomain, with absent fields represented by the empty sequence `ε` (unambiguous because, by T4a, every *present* field segment is non-empty). Since `t` is a finite sequence, the extraction terminates: scan `t` once, record the positions of zero-valued components, and partition the remaining components into the corresponding field sub-sequences. Write `N(t)`, `U(t)`, `D(t)`, `E(t)` for the node, user, document, and element projections of `fields` respectively — each a partial map `T ⇀ Seq(ℕ⁺)` defined on the same T4-valid subdomain. Each projection returns a finite sequence of strictly positive natural numbers, equal to `ε` iff the corresponding field is absent from `t`.

*Ingredient 2 (field presence).* Define `zeros(t) = #{i : 1 ≤ i ≤ #t ∧ tᵢ = 0}`, the count of zero-valued components in a tumbler `t`; the count is computable from `t` alone by a single finite scan. The biconditionals that tie this count to *field presence* — whether a projection of `fields(t)` equals `ε` — live in T4b's Postconditions, not in T4c: T4c defines the four hierarchical *level labels* (*node, user, document, element address*) by zero count on the T4-valid subdomain, and its Postcondition explicitly disclaims the field-presence reading ("any independent characterisation of the levels — for example, *user address* as 'the tumbler has exactly `N(t)` and `U(t)` non-empty in T4b's `fields(t)` decomposition' — is not the content of T4c"). The field-presence biconditionals we need for cases (b), (c), (d) are therefore sourced from T4b's absence-pattern postcondition, which fixes, for every T4-valid tumbler: `N(t) ≠ ε` unconditionally (the node field is always present — `zeros(t) = 0` still has `N(t) = (t₁, ..., t_{#t})` non-empty); `U(t) ≠ ε` iff `zeros(t) ≥ 1`; `D(t) ≠ ε` iff `zeros(t) ≥ 2`; `E(t) ≠ ε` iff `zeros(t) = 3`. Each presence check thus reduces to a comparison of `zeros(t)` against a fixed natural-number threshold, decidable in finitely many steps.

*Ingredient 3 (finite-sequence equality).* Two finite sequences of natural numbers `S = (s₁, ..., sₘ)` and `R = (r₁, ..., rₙ)` are equal iff `m = n` and `(A i : 1 ≤ i ≤ m : sᵢ = rᵢ)`. This check requires one length comparison and at most `m` equality tests on natural numbers, each decidable. The procedure terminates in at most `m + 1` steps. We call this *componentwise comparison* and use it in every case below.

With these three ingredients, each case proceeds by extraction, presence check (where needed), and componentwise comparison.

*(a) Same node field.* Extract `N(a)` and `N(b)` via `fields` (Ingredient 1). Every tumbler has a node field (Ingredient 2), so both `N(a)` and `N(b)` are present. Apply componentwise comparison (Ingredient 3): check `#N(a) = #N(b)` and, if so, `(A i : 1 ≤ i ≤ #N(a) : N(a)ᵢ = N(b)ᵢ)`. This requires at most `#N(a) + 1` comparisons, each decidable. The procedure terminates and returns *yes* iff the node fields are identical.

*(b) Same node and user fields.* Extract `N(a), U(a)` and `N(b), U(b)` via `fields` (Ingredient 1). Both `a` and `b` must possess user fields — that is, `zeros(a) ≥ 1` and `zeros(b) ≥ 1` — which is decidable (Ingredient 2). If either tumbler lacks a user field, the answer is *no*: they cannot share a field that one does not possess. When both are present, apply componentwise comparison (Ingredient 3) to each pair: first `N(a) = N(b)`, then `U(a) = U(b)` — checking `#U(a) = #U(b)` and `(A j : 1 ≤ j ≤ #U(a) : U(a)ⱼ = U(b)ⱼ)`. The procedure terminates and returns *yes* iff both pairs are identical.

*(c) Same node, user, and document fields.* This case asks for *exact equality of the full document projection* — `D(a) = D(b)` componentwise — and is terminologically disjoint from the prefix relation of (d): (c) establishes the symmetric relation `D(a) = D(b)`, whereas (d) establishes the asymmetric relation `D(a) ⪯ D(b)`, and `D(a) = D(b)` implies `D(a) ⪯ D(b)` but not conversely. Extract `N(a), U(a), D(a)` and `N(b), U(b), D(b)` via `fields` (Ingredient 1). Both must possess document fields — `zeros(a) ≥ 2` and `zeros(b) ≥ 2` — decidable by Ingredient 2. If either lacks a document field, the answer is *no*. When both are present, apply componentwise comparison (Ingredient 3) to each of the three pairs: `N(a) = N(b)`, `U(a) = U(b)`, `D(a) = D(b)`. The total number of comparisons is bounded by the sum of the field lengths plus three length checks, all finite. The procedure terminates and returns *yes* iff all three pairs are identical.

*(d) Document-field prefix.* Extract `D(a) = (D₁ᵃ, ..., Dᵧₐᵃ)` and `D(b) = (D₁ᵇ, ..., Dᵧᵦᵇ)` via `fields` (Ingredient 1). Both must possess document fields — decidable as in (c). The sequence `D(a)` is a prefix of `D(b)` iff two conditions hold: `γₐ ≤ γᵦ` and `(A k : 1 ≤ k ≤ γₐ : Dₖᵃ = Dₖᵇ)`. Check the length condition first (one comparison of natural numbers); if it fails, return *no*. Otherwise verify componentwise agreement up to position `γₐ` (at most `γₐ` comparisons, each decidable). The procedure terminates in at most `γₐ + 1` steps and returns a boolean.

In every case the procedure examines only the finite sequence of components in `a` and `b`, performs finitely many equality or ordering tests on natural numbers, and terminates. No mapping tables, version graphs, or system state are consulted — the tumbler representation alone suffices. ∎

*Formal Contract:*
- *Preconditions:* `a, b ∈ T` are valid tumblers satisfying T4 (at most three zeros, no adjacent zeros, no leading or trailing zero). Positivity of non-zero components is supplied by T0's carrier ℕ.
- *Depends:* T0 (CarrierSetDefinition) — invoked at Ingredient 1 ("by T0, every component lies in ℕ, so every non-zero component is strictly positive") and again in the precondition note that T0's carrier ℕ supplies positivity rather than a separate T4 clause; T0 licenses the strictly-positive-non-zero claim used to distinguish field components from separator components throughout. T4 (HierarchicalParsing) — invoked at Ingredient 1 to fix the zero-count bound, the no-adjacent-zeros clause, and the boundary clauses ("By T4, valid address tumblers have at most three zero-valued components, no two zeros are adjacent, and the tumbler neither begins nor ends with zero"), and again at the role-assignment claim ("which, by T4's definitional role-assignment, are exactly the field separators"); T4 supplies the structural conditions on which all four decision procedures rely. T4a (SyntacticEquivalence) — invoked at Ingredient 1 to license the unambiguity of the ε-marker for absent fields ("absent fields represented by the empty sequence `ε` (unambiguous because, by T4a, every *present* field segment is non-empty)"); T4a converts T4's positional clauses into the segment-non-emptiness statement that justifies treating ε as a definite absence indicator. T4b (UniqueParse) — invoked at Ingredient 1 to fix `fields : T ⇀ Seq(ℕ⁺) × Seq(ℕ⁺) × Seq(ℕ⁺) × Seq(ℕ⁺)` as the partial function whose domain is exactly the T4-valid subset of `T` ("The partial function `fields : T ⇀ Seq(ℕ⁺) × Seq(ℕ⁺) × Seq(ℕ⁺) × Seq(ℕ⁺)` introduced in T4b … T4b fixes the domain of `fields` as exactly the T4-valid subset of `T`") and to fix `fields(t) = (N(t), U(t), D(t), E(t))` as well-defined and uniquely determined on that subdomain ("By T4(b), `fields(t) = (N(t), U(t), D(t), E(t))` is well-defined and uniquely determined by `t` alone on this subdomain"); invoked again at Ingredient 2 for the *absence-pattern postcondition* that ties field presence to zero count — `N(t) ≠ ε` unconditionally, `U(t) ≠ ε` iff `zeros(t) ≥ 1`, `D(t) ≠ ε` iff `zeros(t) ≥ 2`, `E(t) ≠ ε` iff `zeros(t) = 3` (stated as the case analysis "`zeros(t) = 0` → only `N(t)` non-empty; `zeros(t) = 1` → `N(t), U(t)` non-empty, `D(t) = E(t) = ε`; `zeros(t) = 2` → `N(t), U(t), D(t)` non-empty, `E(t) = ε`; `zeros(t) = 3` → all four non-empty" in T4b's Postconditions) — which supplies the presence-check thresholds (`zeros(a) ≥ 1`, `zeros(a) ≥ 2`) consumed by cases (b), (c), (d); and once more in the closing post-condition referencing `fields(t)` (T4(b)); T4b therefore supplies both the partial-function signature / field-extraction action of Ingredient 1 and the field-presence biconditionals of Ingredient 2, with T6's precondition that `a, b` satisfy T4 discharging the domain restriction so that `fields(a)`, `fields(b)`, and each presence-threshold check are well-defined throughout cases (a)–(d). T4c (LevelDetermination) — T4c defines the four hierarchical level labels (*node address*, *user address*, *document address*, *element address*) by zero count on the T4-valid subdomain, establishing the enumeration `zeros(t) ∈ {0, 1, 2, 3}` as a bijection onto the four address levels. T6's field-presence thresholds (`U(t) ≠ ε` iff `zeros(t) ≥ 1`, `D(t) ≠ ε` iff `zeros(t) ≥ 2`, `E(t) ≠ ε` iff `zeros(t) = 3`) are distinct from T4c's level-label biconditionals — T4c's Postcondition explicitly disclaims the field-presence reading, noting that "any independent characterisation of the levels … is not the content of T4c" — so the inline citation at Ingredient 2 routes through T4b's absence-pattern postcondition rather than through T4c. T4c is retained in this depends list as the corollary that pairs the zero-count structure on which Ingredient 2's thresholds rely with the hierarchical-level vocabulary ("node", "user", "document", "element") under which cases (a)–(d) of T6 are stated; the proof body makes no direct citation of T4c.
- *Postconditions:* (a) The procedure terminates and returns YES iff `N(a) = N(b)` (componentwise). (b) The procedure terminates and returns YES iff `zeros(a) ≥ 1 ∧ zeros(b) ≥ 1 ∧ N(a) = N(b) ∧ U(a) = U(b)`; returns NO if either tumbler lacks a user field. (c) The procedure terminates and returns YES iff `zeros(a) ≥ 2 ∧ zeros(b) ≥ 2 ∧ N(a) = N(b) ∧ U(a) = U(b) ∧ D(a) = D(b)`; returns NO if either tumbler lacks a document field. (d) The procedure terminates and returns YES iff `zeros(a) ≥ 2 ∧ zeros(b) ≥ 2 ∧ #D(a) ≤ #D(b) ∧ (A k : 1 ≤ k ≤ #D(a) : D(a)ₖ = D(b)ₖ)`; returns NO if either tumbler lacks a document field. All decisions use only the tumbler representations of `a` and `b`, via `fields(t)` (T4(b)) and componentwise comparison on finite sequences of natural numbers.

T6 is a corollary of the T4-family (T4, T4a, T4b, T4c) rather than of T4 alone: the proof consumes T4's zero-count and adjacency constraints at Ingredient 1, T4a's unambiguity of the ε-marker for absent fields at Ingredient 1, T4b's partial function `fields` together with its absence-pattern postcondition at Ingredients 1 and 2, and — implicitly, via the level-label vocabulary (*node*, *user*, *document*, *element address*) under which cases (a)–(d) are stated — T4c; T0 additionally supplies the carrier ℕ from which the strictly-positive subcarrier ℕ⁺ used by the projections is carved. Beyond these dependencies the proof adds only field extraction, presence checks, and componentwise comparison on finite sequences, each decidable in finitely many steps. We state T6 separately because the decidability claim is load-bearing for decentralized operation, but it introduces no independent content beyond what T0, T4, T4a, T4b, and T4c jointly supply.

We must note what T6(d) does NOT capture. The document field records the *allocation hierarchy* — who baptised which sub-number — not the *derivation history*. Version `5.3` was allocated under document `5`, but this tells us nothing about which version's content was copied to create `5.3`. Nelson: "the version, or subdocument number is only an accidental extension of the document number, and strictly implies no specific relationship of derivation." Formal version-derivation requires the version graph, not just the address.

Nelson confirms that shared prefix means shared containing scope: "The owner of a given item controls the allocation of the numbers under it." The prefix IS the path from root to common ancestor. But he cautions: "Tumblers do not affect the user-level structure of the documents; they only provide a mapping mechanism, and impose no categorization and no structure on the contents of a document." Shared prefix guarantees containment and ownership, never semantic categorization.

Gregory's implementation confirms the distinction between ordering and containment. The codebase provides both `tumblercmp` (total order comparison) and `tumbleraccounteq` (prefix-matching predicate with zero-as-wildcard semantics). The latter truncates the candidate to the length of the parent and checks for exact match — this is the operational realization of T6, and it is a genuinely different algorithm from the ordering comparison.
