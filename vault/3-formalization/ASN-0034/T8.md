## Allocation permanence

The most consequential property of the address system is that once an address is allocated, it persists forever:

**T8 (AllocationPermanence).** If tumbler `a ∈ T` has been allocated at any point in the system's history, then for all subsequent states, `a` remains in the set of allocated addresses. No operation removes an allocated address from the address space. The set of allocated addresses is monotonically non-decreasing.

*Proof.* The argument depends on the characterizations of every operation the system defines — T1, T2, T4 establish that comparison and parsing are read-only; TumblerAdd, TumblerSub, and TA5 define ⊕, ⊖, and inc as pure functions on T with no allocation side effects; and T10a constrains allocation to an insert-only mechanism via frontier advance — together with the NoDeallocation axiom, which stipulates that no removal operation exists in the system's operation vocabulary. We must show that the set of allocated addresses grows monotonically: for every state transition s → s', `allocated(s) ⊆ allocated(s')`.

Every operation falls into exactly one of three classes; we treat each in turn.

*Case 1: Read-only operations.* Comparison (T1, T2) and parsing (T4) inspect tumbler values without modifying any state. Since these operations only examine components and return boolean or structural results, `allocated(s') = allocated(s)`, and the inclusion holds trivially.

*Case 2: Pure arithmetic.* The operations ⊕ (TumblerAdd), ⊖ (TumblerSub), and inc (TA5) are pure functions on T — each takes tumbler values as input and returns a tumbler value as output, without mutating the allocated set. These transitions satisfy `allocated(s') = allocated(s)`.

*Case 3: Allocation.* T10a constrains allocation to a single mechanism: each allocator advances its frontier by repeated application of `inc(·, 0)` (TA5), producing an address strictly greater than the previous, and inserts it into the allocated set. The transition satisfies `allocated(s') = allocated(s) ∪ {a_new}` for some fresh address `a_new`. Since `allocated(s) ⊆ allocated(s) ∪ {a_new} = allocated(s')`, the inclusion holds.

These three cases are exhaustive: Cases 1 and 2 cover every operation that does not modify the allocated set (read-only by T1, T2, T4; pure arithmetic by TumblerAdd, TumblerSub, TA5), and Case 3 covers the sole operation that does (allocation by T10a). The exhaustiveness of this case analysis rests on the NoDeallocation axiom: the system's operation vocabulary admits no removal operation, so every state transition falls into one of the three classes above.

Since every individual transition preserves the inclusion, an immediate induction on the length of any transition sequence s₀ → s₁ → ··· → sₙ yields `allocated(s₀) ⊆ allocated(sₙ)` for all reachable states. ∎

*Formal Contract:*
- *Invariant:* For every state transition s → s', `allocated(s) ⊆ allocated(s')`.
- *Depends:* NoDeallocation (the system defines no removal operation — the closed-world constraint that makes the case analysis exhaustive); T10a (allocation is an insert-only frontier advance via repeated `inc(·, 0)`, characterizing the sole operation that modifies the allocated set).
- *Frame:* Read-only operations (T1, T2, T4) and pure arithmetic (⊕, ⊖, inc) preserve the allocated set exactly: `allocated(s') = allocated(s)`. Allocation transitions (T10a) extend the set: `allocated(s') = allocated(s) ∪ {a_new}` for some fresh address `a_new`.
