## Hierarchical structure

Tumblers encode a containment hierarchy. Nelson uses zero-valued components as structural delimiters:

> "There are four major fields, each expandable indefinitely, with three major dividers between them. These fields are Server, User, Document and Contents."

> "A tumbler address may have at most three zero digits... These are interpreted as the major dividers, and have lexical significance as punctuation."

We formalize this. Define a *field separator* as a component with value zero. An address tumbler has the form:

`t = N₁. ... .Nₐ . 0 . U₁. ... .Uᵦ . 0 . D₁. ... .Dᵧ . 0 . E₁. ... .Eδ`

where each `Nᵢ, Uⱼ, Dₖ, Eₗ > 0`. The four fields are:

- **Node field** `N₁. ... .Nₐ`: identifies the server. "The server address always begins with the digit 1, since all other servers are descended from it."
- **User field** `U₁. ... .Uᵦ`: identifies the account. "Typically, the user will have no control over the node address he, she or it is assigned; but once assigned a User account, the user will have full control over its subdivision forevermore."
- **Document field** `D₁. ... .Dᵧ`: identifies the document and version. Nelson notes the boundary between base document and version is not syntactically marked — "the version, or subdocument number is only an accidental extension of the document number, and strictly implies no specific relationship of derivation."
- **Element field** `E₁. ... .Eδ`: identifies the content element. T4 constrains only `Eₗ > 0` at every element-field position (the canonical-form clause admits every strictly positive integer there); any *typed* interpretation of the first component — for instance, a convention fixing which values of `E₁` identify which subspace — is the concern of a downstream ASN that formalises subspace identification, not of T4's axiom. T7 below carries this scope split explicitly into its own preamble, which records that "the stronger claim is relocated to a downstream property in a later ASN, where the subspace identifier receives its typed treatment."

Not every tumbler need have all four fields. A tumbler with zero zeros addresses a node. One zero: a user account. Two zeros: a document. Three zeros: an element. The count of zero-valued components determines the specificity level.

**T4 (HierarchicalParsing).** Define `zeros(t) = #{i : 1 ≤ i ≤ #t ∧ tᵢ = 0}`, the count of zero-valued components in `t`. Every tumbler `t ∈ T` used as an address satisfies four positional conditions, stated entirely in terms of the indices at which zero occurs: at most three of its components are zero, no two zeros are adjacent, `t₁` is not zero, and `t_{#t}` is not zero. Formally: `zeros(t) ≤ 3`; `(A i : 1 ≤ i < #t : ¬(tᵢ = 0 ∧ tᵢ₊₁ = 0))`; `t₁ ≠ 0`; `t_{#t} ≠ 0`. We call the last three conditions (no adjacent zeros, `t₁ ≠ 0`, `t_{#t} ≠ 0`) collectively the *field-segment constraint*. T0 already fixes the carrier as ℕ, so a component is either zero or strictly positive — no further clause is needed to force non-zero components to be positive, which is why positivity is not listed as a separate axiom here. Each condition refers only to which indices of `t` contain zero; none appeals to any prior notion of which fields are "present" in the address. Checking them requires only a single scan of the raw component sequence. The *field segments* of `t` are the maximal contiguous sub-sequences of non-zero positions — `k+1` of them when `zeros(t) = k`, separated by the zeros — and the field-segment constraint is exactly the requirement that all of them are non-empty. Without the field-segment constraint, a tumbler like `[1, 0, 0, 3]` would have `zeros(t) = 2` yet position `2` would be adjacent to position `3`, leaving an empty interior field segment between them; the no-adjacent-zeros clause rules exactly this out, and the two boundary clauses rule out the analogous degeneracies at the ends. The correspondence between the canonical written form and the positional conditions rests on a definitional role-assignment: T4 stipulates that zero components are field separators, while the non-zero components (strictly positive by T0's carrier ℕ) are field components. The two roles are therefore positionally decidable by inspecting `tᵢ` — zero or non-zero — with no further axiom required, and the number of zero-valued components determines the number of separators, which determines the number of fields.

Three consequences of T4's axiom — each stated and proved as its own first-class numbered theorem in this ASN, with its own statement file (`T4a.md`, `T4b.md`, `T4c.md`), proof object, and formal contract — round out the parsing picture:

  - **T4a (SyntacticEquivalence)** proves that the field-segment constraint (no adjacent zeros, `t₁ ≠ 0`, `t_{#t} ≠ 0`) is logically equivalent to the condition that every field segment of `t` is non-empty. The equivalence is the bridge between T4's positional form and the semantic reading that every present field has at least one component; the proof proceeds by case analysis on `k = zeros(t) ∈ {0, 1, 2, 3}` in both the forward and reverse directions and is discharged in T4a.md.
  - **T4b (UniqueParse)** introduces the partial function `fields(t)` — via a Definition clause in T4b's Formal Contract that fixes its domain (the T4-valid subset of `T`), its codomain (`Seq(ℕ⁺) × Seq(ℕ⁺) × Seq(ℕ⁺) × Seq(ℕ⁺)`), and its action (split the component sequence of `t` at its zero positions, assigning the `zeros(t) + 1` resulting field segments to the first `zeros(t) + 1` tuple components and `ε` to the rest) — and proves the function so defined well-defined and uniquely determined by `t` alone. T3 (CanonicalRepresentation) supplies the rigidity of component values that licenses uniqueness, and T4a supplies the segment-non-emptiness reading that the case analysis consumes; the Definition clause and the uniqueness proof are discharged in T4b.md.
  - **T4c (LevelDetermination)** proves that, on the T4-valid subset of `T` (tumblers satisfying T4's positional conditions), `zeros(t)` bijects onto the four hierarchical levels: `(A t ∈ T : t is T4-valid :: zeros(t) = 0 ↔ node address (node field only) ∧ zeros(t) = 1 ↔ user address (node and user fields) ∧ zeros(t) = 2 ↔ document address (node, user, and document fields) ∧ zeros(t) = 3 ↔ element address (all four fields))`; outside that subdomain — tumblers with `zeros(t) ≥ 4` or with `zeros(t) ≤ 3` but violating the field-segment constraint — T4c assigns no hierarchical level, mirroring T4b's restriction of `fields(t)` to the T4-valid subdomain. T4's own definitional role-assignment biconditional — that a position `i` is a separator iff `tᵢ = 0` — supplies the equality between `zeros(t)` and the separator count on this subdomain, so the number of fields present equals `zeros(t) + 1` whenever T4-validity holds; the proof is discharged in T4c.md, and rests on T4 alone among this ASN's properties (together with T0 for carrier-related side conditions).

Because each of T4a, T4b, T4c is a first-class theorem with its own numbered statement and proof rather than an inline consequence of T4's body prose, downstream citations to these labels — for instance in T4b's Depends list for T4a, in T7's Depends list for T4a and T4b, and in T4a's framing reference to T4c — point at those separate proof objects and terminate an unbroken chain; no citation here relies on a named-but-unstated corollary.

A subtlety deserves emphasis: the hierarchy is *convention layered over flat arithmetic*, not enforcement by the algebra. Gregory's analysis reveals that the comparison, addition, subtraction, and increment operations treat every mantissa slot identically. There is no `isparent`, `isancestor`, or `ischild` primitive in the arithmetic layer. The algebra operates on flat sequences of non-negative integers; the hierarchical interpretation is projected onto those sequences by the allocation machinery and by the field-parsing function that T4b introduces (the partial `fields(t)` whose Definition clause is stated in T4b's Formal Contract). This is a deliberate design choice. The arithmetic layer is kept flat so that comparison and span computation are simple, uniform operations with no parsing of semantic structure.

Hierarchy is constructed by the allocation machinery, not by the algebra. The `.0.` separator is produced when the allocation `depth` parameter equals 2 — creating a child at a *different hierarchical type* than its parent (e.g., an ACCOUNT creating a DOCUMENT). When creating a same-type child (DOCUMENT creating DOCUMENT = versioning), `depth = 1`, and no zero separator is introduced. Gregory confirms: there was even a bug where the first document under an account failed to receive its `.0.` separator — the convention had to be explicitly constructed by the allocator, not enforced by any algebraic invariant.

T4 is an axiom: it constrains which tumblers the system admits as valid addresses. Its three consequences — T4a (SyntacticEquivalence), T4b (UniqueParse), T4c (LevelDetermination) — are each stated and proved as separate first-class numbered theorems in this ASN, with their own statement files (`T4a.md`, `T4b.md`, `T4c.md`) and formal contracts; together they verify the parsing and level-determination properties that follow from the T4 constraints, and two inter-consequence preconditions — recorded in the Preconditions field of the Formal Contract below — complete the DAG among the three postconditions: T3 (CanonicalRepresentation) and T4a (SyntacticEquivalence) are both additional preconditions for T4b's uniqueness argument, while T4c (LevelDetermination) requires no inter-consequence precondition because both the zero↔separator biconditional and the bound `zeros(t) ≤ 3` that drive its bijection are clauses of T4's own axiom. T3 is essential for T4b: canonical representation guarantees that the component sequence of `t` is fixed, so the separator positions computed by scanning for zeros are uniquely determined. Without T3, two representations of the same tumbler might yield different separator sets, and uniqueness of the parse would not follow. T4a is essential for T4b because T4b's case analysis on `zeros(t) ∈ {0, 1, 2, 3}` reads each zero-count case through T4a's segment-non-emptiness lemma, which converts T4's positional field-segment clauses into the non-empty-segment statements that the case analysis consumes when it certifies that each prescribed-non-empty projection has at least one source position. T4c, by contrast, draws its load-bearing biconditional `tᵢ = 0 ⇔ i` is a field separator directly from T4's Definitional role-assignment clause — both directions, `tᵢ = 0 ⇒ i` is a separator and `i` is a separator `⇒ tᵢ = 0`, are licensed by T4's axiom rather than by T4b — and the bound `zeros(t) ≤ 3` is likewise a T4-axiomatic clause; the equality between `zeros(t)` and the separator count that drives T4c's bijection therefore follows from T4 alone, with no precondition on T4b's `fields(t)` decomposition. These two edges (T3 and T4a as preconditions for T4b) are the only inter-consequence preconditions required, and citations to T4a, T4b, T4c elsewhere in the ASN resolve to those separate proof objects rather than to prose summaries here. ∎

*Formal Contract:*
- *Axiom:* Valid address tumblers satisfy the following positional conditions on the component sequence, none of which appeals to any prior notion of field presence: `zeros(t) ≤ 3`; `(A i : 1 ≤ i < #t : ¬(tᵢ = 0 ∧ tᵢ₊₁ = 0))`; `t₁ ≠ 0`; `t_{#t} ≠ 0` (the last three together form the field-segment constraint, guaranteeing that the maximal non-zero contiguous sub-sequences of `t` — the field segments — are all non-empty). Positivity of non-zero components is not a separate axiom clause: T0's carrier ℕ already makes `tᵢ ≠ 0 ⇔ tᵢ > 0`. **Definitional role-assignment:** on top of these positional conditions, T4 further stipulates that a position `i` of `t` is a *field separator* iff `tᵢ = 0`, fixing as part of the axiom the biconditional identification of the set of separator positions with the set of zero-valued positions — the two directions `tᵢ = 0 ⇒ i` is a separator and `i` is a separator `⇒ tᵢ = 0` are both licensed here rather than in body prose alone. Given this role-assignment together with the positional conditions above, a T4-valid address tumbler has the canonical written form `t = N₁. ... .Nₐ . 0 . U₁. ... .Uᵦ . 0 . D₁. ... .Dᵧ . 0 . E₁. ... .Eδ` where each `Nᵢ, Uⱼ, Dₖ, Eₗ > 0`, with only the first `zeros(t) + 1` field segments present in the address; this canonical form is part of the axiom, so downstream properties may cite the address-format skeleton directly rather than through the body prose.
- *Preconditions:* T4b (UniqueParse) requires T3 (CanonicalRepresentation) — canonical representation fixes the component sequence of `t`, ensuring separator positions computed by scanning for zeros are uniquely determined; T4b (UniqueParse) also requires T4a (SyntacticEquivalence) — T4a converts T4's positional field-segment constraint (no two adjacent zeros, `t₁ ≠ 0`, `t_{#t} ≠ 0`) into the equivalent segment-non-emptiness reading that T4b's case analysis on `zeros(t) ∈ {0, 1, 2, 3}` consumes when it certifies each prescribed-non-empty projection as having at least one source position. T4c (LevelDetermination) requires no inter-consequence precondition: T4's axiom directly supplies both the bound `zeros(t) ≤ 3` and the role-assignment biconditional `tᵢ = 0 ⇔ i` is a field separator, so the equality between `zeros(t)` and the separator count that drives T4c's bijection — and hence the bijection between `{0, 1, 2, 3}` and hierarchical level — follows from T4 alone, without depending on T4b's `fields(t)` decomposition.
- *Postconditions:* Each of the three postconditions below is stated and proved as a separate first-class numbered theorem in this ASN — T4a in `T4a.md`, T4b in `T4b.md`, T4c in `T4c.md`, each with its own formal contract — rather than discharged inline in T4's body prose, so downstream citations to these labels resolve to independent proof objects. T4a (SyntacticEquivalence): the field-segment constraint (no two adjacent zeros, `t₁ ≠ 0`, `t_{#t} ≠ 0`) is equivalent to the condition that every field segment of `t` is non-empty — giving the semantic reading (once T4c supplies labels) that every present field has at least one component. T4b (UniqueParse): introduces — via a Definition clause in T4b's own Formal Contract — the partial function `fields : T ⇀ Seq(ℕ⁺) × Seq(ℕ⁺) × Seq(ℕ⁺) × Seq(ℕ⁺)` whose domain is exactly the T4-valid subset of `T` and whose action splits `t`'s component sequence at its zero positions, returning the 4-tuple `(N(t), U(t), D(t), E(t))` with the first `zeros(t) + 1` components assigned to the `zeros(t) + 1` non-empty field segments in order and the remaining components set to the empty sequence `ε`; this corollary proves the function so defined well-defined and uniquely computable from `t` alone, so downstream citations to T4b for `fields(t)` resolve to an object whose domain, codomain, and action are all fixed in T4b's contract rather than inferred from body prose. T4c (LevelDetermination): on the T4-valid subset of `T` — tumblers satisfying T4's positional conditions (`zeros(t) ≤ 3`, no two zeros adjacent, `t₁ ≠ 0`, `t_{#t} ≠ 0`) — `zeros(t)` bijects with hierarchical level: `(A t ∈ T : t is T4-valid :: zeros(t) = 0 ↔ node address ∧ zeros(t) = 1 ↔ user address ∧ zeros(t) = 2 ↔ document address ∧ zeros(t) = 3 ↔ element address)`; outside that subdomain (e.g., `zeros(t) ≥ 4`, or tumblers like `[1, 0, 0, 2]` with adjacent zeros, or `[0, 1, 0, 2]` with a leading zero) T4c assigns no hierarchical level, so downstream citations of a level from `zeros(t)` must carry T4-validity as a precondition — paralleling T4b's restriction of `fields(t)` to the T4-valid subdomain.
