## What tumbler arithmetic is NOT

We must state explicitly what the tumbler algebra does not guarantee. These negative results constrain what an implementer may assume.

**The algebra is not a group.** There is no additive identity — the zero tumbler is a sentinel, not a neutral element for addition. There is no additive inverse for every element — subtraction is only defined when `a ≥ w`. The algebra is not closed under subtraction in general.

**TA-assoc (AdditionAssociative).** Addition is associative where both compositions are defined: `(a ⊕ b) ⊕ c = a ⊕ (b ⊕ c)` whenever both sides are well-defined. Three cases exhaust the possibilities. Let `k_b` and `k_c` be the action points of `b` and `c` respectively. When `k_b < k_c`: both sides produce `aᵢ` for `i < k_b`, `aₖ_b + bₖ_b` at `k_b`, `bᵢ` for `k_b < i < k_c`, `bₖ_c + cₖ_c` at `k_c`, and `cᵢ` beyond — identical. When `k_b = k_c = k`: both sides produce `aₖ + bₖ + cₖ` at `k` (by NAT-addassoc, which states `(m + n) + p = m + (n + p)` on ℕ and so identifies the two parenthesizations `(aₖ + bₖ) + cₖ` and `aₖ + (bₖ + cₖ)`) and `cᵢ` beyond — identical. When `k_b > k_c`: both sides produce `aₖ_c + cₖ_c` at `k_c` and `cᵢ` beyond — identical (the deeper displacement `b` is overwritten by the shallower `c` in both cases). The domain conditions are asymmetric — the left side requires `k_b ≤ #a`, while the right requires only `min(k_b, k_c) ≤ #a` — but on the intersection, the values agree.

The design does not depend on associativity. Shifts are applied as single operations in practice, never composed from multiple smaller shifts. An implementation with finite representations may break associativity through overflow at the action-point component, but the abstract algebra carries no such limitation.

*Proof.* We show that for all `a, b, c ∈ T` with `Pos(b)`, `Pos(c)`, whenever both `(a ⊕ b) ⊕ c` and `a ⊕ (b ⊕ c)` are well-defined, every component of the left side equals the corresponding component of the right side.

Throughout, write `k_b` for the action point of `b` and `k_c` for the action point of `c`. Recall TumblerAdd's constructive definition: for `x ⊕ w` with `w` having action point `k`, the result has `(x ⊕ w)ᵢ = xᵢ` for `i < k` (prefix copy), `(x ⊕ w)ₖ = xₖ + wₖ` (advance), and `(x ⊕ w)ᵢ = wᵢ` for `i > k` (tail copy), with `#(x ⊕ w) = #w` (the result-length identity from TA0).

*Lengths (left side, and the inner right factor).* By TA0's result-length identity, `#(a ⊕ b) = #b` — TA0's preconditions `Pos(b)` and `k_b ≤ #a` are supplied by the formal contract. Applying it again: `#((a ⊕ b) ⊕ c) = #c`, where TA0's preconditions `Pos(c)` and `k_c ≤ #(a ⊕ b) = #b` reduce to the formal-contract `Pos(c)` and `k_c ≤ #b` after substituting the identity just computed. For the inner right factor, `#(b ⊕ c) = #c`, licensed by the formal-contract `Pos(c)` and `k_c ≤ #b`. The outer right-side length `#(a ⊕ (b ⊕ c))` is not yet derivable at this point: TA0 applied to `a ⊕ s` (where `s = b ⊕ c`) requires `Pos(s)` and `actionPoint(s) ≤ #a`, neither of which is in the formal contract — the first is discharged in *Action point of `s`* by exhibiting a nonzero component of `s` to witness TA-Pos's defining existential, and the second is discharged in *Domain conditions* via the subsumption `k_b ≤ #a ⟹ min(k_b, k_c) ≤ #a`. We defer the `#(a ⊕ (b ⊕ c)) = #(b ⊕ c) = #c` step to *Right-side length* below, after those prerequisites have been established.

*Action point of `s = b ⊕ c`.* We must first establish `Pos(s)` so that ActionPoint and TA0 become applicable to `s` (both require `Pos(·)` on their tumbler argument), and only then compute `actionPoint(s)` to expand the right side `a ⊕ s`. By TumblerAdd on `b ⊕ c` (licensed by `Pos(c)`, which the formal contract supplies as a precondition): `sᵢ = bᵢ` for `i < k_c`, `s_{k_c} = b_{k_c} + c_{k_c}`, and `sᵢ = cᵢ` for `i > k_c`. For `i < min(k_b, k_c)`, we have `i < k_b` (so `bᵢ = 0` by definition of action point) and `i < k_c` (so `sᵢ = bᵢ = 0` by the prefix-copy rule). At position `min(k_b, k_c)` three sub-cases arise, exhaustive by NAT-order's trichotomy on ℕ instantiated at `m = k_b, n = k_c` (exactly one of `k_b < k_c`, `k_b = k_c`, `k_b > k_c` holds). If `k_b < k_c`: `s_{k_b} = b_{k_b} > 0`, since `k_b` is the action point of `b` and `k_b < k_c` places it in the prefix-copy region. If `k_b = k_c`: `s_{k_b} = b_{k_b} + c_{k_b}`, and `s_{k_b} > 0` routes per-summand positivity through NAT-addcompat rather than background monotonicity of `+`. NAT-addcompat's left order-compatibility (`n ≥ p ⟹ m + n ≥ m + p`, instantiated at `m = b_{k_b}, n = c_{k_b}, p = 1`), applied to ActionPoint's minimum-nonzero `c_{k_b} ≥ 1` at `w = c`, lifts to the non-strict `b_{k_b} + c_{k_b} ≥ b_{k_b} + 1`; NAT-addcompat's strict successor inequality (`n < n + 1`, instantiated at `n = b_{k_b}`) supplies the strict `b_{k_b} + 1 > b_{k_b}`; NAT-order's transitivity composes the non-strict `b_{k_b} + c_{k_b} ≥ b_{k_b} + 1` with the strict `b_{k_b} + 1 > b_{k_b}` into the strict `b_{k_b} + c_{k_b} > b_{k_b}`, and ActionPoint's minimum-nonzero at `w = b` giving `b_{k_b} ≥ 1` yields `s_{k_b} = b_{k_b} + c_{k_b} > 0` — the same per-summand-positivity-through-`+` pattern TumblerAdd's dominance proof applies at its structurally identical `aₖ + wₖ > wₖ` site. If `k_b > k_c`: `s_{k_c} = b_{k_c} + c_{k_c} = 0 + c_{k_c} = c_{k_c} > 0`, since `k_c < k_b` gives `b_{k_c} = 0`. In every sub-case `s` has a nonzero component at position `min(k_b, k_c)`. To witness TA-Pos's defining existential `(E i : 1 ≤ i ≤ #s : sᵢ ≠ 0)` at `i = min(k_b, k_c)`, the index bounds `1 ≤ min(k_b, k_c) ≤ #s` must also hold — TA-Pos's existential quantifier ranges over `1 ≤ i ≤ #s`, and only an index within this range is a licit witness. We discharge both bounds case by case over the three trichotomy branches established above. *Lower bound `1 ≤ min(k_b, k_c)`.* In the sub-case `k_b < k_c`, `min(k_b, k_c) = k_b`, and ActionPoint's first postcondition `1 ≤ actionPoint(w) ≤ #w` at `w = b` (precondition `Pos(b)` from the formal contract) supplies `1 ≤ k_b`, yielding `1 ≤ min(k_b, k_c)`; in the sub-case `k_b = k_c`, `min(k_b, k_c) = k_b`, and the same ActionPoint postcondition at `w = b` again supplies `1 ≤ k_b`; in the sub-case `k_b > k_c`, `min(k_b, k_c) = k_c`, and ActionPoint's first postcondition at `w = c` (precondition `Pos(c)` from the formal contract) supplies `1 ≤ k_c`, yielding `1 ≤ min(k_b, k_c)`. *Upper bound `min(k_b, k_c) ≤ #s`.* *Lengths* established `#s = #(b ⊕ c) = #c`, and ActionPoint's first postcondition at `w = c` supplies `k_c ≤ #c`. In the sub-case `k_b < k_c`, `min(k_b, k_c) = k_b`, and NAT-order's definition `m ≤ n ⟺ m < n ∨ m = n` instantiated at `m = k_b, n = k_c` (first disjunct, from the sub-case hypothesis) unfolds `k_b < k_c` into `k_b ≤ k_c`, which NAT-order's transitivity of `≤` (derivable from NAT-order's transitivity of `<` and the `m ≤ n ⟺ m < n ∨ m = n` definition by case analysis on the equality disjuncts) composes with `k_c ≤ #c` to yield `k_b ≤ #c`, and *Lengths*' `#s = #c` then gives `min(k_b, k_c) = k_b ≤ #s`; in the sub-cases `k_b = k_c` and `k_b > k_c`, `min(k_b, k_c) = k_c`, and ActionPoint's `k_c ≤ #c` composed with *Lengths*' `#s = #c` directly yields `min(k_b, k_c) = k_c ≤ #s`. In every sub-case `1 ≤ min(k_b, k_c) ≤ #s`, so the exhibited nonzero component of `s` at position `min(k_b, k_c)` is a licit witness of TA-Pos's defining existential `(E i : 1 ≤ i ≤ #s : sᵢ ≠ 0)` at `i = min(k_b, k_c)` — thereby establishing `Pos(s)`, i.e., `Pos(b ⊕ c)`, the middle link in the precondition chain `Pos(b), Pos(c)` → `Pos(b ⊕ c)` → `actionPoint(b ⊕ c)` that the right-side expansion and the `actionPoint(b ⊕ c) = min(k_b, k_c)` postcondition both consume. With `Pos(s)` now in hand, ActionPoint is applicable to `s`: its definition `actionPoint(s) = min({i : 1 ≤ i ≤ #s ∧ sᵢ ≠ 0})` consumes the zeros `sᵢ = 0` for `i < min(k_b, k_c)` established above together with the nonzero component at `min(k_b, k_c)` to yield `actionPoint(s) = min(k_b, k_c)`.

*Domain conditions.* The left side `(a ⊕ b) ⊕ c` requires two well-defined additions: `a ⊕ b` requires `Pos(b)` and `k_b ≤ #a` (TA0), and `(a ⊕ b) ⊕ c` requires `Pos(c)` and `k_c ≤ #(a ⊕ b) = #b` (TA0 applied to the intermediate result). The right side `a ⊕ (b ⊕ c)` requires `b ⊕ c` with `Pos(c)` and `k_c ≤ #b` (TA0), and `a ⊕ s` with `Pos(s)` and `actionPoint(s) = min(k_b, k_c) ≤ #a` (TA0) — `Pos(s)` discharged by the *Action point of `s`* paragraph above, which is why that paragraph must precede this one. The domains are asymmetric: the left requires `k_b ≤ #a`, the right requires only `min(k_b, k_c) ≤ #a`. The subsumption `k_b ≤ #a ⟹ min(k_b, k_c) ≤ #a` is discharged case by case over the three trichotomy branches: in the sub-cases `k_b < k_c` and `k_b = k_c`, `min(k_b, k_c) = k_b`, so the formal-contract precondition `k_b ≤ #a` supplies `min(k_b, k_c) ≤ #a` directly; in the sub-case `k_b > k_c`, `min(k_b, k_c) = k_c`, and NAT-order's definition `m ≤ n ⟺ m < n ∨ m = n` instantiated at `m = k_c, n = k_b` (first disjunct, from the sub-case hypothesis) unfolds `k_c < k_b` into `k_c ≤ k_b`, which NAT-order's transitivity of `≤` composes with `k_b ≤ #a` to yield `k_c ≤ #a`. In every sub-case `min(k_b, k_c) ≤ #a`, so the left-side conditions subsume the right-side conditions. The intersection of both domains is therefore `k_b ≤ #a` and `k_c ≤ #b`. We assume these hold and show the values agree by exhaustive case analysis on the relationship between `k_b` and `k_c` — NAT-order's trichotomy on ℕ, instantiated at `m = k_b, n = k_c`, asserts exactly one of `k_b < k_c`, `k_b = k_c`, `k_b > k_c` holds, making Cases 1–3 below exhaustive.

*Right-side length.* With `Pos(s)` now discharged by *Action point of `s`* (via the nonzero component of `s` at position `min(k_b, k_c)` witnessing TA-Pos's defining existential) and `actionPoint(s) = min(k_b, k_c) ≤ #a` now discharged by *Domain conditions* (via the subsumption `k_b ≤ #a ⟹ min(k_b, k_c) ≤ #a` with `k_b ≤ #a` from the formal contract), TA0's preconditions for `a ⊕ s` are satisfied. TA0's result-length identity then gives `#(a ⊕ s) = #s`, i.e., `#(a ⊕ (b ⊕ c)) = #(b ⊕ c) = #c` (the second equality from the inner right factor computed in *Lengths*). Both sides of the theorem equality therefore have length `#c`.

*Case 1: `k_b < k_c`.* The action point of `s` is `k_b`, with `s_{k_b} = b_{k_b}` (from the prefix-copy region of `b ⊕ c`, since `k_b < k_c`).

Let `r = a ⊕ b`. By TumblerAdd: `rᵢ = aᵢ` for `i < k_b`, `r_{k_b} = a_{k_b} + b_{k_b}`, and `rᵢ = bᵢ` for `i > k_b`.

*Left side* `(r ⊕ c)` with action point `k_c`: for `i < k_b` we have `i < k_c`, so `(r ⊕ c)ᵢ = rᵢ = aᵢ`. At `i = k_b < k_c`: position `k_b` falls in the prefix-copy region of `r ⊕ c`, so `(r ⊕ c)_{k_b} = r_{k_b} = a_{k_b} + b_{k_b}`. For `k_b < i < k_c`: `(r ⊕ c)ᵢ = rᵢ = bᵢ` (prefix-copy from `r`, and `i > k_b` puts `rᵢ` in the tail-copy region of `a ⊕ b`). At `i = k_c`: `(r ⊕ c)_{k_c} = r_{k_c} + c_{k_c} = b_{k_c} + c_{k_c}`, since `k_c > k_b` gives `r_{k_c} = b_{k_c}` by the tail-copy rule of `a ⊕ b`. For `i > k_c`: `(r ⊕ c)ᵢ = cᵢ`.

*Right side* `(a ⊕ s)` with action point `k_b`: for `i < k_b`, `(a ⊕ s)ᵢ = aᵢ`. At `i = k_b`: `(a ⊕ s)_{k_b} = a_{k_b} + s_{k_b} = a_{k_b} + b_{k_b}`. For `i > k_b`: `(a ⊕ s)ᵢ = sᵢ` by the tail-copy rule. Expanding `sᵢ` via TumblerAdd on `b ⊕ c`: for `k_b < i < k_c`, `sᵢ = bᵢ` (prefix-copy, since `i < k_c`); at `i = k_c`, `s_{k_c} = b_{k_c} + c_{k_c}` (advance); for `i > k_c`, `sᵢ = cᵢ` (tail-copy).

Comparing position by position: `aᵢ = aᵢ` for `i < k_b`; `a_{k_b} + b_{k_b} = a_{k_b} + b_{k_b}` at `k_b`; `bᵢ = bᵢ` for `k_b < i < k_c`; `b_{k_c} + c_{k_c} = b_{k_c} + c_{k_c}` at `k_c`; `cᵢ = cᵢ` for `i > k_c`. Every component agrees.

*Case 2: `k_b = k_c = k`.* The action point of `s` is `k`, with `s_k = b_k + c_k`.

Let `r = a ⊕ b`: `rᵢ = aᵢ` for `i < k`, `r_k = a_k + b_k`, `rᵢ = bᵢ` for `i > k`. The left side `(r ⊕ c)` has action point `k`: for `i < k`, `(r ⊕ c)ᵢ = rᵢ = aᵢ`; at `k`, `(r ⊕ c)_k = r_k + c_k = (a_k + b_k) + c_k`; for `i > k`, `(r ⊕ c)ᵢ = cᵢ`. The right side `(a ⊕ s)` has action point `k`: for `i < k`, `(a ⊕ s)ᵢ = aᵢ`; at `k`, `(a ⊕ s)_k = a_k + s_k = a_k + (b_k + c_k)`; for `i > k`, `(a ⊕ s)ᵢ = sᵢ = cᵢ`.

At position `k`, the left gives `(a_k + b_k) + c_k` and the right gives `a_k + (b_k + c_k)`. These are equal by NAT-addassoc (NatAdditionAssociative), which states `(m + n) + p = m + (n + p)` for every `m, n, p ∈ ℕ`: instantiated with `a_k, b_k, c_k`, it gives `(a_k + b_k) + c_k = a_k + (b_k + c_k)`. All other positions agree by direct comparison.

*Case 3: `k_b > k_c`.* The action point of `s` is `k_c`, with `s_{k_c} = b_{k_c} + c_{k_c} = 0 + c_{k_c} = c_{k_c}` (since `k_c < k_b` gives `b_{k_c} = 0`).

Let `r = a ⊕ b`: `rᵢ = aᵢ` for `i < k_b`, `r_{k_b} = a_{k_b} + b_{k_b}`, `rᵢ = bᵢ` for `i > k_b`. The left side `(r ⊕ c)` has action point `k_c`. Since `k_c < k_b`: for `i < k_c` we have `i < k_b`, so `(r ⊕ c)ᵢ = rᵢ = aᵢ` (both prefix-copy rules apply). At `i = k_c < k_b`: `r_{k_c} = a_{k_c}` (position `k_c` falls in the prefix-copy region of `a ⊕ b`), so `(r ⊕ c)_{k_c} = r_{k_c} + c_{k_c} = a_{k_c} + c_{k_c}`. For `i > k_c`: `(r ⊕ c)ᵢ = cᵢ`. The components of `r` at and beyond `k_b` — where `b`'s contribution appears — are entirely overwritten by `c`'s tail, since `k_c < k_b`.

The right side `(a ⊕ s)` has action point `k_c`: for `i < k_c`, `(a ⊕ s)ᵢ = aᵢ`; at `k_c`, `(a ⊕ s)_{k_c} = a_{k_c} + s_{k_c} = a_{k_c} + c_{k_c}`; for `i > k_c`, `(a ⊕ s)ᵢ = sᵢ = cᵢ` (since `sᵢ = cᵢ` for `i > k_c` by the tail-copy rule of `b ⊕ c`).

Comparing: `aᵢ = aᵢ` for `i < k_c`; `a_{k_c} + c_{k_c} = a_{k_c} + c_{k_c}` at `k_c`; `cᵢ = cᵢ` for `i > k_c`. Every component agrees. The displacement `b` is entirely overwritten — TumblerAdd's tail-replacement semantics means the shallower displacement `c` discards everything below its action point on both sides, rendering `b`'s deeper contribution invisible in the final result.

In all three cases, both sides produce the same sequence of length `#c`, so `(a ⊕ b) ⊕ c = a ⊕ (b ⊕ c)` by T3 (CanonicalRepresentation). ∎

**Addition is not commutative.** We do NOT require `a ⊕ b = b ⊕ a`. The operands play asymmetric roles: the first is a *position*, the second is a *displacement*. Swapping them is semantically meaningless. Gregory's `absadd` confirms: the first argument supplies the prefix, the second the suffix — the reverse call gives a different (and typically wrong) result.

**There is no multiplication or division.** Gregory's analysis of the complete codebase confirms: no `tumblermult`, no `tumblerdiv`, no scaling operation of any kind. The arithmetic repertoire is: add, subtract, increment, compare. Tumblers are *addresses*, not quantities. You don't multiply two file paths or divide an address by three.

**Tumbler differences are not counts.** Nelson is emphatic: "A tumbler-span is not a conventional number, and it does not designate the number of bytes contained. It does not designate a number of anything." The difference between two addresses specifies *boundaries*, not *cardinality*. What lies between those boundaries depends on the actual population of the tree, which is dynamic and unpredictable. Between sibling addresses 3 and 7, document 5 might have arbitrarily many descendants — the span from 3 to 7 encompasses all of them, and their count is unknowable from the addresses alone. The arithmetic is an *addressing calculus*, not a *counting calculus*.

The absence of algebraic structure beyond a monotone shift is not a deficiency. It is the *minimum* that addressing requires. An ordered set with an order-preserving shift and an allocation increment is sufficient. Anything more would be unused machinery and unverified obligation.
