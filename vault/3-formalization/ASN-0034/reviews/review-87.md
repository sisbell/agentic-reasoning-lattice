### `Span`'s Formal Contract omits the action-point precondition required for `s ⊕ ℓ` to be well-defined
**Foundation**: `TA0 (WellDefinedAddition)` — `a ⊕ w ∈ T` requires both `Pos(w)` and `actionPoint(w) ≤ #a`; without the second condition, `s ⊕ ℓ` is undefined.
**ASN**: The `Definition (Span)` Formal Contract states: "`span(s, ℓ) = {t ∈ T : s ≤ t < s ⊕ ℓ}`, where `s ∈ T` is the start address and `ℓ ∈ T` with `Pos(ℓ)` is the displacement length". The Vocabulary entry is similarly partial: "span(s, ℓ) — set of tumblers in range: {t ∈ T : s ≤ t < s ⊕ ℓ}, where s is start address and ℓ is a positive tumbler displacement". The corresponding well-formedness condition — `actionPoint(ℓ) ≤ #s` — appears only as a precondition of `T12 (SpanWellDefinedness)`: "A span `(s, ℓ)` is well-formed when `Pos(ℓ)` and the action point `k` of `ℓ` satisfies `k ≤ #s` (the TA0 precondition for `s ⊕ ℓ`)."
**Issue**: The Span definition's Formal Contract declares `span(s, ℓ)` with only `Pos(ℓ)`, yet the set it defines refers to `s ⊕ ℓ`, which TA0 requires `actionPoint(ℓ) ≤ #s` for. Under the Span contract as written, the set `{t ∈ T : s ≤ t < s ⊕ ℓ}` is undefined whenever `ℓ`'s action point exceeds `#s` — the upper bound does not denote a tumbler in T. T12 treats this as a subsequent "well-formedness" refinement, but the base definition itself is not total on its stated domain. A downstream pipeline that formalizes Span from its Formal Contract alone would accept ill-formed pairs `(s, ℓ)` and attempt to evaluate an undefined expression.
**What needs resolving**: Either fold the action-point condition into Span's own contract (so that `span(s, ℓ)` is only defined for `(s, ℓ)` with `Pos(ℓ)` and `actionPoint(ℓ) ≤ #s`), or restructure so that Span is a partial construct whose totality is established by T12 — in which case every downstream use of Span must cite T12 rather than Span directly. The Vocabulary entry should be updated to match whichever convention is chosen.

### `D2`'s Step 2 derives `Pos(w)` circularly from the hypothesis that `a ⊕ w` is defined
**Foundation**: `TA0 (WellDefinedAddition)` — `Pos(w)` is a *precondition* for `a ⊕ w` to be defined; it cannot be recovered from the defined-ness of the result.
**ASN**: `D2 (DisplacementUnique)`, Step 2: "For w: the hypothesis states a ⊕ w = b, so the addition is well-defined. TA0 requires Pos(w) (satisfied since the addition produces a result) and actionPoint(w) ≤ #a." D2's Preconditions list includes `a ⊕ w = b` but not `Pos(w)` or `actionPoint(w) ≤ #a` directly.
**Issue**: The proof argues "Pos(w) is satisfied since the addition produces a result." But TA0 is a precondition on inputs, not a conclusion from the result. If `w` were a zero tumbler, `a ⊕ w` is undefined as a TumblerAdd expression — the hypothesis `a ⊕ w = b` is ill-formed, not a witness that `Pos(w)` holds. The preconditions for TA0 must be furnished independently, not recovered from the formula they are preconditions of. The same circularity applies to `actionPoint(w) ≤ #a`.
**What needs resolving**: Either add `Pos(w)` and `actionPoint(w) ≤ #a` to D2's Preconditions so TA-LC's preconditions are established from explicit hypotheses, or reformulate D2 so that the hypothesis is stated in terms that entail well-definedness (e.g., "for any `w ∈ T` with `Pos(w)` and `actionPoint(w) ≤ #a` such that `a ⊕ w = b`").

### `T4`'s "when present" qualifiers on field-length constraints are informal
**Foundation**: `T4c (LevelDetermination)` — formalizes which fields are present as a function of `zeros(t)`.
**ASN**: T4's statement: "`α ≥ 1`, `β ≥ 1` when present, `γ ≥ 1` when present, `δ ≥ 1` when present". T4's Formal Contract axiom: "every field present in the address has at least one component (non-empty field constraint)". The notion of "present" is not defined in T4 itself — it is established only later by T4c's bijection between `zeros(t)` and hierarchical level.
**Issue**: T4 states its non-empty field constraint in terms of "present" fields, but the formalization of presence (via `zeros(t)`) is a consequence theorem (T4c) that itself depends on T4b (UniqueParse), which in turn depends on T3 and on the T4 constraints. T4's axiom as written refers forward to a notion it does not itself define — the pipeline cannot evaluate T4's constraints without first evaluating T4c. This is not an outright contradiction, but the axiom is not self-contained: a reader or formalizer needs to know which fields are "present" for a given `t` before T4 can be checked, yet T4 is what is supposed to determine that.
**What needs resolving**: Either reformulate T4 so the field presence notion is defined internally (e.g., by the positional structure of zeros directly, without appeal to T4c), or restructure so that "field presence" is a primitive notion in T4 and T4c derives the bijection with hierarchical level from that primitive rather than defining it.
