### "Sibling output" used as justification for T4-validity of domain prefixes, but domain prefixes can be base addresses
**Foundation**: (internal consistency — terminology / justification precision)
**ASN**: GlobalUniqueness Case 5: *"By T10a.4, every domain prefix is a T4-valid address (it is a sibling output of a conforming allocator)."* Domain definition: *"dom(A) = {tₙ : n ≥ 0} where tₙ₊₁ = inc(tₙ, 0) — the base address together with every sibling output of the inc(·, 0) chain."* T10a: *"Allocators produce sibling outputs exclusively by inc(·, 0)."*
**Issue**: The domain definition explicitly distinguishes the base address (t₀) from "sibling outputs" (t₁, t₂, ..., produced by `inc(·, 0)`). T10a establishes this distinction: "sibling outputs" are `inc(·, 0)` products; the base is given by initialization (root) or child-spawning (non-root). A "domain prefix" is the parent-domain element from which a child allocator was spawned. Nothing in T10a prevents spawning from the parent's base address t₀ (before any sibling has been produced) — the parent "performs one `inc(·, k')`" on any current domain element. If a child is spawned from t₀, the child's domain prefix is t₀, which is *not* a sibling output by the ASN's own terminology. The T4-validity claim is correct — T10a.4's induction covers all domain elements: the root base by initialization, non-root bases by the child-spawning step, and siblings by the `inc(·, 0)` preservation step. But the parenthetical justification "it is a sibling output" points a verifier to only the `inc(·, 0)` preservation step, which is the wrong guarantee for base addresses. A verifier tracing this chain would need the initialization axiom (root base) or the induction's child-spawning step (non-root base), neither of which "sibling output" invokes. The same misidentification appears in the formal contract's preconditions: *"every a ∈ dom(A) satisfies t ≼ a (by TA5(b), TA5(d), TA5-SigValid, and T10a.1)"* — where the bridge argument for the base c₀ = t₀ uses TA5(b) and TA5(d) from the child-spawning `inc(·, k')`, not from sibling-stream `inc(·, 0)`.
**What needs resolving**: Replace "sibling output" with "output" (or "domain element") in the Case 5 justification, so the parenthetical correctly covers both base addresses and sibling outputs under T10a.4. Alternatively, constrain child-spawning to sibling outputs only (excluding the base), making "sibling output" accurate — but this would need an explicit statement in T10a and would require the root allocator to produce at least one sibling before spawning any child.
