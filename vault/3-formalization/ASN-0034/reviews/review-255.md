### Free variable `m` in TS1's property statement and contract preconditions
**Foundation**: A precisely-specified universal property must bind every variable appearing in the quantifier body, either by the outer quantifier itself or by a pre-declared constant; other properties in this ASN (OrdinalShift, OrdinalDisplacement, Divergence, TA1-strict) all bind every variable of the contract either as an operand or as a locally-introduced abbreviation inside the Definition clause.
**ASN**: TS1's stated property — `(A v₁, v₂, n : n ≥ 1 ∧ #v₁ = #v₂ = m ∧ v₁ < v₂ : shift(v₁, n) < shift(v₂, n))` — and TS1's *Formal Contract* Preconditions — `v₁ ∈ T, v₂ ∈ T, n ≥ 1, #v₁ = #v₂ = m, v₁ < v₂`. The identifier `m` appears in both but is not among the quantified variables `v₁, v₂, n` and is not a pre-declared ASN-level constant (no foundation names `m`; OrdinalShift and OrdinalDisplacement each introduce `m` as a *locally bound* shorthand for `#v`).
**Issue**: TS1's `m` is a free variable: the quantifier binds `v₁, v₂, n` only, and the clause `#v₁ = #v₂ = m` uses `m` as if it were externally given. Under a strict reading this makes the property ill-formed (its truth-value depends on an unbound symbol); under the intended reading — "there exists `m` with `#v₁ = m ∧ #v₂ = m`", which collapses to `#v₁ = #v₂` — the symbol `m` is redundant. The proof reinforces the ambiguity by writing "Fix v₁, v₂ ∈ T with #v₁ = #v₂ = m" as though `m` were locally named at that point, while the *Preconditions* list still carries `m` as a contract-level name. Per-step citation discipline is unaffected, but Lamport-style precision — every symbol has exactly one meaning, bound where used — is violated at the very statement of the property.
**What needs resolving**: TS1 must make `m`'s binding explicit: either add `m` to the quantifier and contract preconditions so that it is universally quantified alongside `v₁, v₂, n`, or remove `m` from the property statement and contract (replacing `#v₁ = #v₂ = m` with `#v₁ = #v₂`) and introduce `m = #v₁` as a local abbreviation only within the proof where it is used.

### Citation-scope paragraph demotes NAT-zero to "a role of T0", but OrdinalDisplacement and OrdinalShift cite NAT-zero as a distinct foundation
**Foundation**: Per-step citation discipline as articulated throughout this ASN: "every operator consumed at a step be sourced at that step"; the `.md` file-location disclosure is the Depends-backed handle a reviser uses to locate the axiom a citation consumes.
**ASN**: TS1's *Citation scope* paragraph — in the upstream-cone sentence — lists `NAT-addcompat.md, NAT-cancel.md, NAT-closure.md,` then writes "the ℕ-component role of T0 standing in for any 'NAT-zero' reading". This asserts NAT-zero is not a distinct file but a rename of part of T0's content. Yet OrdinalDisplacement's *Depends* has a full-weight NAT-zero entry — "NAT-zero (NatZeroMinimum) — invoked to supply `0 ∈ ℕ` … NAT-zero's axiom `(A n ∈ ℕ :: 0 ≤ n)` … No other listed NAT-* axiom supplies this membership: NAT-closure's additive identity clause … presupposes, rather than supplies, `0 ∈ ℕ`; NAT-order's trichotomy … NAT-wellorder names a least element of ℕ without identifying it as `0`." OrdinalShift's *Depends* likewise has a full-weight NAT-zero entry — "NAT-zero (NatZeroMinimum) — invoked at the first arithmetic step … NAT-zero's axiom `(A n ∈ ℕ :: 0 ≤ n)` instantiated at `n = vₘ` supplies the lower-bound premise … No other listed NAT-* axiom supplies this premise".
**Issue**: The two pieces of the document contradict each other about whether NAT-zero is a foundation. The Depends entries treat NAT-zero as a peer of NAT-closure, NAT-order, NAT-wellorder — explicitly differentiating its content from each of the others and naming a distinct axiom `(A n ∈ ℕ :: 0 ≤ n)` — so a reviser following the per-step discipline would expect `NAT-zero.md`. The citation-scope paragraph instead says NAT-zero's content is supplied by T0's ℕ-component role, with no `NAT-zero.md` file. A reviser told by the Depends entry "tighten NAT-zero" has no named file to tighten; a reviser told by the citation-scope paragraph "NAT-zero is T0" must then interpret OrdinalDisplacement's/OrdinalShift's differentiation arguments (which explicitly contrast NAT-zero against T0's content by pointing out that other NAT-* axioms fail to supply `0 ∈ ℕ` or `0 ≤ n`) as routing that indirection — which is not what the Depends text says.
**What needs resolving**: Either NAT-zero exists as its own foundation file (and the citation-scope paragraph must add `NAT-zero.md` to its file list, moving NAT-zero out of the "standing in for T0" clause), or NAT-zero does not exist as a distinct foundation (and OrdinalDisplacement's and OrdinalShift's Depends entries must re-source the `0 ∈ ℕ` / `0 ≤ n` content to T0 — with their differentiation arguments against other NAT-* axioms rewritten so they no longer treat NAT-zero as a peer).
