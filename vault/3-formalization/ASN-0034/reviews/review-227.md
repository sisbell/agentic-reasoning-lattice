# Cone Review — ASN-0034/TA1-strict (cycle 7)

*2026-04-18 03:00*

### Divergence Postconditions omit well-definedness, ℕ-typing, and bounds of `divergence(a, b)`
**Foundation**: Downstream callers consume `divergence(a, b)` as a ℕ-valued expression in comparisons (TA1-strict's precondition `actionPoint(w) ≥ divergence(a, b)`) and as a specific value in case analyses (TA1-strict's case-(ii) rule-out reads `j = #a + 1` and `j = #b + 1` off sub-cases (ii-a)/(ii-b)). T0's convention is that every operator and typing consumed downstream should be surfaced where it is obtained.
**ASN**: Divergence's `*Postconditions:*` lists only `divergence(a, b) = divergence(b, a) for all a ≠ b`. The body establishes (i) well-definedness as a function (uniqueness of case (i)'s `k` via NAT-wellorder; case (ii)'s arithmetic value), (ii) ℕ-typing (case (i)'s `k ∈ ℕ` via index range, case (ii)'s `#a + 1 / #b + 1 ∈ ℕ` via NAT-closure), (iii) range bounds (case (i): `1 ≤ k, k ≤ #a, k ≤ #b`; case (ii): `divergence ∈ {#a + 1, #b + 1}`), (iv) exhaustiveness — none of which appear as postconditions.
**Issue**: Sibling properties expose their load-bearing facts explicitly: TumblerAdd lists four postconditions (membership, length identity, ordering, dominance); TA0 re-exports two of them for cleaner downstream citation; ActionPoint lists three (bounds, zeros-below, minimum-nonzero value). Divergence's contract exposes only symmetry, leaving TA1-strict and every other downstream caller to re-derive "divergence(a, b) is a natural number equal to one of these specific values" from the Definition body by direct inspection. A mechanical contract-translator reading only Divergence's Postconditions will not find the well-definedness or typing facts TA1-strict's precondition `actionPoint(w) ≥ divergence(a, b)` rests on.
**What needs resolving**: Either surface the well-definedness, ℕ-typing, case-exhaustiveness, and range/value facts as explicit postconditions of Divergence (paralleling TumblerAdd's four-postcondition profile), or document a convention under which Definitions may keep load-bearing facts in the body rather than the Postconditions field and amend Postconditions sections of sibling function-introducing properties (ActionPoint, TumblerAdd) to match the same convention.

### Inconsistent precondition-source annotations across properties of this ASN
**Foundation**: The ASN uses an annotation style `X (SourceProp)` or `X (SourceProp, this ASN)` on precondition entries to declare where each hypothesis is sourced (e.g., TumblerAdd: `Pos(w) (TA-Pos), actionPoint(w) ≤ #a (ActionPoint)`; TA0: `Pos(w) (TA-Pos, this ASN), actionPoint(w) ≤ #a (ActionPoint, this ASN)`).
**ASN**: Annotation practice varies across properties introducing tumbler hypotheses: Divergence — `a ∈ T (T0), b ∈ T (T0), a ≠ b` (T0-annotated for the typing premise). TumblerAdd — `a ∈ T, w ∈ T, Pos(w) (TA-Pos), actionPoint(w) ≤ #a (ActionPoint)` (no annotation on `a ∈ T` / `w ∈ T`; bare "(TA-Pos)" without "this ASN"). TA0 — `a ∈ T, w ∈ T, Pos(w) (TA-Pos, this ASN), actionPoint(w) ≤ #a (ActionPoint, this ASN)` (no annotation on `∈ T`; uses "this ASN" suffix). ActionPoint — `w ∈ T, Pos(w)` (no annotations at all). TA1-strict — `a ∈ T, b ∈ T, w ∈ T, a < b, Pos(w), actionPoint(w) ≤ #a, actionPoint(w) ≤ #b, actionPoint(w) ≥ divergence(a, b)` (no annotations at all).
**Issue**: Only Divergence annotates `∈ T` with `(T0)`; only TA0 uses the `(…, this ASN)` suffix; TA1-strict and ActionPoint use no annotations at all. A mechanical reader (or a contract-translator following the Depends-plus-annotation convention for precondition sourcing) cannot tell whether the missing annotations elsewhere mean "sourced from T0/TA-Pos/ActionPoint by convention" or "left to the reader" or "not meant to be sourced." This parallels the previous TA1-strict finding about T0 omission from Depends — both stem from no settled convention about where `∈ T`, `Pos(·)`, and `actionPoint(·) ≤ ·` bindings are surfaced in the precondition vs. Depends split.
**What needs resolving**: Adopt one annotation policy and apply it uniformly — either annotate every precondition with its source property in parentheses (making Divergence's `(T0)` the template), or drop all parenthetical source annotations and rely on the Depends list for sourcing. Whichever route is taken, TumblerAdd, TA0, ActionPoint, TA1-strict, and Divergence should use the same precondition-annotation style.
