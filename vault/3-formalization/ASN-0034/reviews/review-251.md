# Cone Review — ASN-0034/T12 (cycle 8)

### Definition (Span) opens "a span is a pair" but the disambiguation paragraph fixes *span* as the address-set
**Foundation**: Definition (Span) — must introduce its term with one stable referent, particularly when a later narrative paragraph asserts that referent is canonical "throughout this document".
**ASN**: Definition (Span), opening sentence: "A *span* is a pair `(s, ℓ)` where `s ∈ T` is a start address and `ℓ ∈ T` is a length". Same definition body, a few clauses later: "the set `{t ∈ T : s ≤ t < s ⊕ ℓ}` — **the span** — is well-defined; pairs `(s, ℓ)` violating any of them are not spans." Formal Contract: "`span(s, ℓ) = {t ∈ T : s ≤ t < s ⊕ ℓ}`" — treats `span(·,·)` as a function from pairs to sets. The disambiguation paragraph in the "Spans" section then asserts: "Throughout this document, *span* denotes the address-set `span(s, ℓ) = {t ∈ T : s ≤ t < s ⊕ ℓ}` fixed by Definition (Span)". T12 postcondition (b): "`s ∈ span(s, ℓ)`" — uses `span(s, ℓ)` as a set one can be a member of.
**Issue**: Within Definition (Span)'s single paragraph, "span" refers to both the pair `(s, ℓ)` (opening and closing sentences: "is a pair", "pairs ... are not spans") and the address-set `{t ∈ T : s ≤ t < s ⊕ ℓ}` (middle phrase: "the set ... — the span — is well-defined"). This duplication is not between span-as-address-set and span-as-content (which the disambiguation paragraph already addresses) but between span-as-pair and span-as-set — a distinction the disambiguation paragraph does not cover. The Formal Contract resolves the ambiguity functionally (`span(s, ℓ)` is the set assigned to the pair), but Definition (Span)'s own prose uses the bare word "span" for the input (pair) and the output (set) interchangeably, and the downstream convention ("throughout this document, span denotes the address-set") contradicts the opening "a span is a pair". A reviser who takes Definition (Span)'s opening at face value reads `s ∈ span(s, ℓ)` in T12(b) as category-mismatched (a tumbler being a member of a pair).
**What needs resolving**: Either (a) revise Definition (Span)'s opening so "span" introduces the address-set directly (e.g., "A *span* is the set `span(s, ℓ) = {t ∈ T : s ≤ t < s ⊕ ℓ}` determined by a pair `(s, ℓ)` satisfying ..."), aligning with the Formal Contract and the "throughout this document" convention; or (b) keep "pair" as the primary referent and name the set by a distinct term (e.g., *span-set* or *the address-set of a span*) consistently across Definition (Span)'s body, the disambiguation paragraph, and T12's postconditions. Whichever choice is made, apply it uniformly in the definition's opening, its middle ("the set — the span — is well-defined"), its closing ("pairs ... are not spans"), the disambiguation paragraph, and every downstream invocation of `span(s, ℓ)` as a set.
