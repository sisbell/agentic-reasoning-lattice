# Cone Review — ASN-0034/TA6 (cycle 4)

*2026-04-17 15:09*

### T4c's precondition on T4b misattributes a biconditional that is already in T4's axiom
**Foundation**: T4 (HierarchicalParsing), Formal Contract *Axiom* and *Preconditions* (T4c → T4b edge).
**ASN**: T4's Axiom explicitly carries the zero↔separator biconditional: "T4 further stipulates that a position `i` of `t` is a *field separator* iff `tᵢ = 0`, fixing as part of the axiom the biconditional identification of the set of separator positions with the set of zero-valued positions — the two directions `tᵢ = 0 ⇒ i` is a separator and `i` is a separator `⇒ tᵢ = 0` are both licensed here rather than in body prose alone."

T4's body prose justifying the T4c → T4b edge: "T4b is essential for T4c because T4c's equality between `zeros(t)` and the separator count — from which the formula 'number of fields present equals `zeros(t) + 1`' and hence the bijection onto hierarchical level follows — rests on T4b's definitional identification of every zero-valued position with a field separator and every separator with a zero."
**Issue**: The stated rationale attributes the zero↔separator biconditional to T4b, but that biconditional is explicitly a clause of T4's own axiom (labeled "Definitional role-assignment"). If the equality `zeros(t) = separator count` is indeed what underwrites T4c's bijection, T4c can draw that biconditional from T4 directly — it does not need T4b. Beyond misattribution, this puts the T4c → T4b edge itself in question: T4c's bijection is characterised in T4's body prose purely by zero-count ("A tumbler with zero zeros addresses a node. One zero: a user account. …"), never through `fields(t)`; no other step in the T4c framing appeals to anything T4b supplies. So either the Preconditions graph overstates the T4c → T4b edge, or the genuine reason T4c consumes T4b is different from what the body claims (e.g., to give meaning to "node/user/document/element address" labels via the `fields(t)` decomposition). Downstream ASNs relying on the recorded DAG cannot tell which reading holds.
**What needs resolving**: Reconcile the T4c → T4b justification with T4's axiom. Either (a) restate the genuine reason T4c requires T4b — for instance, if T4c's postcondition needs `fields(t)` to define "node/user/document/element address", identify that as the load-bearing dependency; or (b) remove the T4c → T4b edge from Preconditions and discharge T4c from T4 alone, since both the zero-separator biconditional and the `zeros(t) ≤ 3` bound that drive T4c's bijection are already T4-axiomatic.
