# Cone Review — ASN-0034/PartitionMonotonicity (cycle 2)

### Notation paragraph: `dom(c)` is not a subset of `subtree(c)`
**Foundation**: T10a (AllocatorDiscipline) — defines `dom(A) = {tₙ : n ≥ 0}` with `t₀` the allocator's base and `tₙ₊₁ = inc(tₙ, 0)`. TA5(c) says `inc(·, 0)` preserves length; TA5(b) says `inc(·, 0)` modifies position `sig(t) = #t`. Prefix — equal-length distinct tumblers are prefix-incomparable.
**ASN**: PartitionMonotonicity, *Notation* paragraph — "write `subtree(c) = {t ∈ T : c ≼ t}`… This set is strictly broader than T10a's formal `dom(c) = {cₙ : n ≥ 0}`: `subtree(c)` additionally contains the bases of child allocators spawned from `c`'s siblings, those children's sibling streams, and so on transitively."
**Issue**: The claim that `subtree(c)` is "strictly broader than" `dom(c)` — implying `dom(c) ⊆ subtree(c)` — is false. For `n ≥ 1`, `cₙ = inc(cₙ₋₁, 0)` has the same length as `c` (TA5(c)) but differs from `c` at position `sig(c) = #c` (TA5(b), TA5(c)), so `c ⋠ cₙ` (Prefix, since equal length requires identity). Hence `dom(c) ∩ subtree(c) = {c}`; the two sets overlap only at the base. Further, the sentence asserts `subtree(c)` contains "bases of child allocators spawned from `c`'s siblings" — but `inc(s, k')` for a sibling `s ≠ c` preserves `s[#c] ≠ c[#c]` (TA5(b)), so `c ⋠ inc(s, k')` and those bases are not in `subtree(c)` either. The Notation paragraph's characterisation of the `subtree`/`dom` relation is inverted: `subtree(c)` is not a superset but an almost-disjoint set relative to `dom(c)`.
**What needs resolving**: Either re-state the relation correctly (these are different slices of address space — `dom(c)` is the same-length sibling stream of `c`, while `subtree(c)` is the prefix-closure below `c`) or reframe the Notation so that whichever set the proof actually needs is defined consistently with what the subsequent arguments require.

### PartitionMonotonicity decomposes `subtree(p)` by a rule that leaves allocated addresses uncovered
**Foundation**: T10a (AllocatorDiscipline) — when a child allocator with base `c₁ = inc(p, 1)` spawns, its sibling stream produces `c₁, inc(c₁, 0), inc(inc(c₁, 0), 0), …` with values `1, 2, 3, …` at position `#p + 1` (by TA5(c) acting at `sig`). Each of these siblings has `p` as a prefix (TA5(b) preserves positions `1..#p`), so every member of `dom(c₁)` lies in `subtree(p)`.
**ASN**: PartitionMonotonicity — *Cross-depth ordering* asserts "Every address in `subtree(c₁)` therefore has component `≥ 1` at position `#p + 1`" and reasons through `dom(c₁)`'s sibling values `2, 3, …`; *Total ordering* asserts "every allocated address `a ≠ p` belongs to exactly one child's subtree" and "the component `a_{#p+1}` decides which subtree — `0` for `subtree(c₂)`, `≥ 1` for `subtree(c₁)`".
**Issue**: Under the Notation-paragraph definition `subtree(c) = {t : c ≼ t}`, `subtree(c₁) = {t : c₁ ≼ t}` is exactly `{t : p ≼ t ∧ #t ≥ #p+1 ∧ t[#p+1] = 1}` — strictly, *equal to* 1, not "≥ 1". The address `inc(c₁, 0)` (component `2` at `#p+1`) is allocated by `c₁`'s allocator, belongs to `subtree(p)`, but belongs to neither `subtree(c₁)` (since `t[#p+1] = 2 ≠ 1`) nor `subtree(c₂)` (since `t[#p+1] = 2 ≠ 0`). The proof's `a_{#p+1}`-based decomposition of `subtree(p)` into `{p} ∪ subtree(c₁) ∪ subtree(c₂)` therefore fails to cover every address reached by the child allocator's sibling stream — and more generally, fails to cover any allocated address whose component at `#p+1` is `≥ 2`, including every sibling of `c₁` beyond the first. The stated postcondition — total ordering of the partition's allocated addresses — then rests on a decomposition that is not exhaustive.
**What needs resolving**: Either (i) redefine the recursive decomposition so that every address allocated under the parent at `p` is placed in a uniquely determined component (for instance, decomposing `subtree(p)` by sibling-prefix bucket: for each `s ∈ dom(c₁) ∪ dom(c₂) ∪ …` present, its associated bucket `{t : s ≼ t}` plus `s` itself), or (ii) show that only addresses satisfying `t[#p+1] ∈ {0, 1}` can be allocated in a conforming system (which is false — `inc(c₁, 0) = (p, 2)` is routinely allocated), or (iii) restate the partition structure using the allocator-reach topology implied by T10a rather than the prefix-closure topology given by the Notation paragraph.

### `T10a` *Axiom* block's `Depends` line does not surface TA5a's `zeros(·)`-boundary semantics
**Foundation**: TA5a (IncrementPreservesT4) — defines `zeros(t)` and supplies the boundary conditions `k' = 1` requires `zeros(t) ≤ 3`, `k' = 2` requires `zeros(t) ≤ 2`, with preservation failing for `k' ≥ 3`.
**ASN**: T10a axiom body writes "satisfying the TA5a bounds (`k' = 1` when `zeros(t)` (TA5a) `≤ 3`, `k' = 2` when `zeros(t) ≤ 2`) to establish the child's prefix"; the corresponding *Formal Contract → Axiom → Depends* line lists "T4 (HierarchicalParsing…), TA5 (IncrementPostconditions…), TA5a (IncrementPreservesT4…)". Only T10a.4's Depends clause directly pairs the axiomatic `k' ∈ {1, 2}` restriction with TA5a's zero-count bounds; the axiom line itself does not quote the bounds it conditions on.
**Issue**: The axiom, as written, imposes a side condition that `zeros(t)` must satisfy a bound *before* every child-spawning event. But T10a.4's proof is the only place that derives T4 preservation inductively — it assumes the allocator discipline already restricts `(t, k')` to pairs within TA5a's bounds. This is circular in tone: the axiom constrains `k'` by `zeros(t)`, but whether that constraint is *actively checked by the allocator* or *derived from T10a.4 given some implicit initial condition* is not stated. A reader auditing the precondition chain cannot tell whether a conforming allocator is required to compute `zeros(t)` at runtime, or whether the axiom is implicitly assuming the inductive invariant holds (making the `zeros(·)` clause redundant given T10a.4).
**What needs resolving**: Clarify whether the `zeros(t)` bound in the axiom is a runtime precondition on the allocator (checked before each spawn) or a derived invariant (guaranteed by T10a.4 given a T4-valid root). If derived, the axiom could simply state `k' ∈ {1, 2}` and let T10a.4's proof supply the bound by induction; if required, the axiom's Depends line should prominently cite TA5a's `zeros(·)` definition as a precondition symbol whose meaning the axiom itself relies on to be well-formed.
