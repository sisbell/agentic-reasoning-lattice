# Cone Review — ASN-0034/TA6 (cycle 8)

*2026-04-17 15:51*

### T4c postcondition lacks parentheses to disambiguate the four biconditionals from their connecting conjunctions
**Foundation**: T4 (HierarchicalParsing), Formal Contract *Postconditions* (T4c) — and the identically-worded body-prose statement in the T4c framing paragraph.
**ASN**: T4c's postcondition is written as `(A t ∈ T : t is T4-valid :: zeros(t) = 0 ↔ t is a node address ∧ zeros(t) = 1 ↔ t is a user address ∧ zeros(t) = 2 ↔ t is a document address ∧ zeros(t) = 3 ↔ t is an element address)`. The same string reappears in the T4c framing prose ("T4c *defines* the four hierarchical level labels... by zero count: ..."). No parentheses delimit which `↔` pairs bind with which `∧`.
**Issue**: Under the standard logical-operator precedence that this ASN uses elsewhere (T1 writes `m + 1 ≤ m` as `m + 1 < m ∨ m + 1 = m`, leaning on `∨`-binds-tighter-than-a-surrounding-biconditional reading), `↔` has lower precedence than `∧`. The given string therefore parses as a chained biconditional `zeros(t) = 0 ↔ (t is a node address ∧ zeros(t) = 1) ↔ (t is a user address ∧ zeros(t) = 2) ↔ (t is a document address ∧ zeros(t) = 3) ↔ (t is an element address)`, which is not what T4c is meant to assert and is not even well-formed as a definition (the right operand of the first `↔` mixes a label predicate with an arithmetic equation at a different zero count). The intended reading — `(zeros(t) = 0 ↔ t is a node address) ∧ (zeros(t) = 1 ↔ t is a user address) ∧ (zeros(t) = 2 ↔ t is a document address) ∧ (zeros(t) = 3 ↔ t is an element address)` — requires the four biconditionals to be parenthesised individually. Since T4c's framing explicitly says the biconditionals are *definitional* (each one defines a label), the parsing ambiguity bears directly on whether a downstream citation of "t is a node address" terminates on a defined predicate. This is the kind of precision gap (one symbol, more than one meaning available from the written form) that Lamport-style review is supposed to catch.
**What needs resolving**: Make the top-level structure of T4c's postcondition unambiguous. Either (a) parenthesise each biconditional — e.g., `(A t ∈ T : t is T4-valid :: (zeros(t) = 0 ↔ t is a node address) ∧ (zeros(t) = 1 ↔ t is a user address) ∧ (zeros(t) = 2 ↔ t is a document address) ∧ (zeros(t) = 3 ↔ t is an element address))`; or (b) restate T4c as a function-style definition, e.g., `level(t) = node if zeros(t) = 0, user if zeros(t) = 1, document if zeros(t) = 2, element if zeros(t) = 3` for T4-valid `t`, and have the body prose cite `level(t)` rather than four independent label predicates. Mirror whichever form is chosen in the T4c framing-paragraph restatement so that contract and body agree on shape.
