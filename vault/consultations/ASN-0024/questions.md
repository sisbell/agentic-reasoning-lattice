# Sub-Questions — Operational Fragmentation Bounds

**Inquiry:** What must the system guarantee about the growth of internal mapping entries as sequences of INSERT, DELETE, and COPY operations are applied? What bounds must hold on state representation size relative to content size?

1. [nelson] When a document undergoes a long sequence of insertions and deletions, must the system's record of that document's structure remain bounded relative to the amount of content it currently contains?
2. [nelson] Must a single deletion operation — removing one character or one span — add at most a fixed, small number of new entries to the system's internal account of document structure?
3. [nelson] If a document is built entirely by copying spans from other documents, must the structural description of that document grow only with the number of distinct copy operations, not with the total volume of content referenced?
4. [nelson] Must the system guarantee that an insertion into the middle of a document adds no more than a constant amount of bookkeeping beyond storing the new content itself?
5. [nelson] When content is shared across many documents through transclusion, must the system's total state grow only with the number of sharing relationships, not with the size of the shared content multiplied by the number of uses?
6. [nelson] As a document accumulates a history of edits across many versions, must the system's representation of any single version remain proportional to that version's content size rather than to the total number of edits ever performed?
7. [nelson] Must the system guarantee that no editing operation — insert, delete, or copy — can cause the structural description of an unrelated document to grow?
8. [nelson] If a user performs a deletion that splits a contiguous span into two parts, must the cost in structural bookkeeping be bounded independently of the size of the spans on either side of the split?
9. [nelson] Must the system ensure that the total representation size across all documents grows at most linearly with the combined number of editing operations applied to those documents?
10. [nelson] When content is permanently stored and later referenced by address in a new document, must the new document's structural description record only the reference boundaries, not duplicate any representation of the content's internal structure?
11. [gregory] When INSERT splits an existing POOM bottom crum at an interior point via `slicecbcpm`, does the operation always produce exactly two crums from one, or can a single INSERT produce three or more bottom crums in the POOM?
12. [gregory] Does DELETE ever increase the number of POOM bottom crums — specifically, when `deletend` slices boundary crums via the THRUME case, does the surviving fragment always replace the original crum in-place, or can it create additional entries?
13. [gregory] Under what exact conditions does `isanextensionnd` coalesce a new POOM entry into an adjacent existing one during COPY — does it require strict I-address contiguity (reach == origin), and does it ever fire for COPY operations or only for sequential character-by-character INSERT?
14. [gregory] When COPY of a source V-span that maps to N non-contiguous I-address ranges in the source POOM is inserted into the target, does the target POOM receive exactly N new bottom crums, or can `vspanset2sporglset` splitting produce more than N entries?
15. [gregory] Since `levelpull` is disabled and trees only grow in height, what is the maximum POOM tree height reachable after a sequence of N INSERT operations given `MAXUCINLOAF=6` — is height bounded by `ceil(log_6(N))`, or can pathological split sequences produce taller trees?
16. [gregory] After a cycle of INSERT followed by DELETE of the same span repeated K times at the same V-position, how many POOM bottom crums remain compared to the initial state — does each cycle leave residual tree structure or fragment existing entries?
17. [gregory] Is there any mechanism in udanax-green — explicit or implicit — that compacts or defragments a POOM, merging adjacent bottom crums that happen to have contiguous I-addresses back into a single entry?
18. [gregory] The spanfilade receives one DOCISPAN entry per contiguous I-span per INSERT and per COPY — after N operations each producing M I-spans, is the spanfilade entry count exactly the cumulative sum of spans across all operations, with no deduplication or merging of overlapping entries?
19. [gregory] For the granfilade with `MAXBCINLOAF=1`, each inserted byte occupies exactly one leaf position — does this mean granfilade tree node count grows as O(B) for B total bytes ever inserted, or do the upper crum layers add significant overhead beyond O(B/6^h) per level?
20. [gregory] In the `recombinend` 2D rebalancing algorithm for POOMs, the commented-out `break` statements allow a single receiver to absorb multiple donors — does this merging reduce the total number of upper crums below what strict pairwise rebalancing would produce, and can it ever reduce bottom crum count?
