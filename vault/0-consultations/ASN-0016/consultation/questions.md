# Sub-Questions — Content Provenance

**Inquiry:** What must the system guarantee about tracing the origin and path of content? Given content at a location, what can be determined about where it came from, how it arrived, and its relationship to the same content appearing elsewhere?

1. [nelson] **Inquiry** — Pose focused questions about specific design guarantees
2. [nelson] **Discovery** — Extract requirements (R), judgments (J), and frame conditions (F) from source documents
3. [nelson] **Specification** — Formalize into definitions and provable properties
4. [nelson] Must the system guarantee that every unit of content can be traced back to a single, identifiable origin — the user who stored it, the moment of storage, and the document in which it first appeared?
5. [nelson] When the same content appears in multiple documents through sharing, must the system be able to produce, on demand, the complete list of all documents currently containing that content?
6. [nelson] Given content found in a document, must the system be able to determine whether that content originated in that document or arrived through transclusion from another document — and if so, identify the source?
7. [nelson] Must the derivation history of a document's versions preserve enough detail to trace exactly which content was added, removed, or rearranged at each step in the version chain?
8. [nelson] Must the system permanently record every operation that caused content to arrive at its current location — including the sequence of transclusions, edits, and version-creation acts along the way?
9. [nelson] If the same content appears in two documents independently — not through sharing but through separate acts of storage — must the system distinguish these as unrelated, even if the content is byte-identical?
10. [nelson] Must the system be able to answer temporal provenance queries — for example, "which documents contained this content at a specific past moment" — not just current-state queries?
11. [nelson] Does the ownership record permanently attached to content provide an unbroken chain of attribution, such that any viewer of shared content can always discover the original author regardless of how many documents it has passed through?
12. [nelson] Must the system record the historical fact that a transclusion relationship once existed, even after the receiving document removes the transclusion — preserving the complete path content has traveled?
13. [nelson] When content from one document is transcluded into a second document, and that second document is then the source of a further transclusion into a third, must the system trace the chain all the way back to the original document of first storage?
14. [gregory] Given content at V-address `v` in document `d`, does walking the POOM to retrieve the I-address unambiguously distinguish whether that content was originally INSERT-ed into `d` versus COPY-ed from another document, based solely on whether the I-address falls within `d`'s own I-address range (`docISA.0.1.*`)?
15. [gregory] When `find_documents_containing` returns a set of documents sharing I-addresses with a query span, does the spanfilade preserve enough information to reconstruct the temporal ordering of which document first received those I-addresses — i.e., can the original creator be distinguished from subsequent transcluders?
16. [gregory] If document A inserts "Hello", document B copies it from A, and document C copies the same span from B, does `find_documents_containing` called from C return both A and B, or only the documents with DOCISPAN entries — and does the DOCISPAN for B's copy point to the same I-address range as A's original?
17. [gregory] The I-address hierarchy encodes document origin (`node.0.user.0.doc.0.element`) — when resolving an I-address retrieved from a POOM entry, can `findpreviousisagr` or similar granfilade traversal determine the originating document address by truncating the I-address at the document level, or is there a dedicated operation for extracting document origin from an I-address?
18. [gregory] After CREATENEWVERSION, both the original and version share identical I-addresses — does `compare_versions` (SHOWRELATIONOF2VERSIONS) return the shared spans as paired V-address ranges, and is this the ONLY mechanism for determining that two documents are related by versioning rather than independent transclusion?
19. [gregory] When a single I-address span appears at multiple V-positions within the same document (via self-transclusion), does `incontextlistnd` return ALL V-positions for that I-address, and can the system distinguish which V-position was the original insertion versus which was the copy?
20. [gregory] If content is deleted from document A and then the same I-addresses are transcluded back into A from document B (which had previously copied them), does the resulting POOM entry in A become indistinguishable from the original state — i.e., is the round-trip invisible at the POOM level?
21. [gregory] Does `vspanset2sporglset` preserve the I-address contiguity structure when converting V-spans to sporgls — specifically, if a V-span maps to three non-contiguous I-address ranges (due to interleaved edits), does the resulting sporgl set contain exactly three entries that individually trace back to three distinct insertion operations?
22. [gregory] For DOCISPAN entries created by COPY versus INSERT, is there any structural difference in the spanfilade entry that would allow distinguishing "this document contains original content" from "this document contains transcluded content," or are both recorded identically as `(document, i-address, width)` tuples?
23. [gregory] When APPEND adds content without creating a DOCISPAN entry (the commented-out `insertspanf` call), is the appended content's I-address still discoverable through any other mechanism — for example, by querying the granfilade directly via `findpreviousisagr` — or is there truly no path from an external document to discover that content exists?
